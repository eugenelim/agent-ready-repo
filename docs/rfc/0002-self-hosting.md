## Summary

This repo is the first consumer of its own build pipeline. The
pipeline reads every `packs/<pack>/.apm/` source primitive plus
every pack's `seeds/` content plus the adapter contract,
and emits a Claude Code overlay (`.claude/skills/`,
`.claude/agents/`, `AGENTS.md`, `CLAUDE.md` symlink) onto the
repo itself — as if this repo had installed *every* pack from the
catalogue and run the adaptation step. CI runs `make build
--check` on every PR and fails if any projected path drifted from
the per-pack sources. The trade: muscle-memory shifts from editing
the projection to editing the pack source; the structural property
is that what we ship to adopters (per pack, via `claude /plugin
install` or `apm install`) and what we run ourselves cannot
diverge by construction. We dogfood every pack on every PR; the
Claude Code adapter is the one we exercise daily, the others get
covered by RFC-0001's CI fixture.

## Motivation

Three observations drive this RFC:

**1. The drift class still applies in the symmetric direction.**
RFC-0001's structural move — source primitives in `packs/*/.apm/` + a build
pipeline + per-IDE projected outputs — closes the *one-source-edit-
demands-N-coupled-edits* class for the bundle's downstream
artifacts. It does not close *this repo edits the projection
directly, the source goes stale*: if a maintainer edits
`.claude/skills/work-loop/SKILL.md` to fix a typo, the change
doesn't land in `packs/core/.apm/skills/work-loop/SKILL.md`, and the next time
the build pipeline runs (for distribution to adopters), the typo
fix evaporates. RFC-0002 makes this impossible by gating PRs on
"build output matches on-disk projection."

**2. We can't dogfood what we don't install.** RFC-0001 lists three
install paths (Claude plugin, APM, constrained-network CLI fallback).
Adopters will hit
build pipeline bugs (placeholder mishandling, adapter-contract
edge cases, recipe-output collisions) in *their* environments
unless we hit them in ours first. Self-hosting puts the maintainer
PR cycle in the path where those failure modes surface.

**3. The build pipeline's Claude Code adapter is the load-bearing
one for us.** This repo runs Claude Code. The Kiro, Copilot, and
Codex adapters get exercised by RFC-0001's CI fixture (Success
Criterion 2) but never by a maintainer's daily PR cycle.
Self-hosting under the Claude Code adapter is the strongest
dogfooding signal we can realistically produce; the asymmetry is
named honestly in [Drawbacks](#drawbacks).

## Proposal

One source-of-truth (`packs/*/.apm/` + `packs/*/seeds/` + the adapter contract
+ the build pipeline), one regeneration command (`make build
--self`), one CI gate (`make build --check`), one local addendum
file (`AGENTS.local.md`), and an explicit list of paths that stay
out.

### Source-of-truth split

Every file in this repo falls into one of two categories: source
(hand-edited; the build pipeline reads from it) or projected
(regenerated by the build pipeline; the CI gate enforces no manual
edits).

| Path | Category | Notes |
|---|---|---|
| `packs/<pack>/pack.toml` | Source | Pack metadata (name, version, deps, `[pack.adaptation]`, seed list). |
| `packs/<pack>/.claude-plugin/plugin.json` | Source | Per-pack Claude Code plugin manifest (hand-authored). |
| `packs/<pack>/.apm/skills/<name>/` | Source | Agent Skills spec layout, scoped to a pack. |
| `packs/<pack>/.apm/agents/<name>.md` | Source | Subagent definitions, scoped to a pack. |
| `packs/<pack>/.apm/hooks/<name>.py` | Source | Hook bodies, scoped to a pack. |
| `packs/<pack>/seeds/**` | Source | Pack seed files (README seeds, layer-0 content templates, `docs/_templates/*`). The upstream for every `docs/`-side projected path. |
| `docs/specs/adapter-contract/contract.toml` | Source | The adapter contract spec. |
| `tools/build/` | Source | The build pipeline. |
| `README.md`, `USING_THIS_TEMPLATE.md`, `LICENSE-*`, `.gitignore`, `.github/workflows/*` | Source | Repo-perspective documents and operational files. |
| `AGENTS.local.md` | Source | This repo's contributor-specific guidance. See [Local addendum](#local-addendum). |
| `AGENTS.md` (root) | Projected | From `packs/core/seeds/AGENTS.md` (the template, which embeds `<!-- agent-skills:start -->` / `<!-- agent-skills:end -->` delimiters). The self-host recipe substitutes `<adapt:NAME>` markers and lets the Codex adapter populate the managed block with per-pack skill descriptions. |
| `CLAUDE.md` (symlink → `AGENTS.md`) | Projected | Symlink recreated by build. |
| `docs/CHARTER.md`, `docs/CONVENTIONS.md` | Projected | From `packs/core/seeds/docs/{CHARTER,CONVENTIONS}.md` with `<adapt:NAME>` substitution. |
| `docs/architecture/README.md`, `docs/architecture/overview.md` | Projected | From `packs/core/seeds/docs/architecture/`. |
| `docs/specs/README.md` | Projected | From `packs/core/seeds/docs/specs/README.md`. |
| `docs/knowledge/README.md`, `docs/knowledge/patterns.jsonl` | Projected | From `packs/core/seeds/docs/knowledge/`. `patterns.jsonl` ships empty; self-learning skills append to it post-projection. |
| `docs/product/README.md`, `docs/product/roadmap.md`, `docs/product/changelog.md` | Projected | From `packs/core/seeds/docs/product/`. |
| `docs/rfcs/README.md` | Projected | From `packs/governance-extras/seeds/docs/rfcs/README.md`. |
| `docs/adrs/README.md` | Projected | From `packs/governance-extras/seeds/docs/adrs/README.md`. |
| `docs/guides/README.md`, `docs/guides/{tutorials,how-to,reference,explanation}/README.md` | Projected | From `packs/user-guide-diataxis/seeds/docs/guides/...`. Self-host iterates every `packs/*/` directory, so this pack's seeds always project when the pack exists on disk. |
| `packages/README.md`, `packages/_example/` | Projected | From `packs/monorepo-extras/seeds/packages/`. Same iteration rule — if the pack directory exists, its seeds project. (This repo today does have a `monorepo-extras` pack source but no in-tree packages of its own.) |
| `docs/_templates/spec.md`, `plan.md` | Projected | From `packs/core/seeds/docs/_templates/`. |
| `docs/_templates/rfc.md`, `adr.md` | Projected | From `packs/governance-extras/seeds/docs/_templates/`. |
| `docs/rfcs/NNNN-*.md`, `docs/adrs/NNNN-*.md`, `docs/specs/<feature>/*` | Manual | THIS repo's own governance entries (this repo is its own first adopter; entries authored at the projected paths post-projection). |
| `.claude/skills/<name>/`, `.claude/agents/<name>.md` | Projected | Claude Code adapter's output across all packs' `.apm/skills/` and `.apm/agents/`. |
| `tools/hooks/<name>.py` | Projected | Claude Code adapter's hook-body output. |
| `.claude/settings.local.json` | Projected (partial) | The `hooks` key only; other keys machine-local. |
| `.claude-plugin/marketplace.json` (root) | Projected | Aggregated from `packs/*/.claude-plugin/plugin.json` by the self-host recipe so this repo is itself a usable marketplace at HEAD. |
| `dist/` | Excluded | Build artifacts. Gitignored. |

The new top-level rule: if a path is in the *Projected* category,
edit its source under `packs/*/.apm/` or `packs/*/seeds/` (or the adapter
contract, if the *projection mode itself* changes). Editing the
projection directly is caught by CI and bounced.

### The self-build command

```
make build --self
```

`--self` is the explicit signal that the build target is this repo
itself, not an external `dist/` directory. The flag changes three
behaviours:

1. **Output root is the repo itself** — projected paths land at
   `.claude/`, `tools/hooks/`, `AGENTS.md`, not under `dist/`.
2. **The recipe is implicit** — `--self` uses a built-in recipe
   that selects the Claude Code adapter (this repo is a Claude
   Code shop) plus the Codex adapter for the `AGENTS.md`
   projection (because `AGENTS.md` is read by tools beyond Claude
   Code via the open AGENTS.md spec).
3. **Refuses on a dirty working tree** unless `--force` is also
   passed. `--force` bypasses *only* the dirty-tree check; it does
   not bypass the comparison-standard checks at build-vs-on-disk
   diff time (that is the gate's whole point). CI runs against a
   clean checkout so this isn't a CI concern; it's a
   maintainer-ergonomics guard against accidental clobbering of
   in-flight edits.

A `make build` target without `--self` runs the full distribution
build into `dist/` (per RFC-0001's recipes). The two modes share
the same build pipeline; only the output root and recipe selection
differ.

**The `self-host.toml` recipe** (this RFC's deliverable; RFC-0001's
F-build leaves room for it) names the composite assembly self-host
needs that RFC-0001's per-pack recipes don't cover:

```toml
# tools/build/recipes/self-host.toml
[recipe]
name = "self-host"
description = "Overlay every pack's Claude Code projection onto this repo, plus a composite AGENTS.md and a root marketplace.json."

[recipe.adapters]
targets = ["claude-code", "codex"]    # Claude Code adapter drives the .claude/ projection;
                                      # Codex adapter contributes its managed-block output to AGENTS.md.

[recipe.iterate]
over = "packs/*/"
read-pack-metadata = "pack.toml"

[recipe.per-pack-overlay]              # NEW recipe section type (defined here, this RFC's deliverable);
                                       # iterates each pack and overlays its outputs onto the repo.
root = "."                            # write directly into the repo
copy-from-pack-apm = [
  { from = "{pack.path}/.apm/skills/", to = ".claude/skills/" },
  { from = "{pack.path}/.apm/agents/", to = ".claude/agents/" },
  { from = "{pack.path}/.apm/hooks/",  to = "tools/hooks/" },
]
copy-from-pack-seeds = [
  { from = "{pack.path}/seeds/", to = "." }   # the pack's seeds/ drops into the repo at the seed-relative paths
]
# Merge semantics: copy-from-pack-seeds merges directory contents
# file-by-file. Two packs may both contribute to the same
# directory (canonical example: docs/_templates/ — core ships
# spec.md+plan.md; governance-extras ships rfc.md+adr.md). File-
# level collisions across packs (same filename, different content)
# are a build-time error and must be resolved by renaming or
# consolidating before the build proceeds.

[recipe.composite-agents-md]           # NEW recipe section type; assembles AGENTS.md from
                                       # the core pack's seed template, with the Codex adapter
                                       # writing the managed block.
output-path = "AGENTS.md"
template-from = "packs/core/seeds/AGENTS.md"   # the seed template (with <adapt:NAME> markers and the managed-block delimiters embedded)
managed-block-adapter = "codex"                # invokes the Codex adapter from RFC-0001
managed-block-content-source = "all-installed-packs"   # aggregates skill descriptions across packs

[recipe.composite-marketplace]         # NEW recipe section type; aggregates per-pack plugin.json files
                                       # *directly from the pack sources* (not from dist/, which --self
                                       # doesn't populate).
output-path = ".claude-plugin/marketplace.json"
aggregate-from = "packs/*/.claude-plugin/plugin.json"
owner = { name = "eugenelim" }
```

Three new recipe section types are defined by this recipe and
become part of the catalogue's recipe schema:
`[recipe.per-pack-overlay]` (iterates packs and overlays both
`.apm/` and `seeds/` content), `[recipe.composite-agents-md]`
(uses a pack-shipped template with delimiters and delegates to a
named adapter for the managed-block body), and
`[recipe.composite-marketplace]` (aggregates per-pack manifests).
RFC-0001's F-spec deliverable picks them up as part of the
contract; F-self-host (this RFC's follow-on) implements them.

**Recipe ordering contract.** Section execution order matches
source order in the TOML. The composites in this recipe happen to
read from source paths (`packs/core/seeds/AGENTS.md`,
`packs/*/.claude-plugin/plugin.json`) rather than projected paths,
so parallel execution would be correctness-safe for the current
shape. But future composite sections may read from
`[recipe.per-pack-overlay]`'s output; to keep the schema
forward-compatible, implementations preserve declared order and
flag any composite that depends on overlay output (a future
`reads-from-overlay = true` field can make that dependency
explicit).

### Round-trip safety

CI gate: `make build --check` runs on every PR. Its contract:

1. Re-run the build pipeline with `--self` into a temporary
   directory.
2. For each path in the *Projected* category, compare the
   re-projected content against the on-disk content
   **byte-for-byte after line-ending normalisation to LF.** The
   gate reads both sides as bytes, normalises CRLF→LF before
   comparison.
3. For regular files, also compare file mode bits. A projected
   `tools/hooks/session-start.py` with mode `0644` and an on-disk
   copy with mode `0755` counts as drift.
4. For symlinks (currently `CLAUDE.md → AGENTS.md`), compare the
   symlink *target* via `lstat`. The gate never follows symlinks.
5. If the on-disk counterpart is missing, that is drift (report:
   "expected projected output, none on disk").
6. For paths *on disk* that fall in neither category, emit an
   `[info]` line to stderr naming the unclassified path.
   Enumeration uses `git ls-files --cached --others
   --exclude-standard` so editor noise and `.gitignore`-d build
   outputs don't surface. Info-level messages don't fail the build;
   they surface the omission so the next PR can classify the path.
7. Exit `0` if no drift; `1` if drift exists.

Failure messages name the source directly. Example:

```
[drift] .claude/skills/work-loop/SKILL.md differs from packs/core/.apm/skills/work-loop/SKILL.md.
        First differing hunk:
          - ## Stop conditions
          + ## Halt criteria
        Edit packs/core/.apm/skills/work-loop/SKILL.md and re-run: make build --self
```

The failure message names the file to edit and the command to
re-run; no second lookup needed.

### Local addendum

Some `AGENTS.md` content is genuinely about contributing to *this*
template repo — Conductor branch conventions, the gh-auth check
before push, repo-specific ADR/RFC links — and isn't portable to
adopters. That content lives in `AGENTS.local.md`, manually
authored, at the repo root.

The projected root `AGENTS.md` ends with a single pointer:

```markdown
> Working on this repo specifically? See [`AGENTS.local.md`](AGENTS.local.md).
```

`AGENTS.local.md` is in the *Source* category — the build pipeline
never touches it. The pointer line is part of the AGENTS.md
projection template (in `packs/core/seeds/AGENTS.md`'s tail block
or in a sibling `packs/core/seeds/_agents-footer.md` — settled in the
migration PR). Adopters who don't write an `AGENTS.local.md` end
up with a link that doesn't resolve; the obvious resolution is to
create the file.

Without an installer, the earlier draft's `local-addendum-pointer`
placeholder mechanism dissolves: the line is plain prose in the
source, not a templated value. Adopters who want to suppress the
line edit the source post-install (the same way they'd customize
any other projected content). The previous draft's whitespace-
sensitive elision-semantics problem disappears with it.

### What stays out of self-hosting

These paths are excluded from the *Projected* category by category,
not file-by-file:

| Path / pattern | Why excluded |
|---|---|
| `.git/` | Repository internals. |
| `.context/`, `.claude/settings.local.json` | Machine-local. |
| `docs/rfcs/NNNN-*.md`, `docs/adrs/NNNN-*.md`, `docs/specs/<feature>/*` (entries themselves) | This repo's own governance entries authored at the projected paths. The directory shapes + READMEs are Projected (per the source table); the NNNN-prefixed entries are Manual. |
| Non-README content under `docs/architecture/`, `docs/product/`, `docs/knowledge/`, `docs/guides/` | This repo's own narrative content at the projected paths. The README seeds + template files in those directories are Projected; everything else under them is Manual. |
| `README.md`, `USING_THIS_TEMPLATE.md` | Repo-perspective documents about the template itself. |
| `LICENSE-*`, `.gitignore`, `.github/workflows/*` | Repo-specific operational files. |
| `AGENTS.local.md` | This repo's contributor-specific guidance. |
| `packs/*/.claude-plugin/plugin.json` | Hand-authored per-pack manifest (Source per the source-of-truth table); never projected. |
| Anything under `packs/*/.apm/`, `packs/*/.claude-plugin/`, `packs/*/seeds/`, `tools/build/`, `docs/specs/adapter-contract/` | The source-of-truth side. The build pipeline reads from these; they aren't projection targets. |

There is no "by default projected" rule — inclusion is opt-in by
appearing in the *Projected* table above.

### Migration plan

Self-hosting can't begin until RFC-0001's F-spec (adapter contract
TOML) and F-build (build pipeline reference implementation) have
landed. Once they have, the cutover is four steps:

1. **Author the source packs.** Hand-extract content into each
   of the four packs per the audit:
   - **`core`:** `.claude/skills/{work-loop,new-spec,bug-fix}/`
     + a new `adapt-to-project/` skill body →
     `packs/core/.apm/skills/`. The four reviewer/implementer
     agents → `packs/core/.apm/agents/`. Existing
     `tools/hooks/session-start.py`, `pre-pr.py` →
     `packs/core/.apm/hooks/`. `.claude/commands/conventions-check.md`
     → `packs/core/.apm/commands/`. Existing root `AGENTS.md`,
     `docs/CHARTER.md`, `docs/CONVENTIONS.md`, `docs/_templates/{spec,plan}.md`,
     and seed READMEs for `docs/architecture/`, `docs/specs/`,
     `docs/knowledge/`, `docs/product/` → `packs/core/seeds/`.
   - **`governance-extras`:** `.claude/skills/{new-rfc,new-adr,update-conventions}/`
     → `packs/governance-extras/.apm/skills/`.
     `docs/_templates/{rfc,adr}.md` + seed READMEs for
     `docs/rfcs/` and `docs/adrs/` → `packs/governance-extras/seeds/`.
   - **`user-guide-diataxis`:** A new `new-guide` skill body →
     `packs/user-guide-diataxis/.apm/skills/`. Seed READMEs for
     `docs/guides/` and its four quadrants →
     `packs/user-guide-diataxis/seeds/`.
   - **`monorepo-extras`:** `.claude/skills/new-package/` →
     `packs/monorepo-extras/.apm/skills/`. Existing
     `packages/_example/` + a seed `packages/README.md` →
     `packs/monorepo-extras/seeds/`.
   - Write a `pack.toml` for each pack with metadata,
     `[pack.adaptation]` markers, and `recommends` dependencies
     where applicable (governance-extras / user-guide-diataxis /
     monorepo-extras all recommend `core`).
   - Write a per-pack `.claude-plugin/plugin.json` for each pack.
   - Move the existing root `AGENTS.md`'s contributor-specific
     paragraphs into a new `AGENTS.local.md`. The criterion: any
     paragraph that names this repo, this template's origin, or a
     contributor workflow not portable to adopters.
2. **Reconcile until the projection is a byte-identical no-op.**
   Run `make build --self --dry-run` repeatedly, editing
   pack-side sources until projected output matches on-disk
   content under the LF-normalised-byte rule.
3. **Run `make build --self` for real.** The commit records that
   projected content matches the source.
4. **Enable `make build --check` as a required CI check.** From
   this point on, drift between source and projected output is a
   merge-blocking condition. May land as one PR or two depending
   on CI-config review cadence; the conceptual step is one.

The cutover is one-shot. Rolling back means either re-committing
the hand-maintained projections as canonical (losing the
structural deduplication) or accepting build-derived output as the
new floor and moving forward. There is no grace period.

### Composition with RFC-0001

RFC-0002 produces one new spec — **self-hosting-bootstrap**, filed
at `docs/specs/self-hosting-bootstrap/` — which sequences
after RFC-0001's F-spec and F-build land. The spec owns the
`--self` flag's exact CLI surface, the dirty-working-tree
behaviour, the migration PR(s), and the CONVENTIONS.md edit
recording the source-of-truth split.

The spec does *not* own the per-IDE adapter contracts (those are
F-spec from RFC-0001), the build pipeline implementation (F-build),
or the distribution-output recipes (F-dist).

### Success criteria

Three checks gate this RFC as delivered:

1. **`make build --check` is a required CI check** and is green on
   the main branch. Every projected path matches its source under
   the LF-normalised-byte rule.
2. **`make build --self` on a clean checkout produces no on-disk
   change.** Byte-identical projected output, per the comparison
   standard.
3. **The first adopter-shaped change exercises the same path
   adopters will use.** A real `packs/*/.apm/`-side or `packs/*/seeds/`-side
   edit (e.g. a CHARTER tweak) lands in the repo via the build
   pipeline, not by direct edit to the projected file. The fact
   that a maintainer can no longer muscle-memory their way into a
   hand-edit is the qualitative signal that self-hosting is real.

## Alternatives considered

**Alt 1 — Do nothing.** Continue hand-maintaining the projections
once F-build authors them, accept that they drift from the source
trees. **Why not chosen:** recreates the drift class RFC-0001
exists to close, in the symmetric direction. The cost compounds
with every distribution release and every governance edit.

**Alt 2 — Use APM on ourselves rather than the build pipeline.**
Run `apm install` against our own `apm.yml` to project `.apm/`
into the repo. **Why not chosen:** APM's adapter implementations
are the part of the ecosystem we're explicitly *not* depending on
at the adapter-contract level (RFC-0001's F-spec is our novel
contribution because APM's adapters aren't spec'd). Self-hosting
via `make build --self` exercises our build pipeline and our
adapter contract; self-hosting via `apm install` would validate
APM's adapter behaviour, not ours. The right tool to dogfood is
the one we own.

**Alt 3 — Use `claude /plugin install` on ourselves.** Build the
Claude plugin distribution, then install it into this repo as if
it were an adopter. **Why not chosen:** same shape as Alt 2 — it
validates Claude's plugin loader rather than our build pipeline.
We get that signal indirectly via Success Criterion 2 of RFC-0001
(install commands work against fresh adopter repos); we don't need
to reproduce it on the maintainer PR cycle.

**Alt 4 — Lint-only drift detection.** Add a CI lint that diffs
hand-maintained `packs/*/.apm/`-side and `.claude/`-side files and fails on
divergence; don't unify the source-of-truth. **Why not chosen:**
keeps two copies, demands every contributor know which to edit, and
treats divergence as a *condition to avoid* rather than a
*structurally-impossible state*. The drift class isn't closed; it's
surveilled.

**Alt 5 — Managed-block addendum for `AGENTS.local.md`.** Instead
of a separate file, ship `<!-- self-host:end -->` delimiters in the
projected `AGENTS.md` and let this repo append below them. **Why
not chosen:** introduces a new managed-block class with its own
conflict policy, parser, and update story for one use case. The
separate-file approach is one new file. The managed-block path
remains revisitable if a second use case appears.

## Drawbacks

**Indirection cost.** Editing the projected root `AGENTS.md` now
requires opening the `packs/core/seeds/`-side source instead. One extra
step in muscle memory, amortised across all maintainers, every
governance edit. The CI gate catches the mistake; it doesn't
remove the cost of having to learn the new path.

**Bootstrap reconciliation effort.** Migration step 2 is real
work. Whitespace, line endings, ordering of unordered lists,
header capitalisation — every inadvertent delta has to be triaged.
Estimate: a half-day of triage during the migration PR, paid once.

**CI overhead.** Re-running the build pipeline and diffing every
projected path adds seconds to every PR. Implementation-cost
negligible; latency non-zero.

**One-shot migration.** Rolling back means either re-vendoring the
hand-maintained projections (losing the structural property the
RFC exists to gain) or accepting build-derived output as the new
floor. There is no grace period.

**Single-adapter dogfooding.** This repo self-hosts only under the
Claude Code adapter (plus the Codex adapter for `AGENTS.md`). The
Kiro and Copilot adapters get exercised by RFC-0001's CI fixture
but never by a maintainer's daily PR loop. Kiro adopters' issues
could ship undetected by us and reach adopters first. We accept
this asymmetry because building first-class adapter parity into
the maintainer's daily flow would require running multiple IDEs
simultaneously, which isn't realistic.

**No install-time placeholder substitution; self-host runs against
concrete content.** The earlier draft of RFC-0002 had a
`<placeholder>` substitution layer with `[self-host]` table values
that ran at every install. The rewrite shifts substitution to the
adaptation step (per RFC-0001's `[pack.adaptation]` markers).
Self-host's twist: this repo's own projected `docs/CHARTER.md` and
related content are *not* templated — they are this repo's
concrete content. The build pipeline projects them into AGENTS.md
literally. The `<adapt:NAME>` markers from `[pack.adaptation]`
appear in the *pack* sources (`packs/<pack>/.apm/...`); this repo's
self-host build resolves them against this repo's actual values
(project name = "agent-ready-repo", etc.) so the projected output
is concrete. Adopters who install our packs see the markers
unresolved until they run `adapt-to-project`.

## Unresolved questions

The RFC-level design is settled. Two implementation-altitude items
defer to the `self-hosting-bootstrap` spec:

1. **Where the AGENTS.md footer pointer lives.** The
   `> Working on this repo specifically? See AGENTS.local.md` line
   needs a home in the source tree. Options: as a trailing block
   in `packs/core/seeds/AGENTS.md`'s tail, as a sibling file
   `packs/core/seeds/_agents-footer.md` that the AGENTS.md projection
   appends, or hard-coded into the build pipeline's AGENTS.md
   renderer. Working assumption: a sibling file, because it keeps
   CHARTER.md focused and makes the addendum convention
   discoverable by adopters who grep `packs/core/seeds/` for "AGENTS".

2. **Whether seed READMEs participate in drift detection.** Each
   pack ships a `packs/<pack>/seeds/.../README.md` describing what
   goes in that home. These READMEs are scaffolding (we ship them;
   adopters use them as written). Should the drift gate cover
   them in self-host (so this repo's projected
   `docs/<home>/README.md` files must match the pack seeds), or
   are they Manual at the projected paths (so this repo can hand-
   edit its own README's without re-rendering)? Working
   assumption: Projected — the drift gate enforces alignment,
   keeping this repo's projected READMEs faithful to the pack
   seeds we ship to adopters.
