## The ask

- **Recommendation (BLUF):** RFC-0041 gave `work-loop` the verification *shape* for infra but left one assumption standing — that the agent **knows the platform and will exercise the whole chain**. Two real multi-day builds on *different* stacks ([`0044-notes/field-report.md`](0044-notes/field-report.md)) falsify it: the loop builds, verifies, and reviews against **model memory or a cheap proxy** rather than the platform's **actual contract and runtime ground truth**, and the failures fall into **six families that are MECE along the inner-loop-stage axis** (Author → Provision → Verify → Debug, with Author and Verify each splitting into two truth-kinds). Close all six — **all doctrine + reference prose, one new `core` skill, no executable tooling, no new reviewer/agent**:
  - **(A1) Author·structural — the contract.** A new `core` skill **`infra-contract-acquisition`**: a tiered, **tool-keyed (not vendor-keyed)** protocol that acquires a stack's real contract from the deterministic oracles its own toolchain ships (validate/plan/preview/synth + a machine-readable schema slice), declares its **oracle tier + confidence**, and degrades honestly across the strong→weak spectrum (so on-prem / Hetzner / bare-metal are covered at the strength their tooling exposes — *not* big-three-only). Plus a **contract-grounding gate** in EXECUTE: acquire before you guess (the infra generalization of "grep to verify a function exists").
  - **(A2) Author·behavioral — the craft.** A new `operational-safety` module **`cloud-implementation-craft`**, loaded into the **implementer's EXECUTE brief** (not only the reviewer's): least-privilege-but-**sufficient** permissions; **eventual-consistency / propagation** waits; **timeout + retry/backoff incl. client/frontend cold-start tolerance**; dependency ordering; terminal-failed-state handling.
  - **(P) Provision — identity & convergence.** Add **durable-credentials-once** as a fifth artifact in RFC-0041 P1's multi-artifact preflight (alongside its verify-status, teardown, test-seed, and policy-as-code/CSPM-scanner artifacts); name **terminal-failed-state** (e.g. `ROLLBACK_COMPLETE` → destroy-recreate) as a convergence case.
  - **(V1) Verify·static — phased oracle fidelity.** Run the right *installed* verification tool at each phase and **don't over-trust the cheap-early one**: lint/validate < plan/preview < runtime deploy + smoke. "Synth passes / status: deployed" is not done.
  - **(V2) Verify·runtime — the data-plane probe (a distinct smoke flavor).** The infra smoke runs **inside the network boundary if the resource is private**, performs a **data-plane round-trip** (write → read-back), is **readiness-aware** (poll with bounded backoff; distinguish not-yet-ready/propagating from broken), and **tears itself down**. This refines RFC-0041 P2.
  - **(D) Debug — failure localization.** Strengthen `operational-safety`'s existing **`observability-and-smoke`** with a **symptom→layer log playbook** (504/timeout→proxy/facade; 403→authorizer/IAM; 500→handler; bisect the chain; correlation IDs) — not a new module.
  - **Delivery:** reviews ride the **existing `quality-engineer`** loading these references in infra-flavored mode — **no new agent**; a dedicated infra reviewer is deferred behind an explicit evidence trigger.

- **Why now (SCQA):** *Situation* — RFC-0041 made the loop able to *verify* a deploy (preflight → plan → idempotent apply → smoke → rollback). *Complication* — two builds on different stacks still failed/▢stalled for days, and in **both** the human had to push for real end-to-end verification; the failures cluster where RFC-0041 didn't reach — the loop reasons from memory/proxy, not platform reality, across author, provision, verify, and debug. *Question* — should the loop gain the discipline to ground itself in the platform's real contract and runtime truth at every inner-loop stage, or keep assuming the agent knows the platform and that a green synth means "works"?

- **Decisions requested:**
  1. **Scope: ground all six failure families (across the four inner-loop stages) as doctrine + references** (one new skill + one new `operational-safety` module + `work-loop` prose), **no executable tooling, no new reviewer/agent.** · recommended: yes · default: yes.
  2. **A1 — the `infra-contract-acquisition` skill** (tool-keyed tiered oracle protocol, oracle-tier-honest, spectrum-covering) **+ the EXECUTE contract-grounding gate.** · recommended: yes · default: yes.
  3. **A2 — a `cloud-implementation-craft` `operational-safety` module, loaded at EXECUTE** (permissions-sufficiency, propagation/eventual-consistency, timeout/retry/backoff/client-cold-start, ordering, terminal-failed-state). · recommended: yes · default: yes.
  4. **P — durable-credentials-once as a fifth preflight artifact; terminal-failed-state as a named convergence case; and the cross-cutting corollary that live-environment interaction (deploy/probe/log-pull/debug) goes through reusable credential-reusing scripts, not one-off command-spray.** · recommended: yes · default: yes.
  5. **V1 — phased oracle-fidelity doctrine** (right installed tool per phase; cheap-early < runtime; don't over-trust synth). · recommended: yes · default: yes.
  6. **V2 — the readiness-aware data-plane probe flavor** (in-network-if-private, write→read-back, poll-with-backoff ready-vs-broken, probe self-teardown), refining RFC-0041 P2. · recommended: yes · default: yes.
  7. **D — symptom→layer log playbook strengthening `observability-and-smoke`** (not a new module). · recommended: yes · default: yes.
  8. **Delivery: conformance/craft review via `quality-engineer` + the references, no new agent; defer a dedicated `infra-contract-reviewer` behind an evidence trigger.** · recommended: yes (no agent) / defer (agent) · default: same.

## Problem & goals

**Diagnosis.** RFC-0041 modelled *how to verify* a deploy but assumed two things the field report falsifies: that the agent **knows the platform's contract**, and that it will **exercise the whole chain** rather than trust a cheap proxy. Both builds show the loop substituting **model memory or an early/cheap oracle** for **ground truth** at every inner-loop stage — and the failure families are MECE along that loop (see `0044-notes/field-report.md` for the evidence and the exhaustiveness argument):

1. **Author·structural (contract).** Invented flags, a violated naming regex, a wrong tool-schema shape, an immutable-field collision — not reasoning failures but *contract* failures: the agent reasoned from half-remembered prose, and the human's fix was to hand it a platform skill mid-session.
2. **Author·behavioral (craft).** Permissions iterated reactively until authorized; propagation/eventual-consistency delays ("active in ~10 min") not designed for; timeouts escalated and `sleep`s sprinkled instead of designed backoff; **no client cold-start tolerance** (a cold-start timeout surfaced raw to the user); dependency cycles fixed by trial. The model does not apply golden cloud-implementation practice by default.
3. **Provision (identity & convergence).** Credentials re-resolved per call until an SSO wrapper timed out — the "establish once, reuse" fix was rediscovered *independently in both builds*; a `ROLLBACK_COMPLETE` stack can't be updated and must be recreated; non-idempotent creates collided on re-run.
4. **Verify·static (oracle fidelity).** The cheap early oracle (synth/validate) **passed while the system was broken**; "deployed ≠ working" was treated as done.
5. **Verify·runtime (the probe).** Proving it works needs a probe that runs **inside the network if the resource is private**, does a **data-plane round-trip**, is **readiness-aware** (poll/backoff; not-ready ≠ broken), and **tears itself down** — a different shape than app verification, and one the human had to demand both times.
6. **Debug (localization).** A facade-emitted timeout was chased dozens of times in the backend log — no symptom→layer map, no chain bisection.

The unifying cause: **the loop builds, verifies, and reviews against memory/proxy, not the platform's contract and runtime truth.** RFC-0041 added the scaffold; this RFC grounds it.

**Goals.**
- Make the loop **acquire the platform's real contract before authoring** (A1) and **apply golden implementation craft** (A2), reasoning over toolchain oracles and operational reality rather than model memory.
- Make **provisioning durable and convergent** (P) and **verification high-fidelity** — phased oracles not over-trusted (V1), and a runtime data-plane probe that actually exercises the deployed chain (V2).
- Give **debugging a localization map** (D).
- Keep it **tool-keyed and vendor-agnostic** so it spans the cloud/on-prem/bare-metal spectrum, and **honest about confidence** where the oracle is weak.
- Serve **both implementer (EXECUTE) and reviewer (REVIEW)** from the same references; re-derive independently at review, never self-reported.

**Non-goals** (could-have-been goals, deliberately dropped):
- *Not* a per-vendor knowledge base or a shipped contract for any platform — we ship the **acquisition protocol** + the existing mechanism to *recommend* a curated platform skill. Pre-enumerating clouds breaks Principle 1 and rots on arrival.
- *Not* executable tooling (schema-diff, plan-parser, probe-runner, oracle runner). Doctrine + references only (inherits RFC-0041 Decision 1; Principle 3).
- *Not* a new reviewer/agent (Delivery; three-reviewer ceiling).
- *Not* a new risk trigger (infra already routes to full mode; "infra-flavored" detection already exists).
- *Not* re-authoring RFC-0041's GATES sequence or `## Rollout` — V1/V2 refine the smoke layer; they don't duplicate sequencing.
- *Not* operating-stage (Day-2) concerns beyond what `operational-safety` already owns (drift, cost, blast-radius).

## Proposal

> *Post-acceptance: see [§ Errata](#errata) (2026-06-24) for additive refinements to A1's EXECUTE gate trigger and A2's craft enumeration — no decision below is reversed.*

Cascaded under the requested decisions; organized by the six MECE families across the four inner-loop stages.

**Decision 1 — scope: doctrine + references, six families, no tooling/agent.** Everything below is prose: one new `core` skill (`infra-contract-acquisition`), one new `operational-safety` module (`cloud-implementation-craft`), a strengthened `observability-and-smoke` module, and `work-loop` SKILL.md edits. No executable code; the orchestrator loads the infra references on the existing **infra-flavored** detection, now into **both** the implementer's EXECUTE brief and the reviewer's REVIEW brief.

**Decision 2 — A1: the `infra-contract-acquisition` skill + EXECUTE grounding gate.** Sibling of `operational-safety`/`security-checklists`, but its content is a **protocol**, not a checklist — shipped as **both** a short protocol body (T0→final) **and** a per-tool `references/` oracle table (the exact validate/plan/schema commands per stack and what each exposes), so strong-tier stacks are concrete without binding the prose to them (resolves Open Q1):
- **T0** detect the stack from the diff. **T1** run the toolchain oracle (`terraform validate`+`plan`, `cdk synth`, `pulumi preview`, CFN change sets, `kubectl --dry-run=server`) **plus a schema slice for the diff's resources** — coverage is **heterogeneous** and routed accordingly: CFN/CDK `createOnlyProperties` (immutable) via `describe-type`; Pulumi `replaceOnChanges` via `get-schema`; **Terraform `providers schema -json` does *not* expose force-new** (only `type`/`description`/`required`/`optional`/`computed`/`sensitive`), so the replace signal is read from `plan` + provider docs. **T2** a curated platform skill for the behavioural contract no schema encodes (managed-platform naming/schema/immutability are API/CLI surfaces in *no* IaC schema) — detect presence; **if absent on an unfamiliar managed surface, recommend authoring/installing one and surface it as a decision** (3-tier dependency policy: toolchain binaries are Tier-1 detect/fail-clean; a platform skill is content). **T3** official-docs retrieval, made searchable. **Final oracle** — the runtime data-plane probe (V2).
- **Oracle-tier honesty (the generality mechanism).** Coverage is a **capability spectrum** keyed to the tool: *strong* (Terraform/OpenTofu, Pulumi, CDK/CFN, k8s/Helm → any provider they address incl. Hetzner/Proxmox/vSphere/OpenStack/on-prem-k8s), *medium* (Ansible `--check --diff`, Bicep, cloud-init), *weak/none* (bespoke REST+curl, hand-rolled bare-metal). The agent **declares its tier + confidence** and on a weak oracle **shifts weight to the runtime probe** rather than faking static coverage.
- **EXECUTE grounding gate:** before generating a CLI invocation or IaC resource against an unfamiliar platform, acquire its contract — never guess a flag, schema shape, or field constraint. The infra generalization of AGENTS.md's "Grep to verify a function exists before importing it."

**Decision 3 — A2: the `cloud-implementation-craft` module (loaded at EXECUTE).** A new `operational-safety` module holding the golden cloud-implementation practices the model doesn't apply by default, **inlined into the implementer's EXECUTE brief** on infra-flavored work (a deliberate extension: `operational-safety` is consumed at EXECUTE here, not only at REVIEW):
- **Least-privilege-but-sufficient permissions** — grant the exact actions the call path needs, derived from the contract, in one pass — not iterated reactively until the call stops failing.
- **Eventual consistency / propagation** — newly-created identities/resources and async-activating features are not immediately usable; design a readiness wait, don't assume synchronous availability.
- **Timeout / cold-start / retry** — set timeouts to the *dependency's* real latency (a cold-starting managed runtime, not a warm local call); use bounded exponential backoff; and make **clients/frontends cold-start-tolerant** (retry/poll a 503/504 from a scale-to-zero dependency rather than surfacing it raw).
- **Dependency ordering** and **terminal-failed-state** handling (paired with P).

**Decision 4 — P: provision identity & convergence.** Add to RFC-0041 P1's multi-artifact preflight (verify-status, teardown, test-seed, policy-as-code/CSPM scanner) a fifth artifact — **a durable credential session**: establish credentials once (resolve into a static session via the toolchain's own export mechanism), assert with a single identity check, reuse for the loop; resolve profile-vs-static precedence up front. And name **terminal-failed-state** convergence in `state-and-idempotency`: a stack in a non-updatable failed state (e.g. `ROLLBACK_COMPLETE`) is destroyed and recreated, not retried in place.

**Corollary (cross-cutting P + D) — reusable scripts over one-off commands.** Because credentials are established once, *every* live-environment interaction — deploy, smoke probe, **log pull, debug step** — goes through a **reusable, idempotent, credential-reusing script** that accumulates into the verify/probe/teardown harness (RFC-0041 P1's task-zero artifacts), not a stream of one-off shell commands that each re-resolve credentials and leave nothing behind. One-off command-spray is the anti-pattern the field report shows twice over: credential thrash, *and* dozens of ad-hoc log/invoke calls and hand-tuned `sleep`s that never became a script. This is a discipline, not a new family — it sharpens RFC-0041 P4 (agent-drives-verification): the agent drives it *through scripts it can re-run and hand back*, so iteration converges instead of restarting.

**Decision 5 — V1: phased oracle fidelity.** Add to the PLAN/EXECUTE infra prose: verification oracles are **phased and increasing-fidelity** — static (lint/validate/policy-as-code) < plan/preview < **runtime deploy + smoke** — and the cheap-early oracle **must not be over-trusted**. Run whichever installed tool fits the phase (`synth`/`validate` early; the runtime probe last), and treat a green early oracle as *necessary, not sufficient*. ("Synth passed" is the local-typecheck analog, not a done-signal.) **Carve with A1:** both touch the same static oracles, but A1 (Author·structural) owns *did the agent consult the oracle to ground authoring before generating the resource?* while V1 owns *is a green early oracle being mistaken for "works" at verify?* — different question, different inner-loop stage.

**Decision 6 — V2: the readiness-aware data-plane probe (refines RFC-0041 P2).** The infra smoke's "active end-to-end probe" gains its real shape:
- **In-network if private** — a resource with no public endpoint is probed from **inside its network boundary** (an in-VPC task/function), not the operator's machine.
- **Data-plane round-trip** — write a record → read it back (and, for a request path, drive it end-to-end to the terminal user-visible result), not "resource exists / status healthy."
- **Readiness-aware** — poll until ready with **bounded backoff**, distinguishing **not-yet-ready/propagating** (wait) from **broken** (fail) — preventing redeploy-thrash on what is a propagation/cold-start delay.
- **Self-teardown against an ephemeral target** — the probe is infra; it runs against an **ephemeral, uniquely-named environment** so re-runs don't collide, and teardown (RFC-0041 P1's teardown artifact) includes the probe. This is the established `defer destroy` / `dev-test-run-uuid` shape from infra-testing tools (Terratest).

**Decision 7 — D: symptom→layer log playbook.** Strengthen `observability-and-smoke` with the localization *method*: enumerate the request/deploy path's log groups up front; **map the failing status to the emitting layer** (timeout/504 → the proxy/facade in front, not the backend; 403 → authorizer/IAM; 500 → handler); **bisect the chain**; carry a **correlation id** through the path; and **match the symptom against a known failure-signature → likely-cause catalog first** (e.g. `ROLLBACK_COMPLETE` → recreate; cold-start 504 → retry/poll; a conditional-write contention error → contention, not an error to fix) — the runbook-match step OSS diagnostic agents resolve ~40% of cases with before deep log reading. No new module. Debugging follows the **reusable-script corollary** (Decision 4): a diagnostic is captured as a re-runnable, credential-reusing script so localization accumulates, not fired as a one-off command.

**Decision 8 — delivery: `quality-engineer` + references, no new agent.** Contract-conformance and craft review ride the existing `quality-engineer` (already the infra reviewer in spirit — owns `operational-safety`, already dispatched on infra-flavored work), loading `infra-contract-acquisition` + `cloud-implementation-craft` and **re-deriving the contract independently from the oracles** — never trusting the implementer's own contract evidence (that reproduces the implementer's blind spot, the field-report failure). The auth-flow-contradiction class is caught earlier at **spec stage by `design-reviewer`**. A dedicated `infra-contract-reviewer` agent is **deferred behind an evidence trigger** (a spike showing oracle-execution traffic — large `plan`/`synth` output — measurably degrades `quality-engineer`'s other lenses); the contract is fetched in *slices*, which softens the a-priori isolation case. **Net: zero new agents.**

## Options considered

**Axis: form × home.** Any solution must answer "what artifact form" (doctrine/reference vs. executable tooling vs. per-vendor data) and "what home" (the existing loop + depth-libraries vs. a new surface). The options are MECE along it; the same axis RFC-0041 used, reused for continuity.

| Option | Form | Home | Verdict |
|---|---|---|---|
| **A. Doctrine + references in `work-loop` / `operational-safety` + one new skill** ★ | prose | existing loop + libraries | **Recommended** |
| B. Executable infra tooling (schema-diff, plan-parser, probe-runner) | code | new runtime | Fails Principle 3; per-tool bindings break Principle 1 |
| C. Per-vendor contract/knowledge base | bundled data | new data surface | Fails Principle 1; no long-tail coverage; stale on arrival |
| D. A new infra-contract reviewer agent | prose | new subagent | Contestable now; deferred behind evidence (Decision 8) |
| E. Rely on model knowledge / RFC-0041 as-is (do-nothing) | — | — | Cost of delay below |

- **A (recommended)** — mirrors `security-checklists`/`operational-safety` (accepted depth-library precedent) and drives the IaC ecosystems' own contract oracles (Terraform `validate`/`plan`/schema; CFN change sets + `describe-type`; Pulumi `get-schema`; k8s `--dry-run=server`) rather than re-implementing them. Clears all four principles.
- **B** — a plan-parser/probe-runner is runtime infrastructure (Principle 3 — the bar that put browser-bridge out of charter) needing per-tool bindings (Principle 1). The agent runs the existing oracles directly.
- **C** — a bundled per-cloud KB can't cover the long tail (on-prem/Hetzner/bespoke), rots between releases, and breaks stack-neutrality. The curated *platform skill* (T2) is the bounded, opt-in, recommend-when-warranted version.
- **D** — viable but premature; the isolation case is real-but-unproven and softened by slice-fetching. Decision 8 names the exact trigger to revisit.
- **E (do-nothing)** — keep RFC-0041 alone. Cost of delay: every unfamiliar-platform build repeats the contract-ignorance and verification-gap failures; the human stays the relay (handing the loop a platform skill, demanding a probe) — observed in **two** independent builds, so the pain is recurring and confirmed, not hypothetical.

Prior art for the *content* of each stage is grounded in the IaC oracles above plus operational taxonomy (AWS Well-Architected, Google SRE black-box monitoring + "created ≠ works", the Day-1/Day-2 split) and the agent-guardrails finding that schema validation "turns an open-ended action into a constrained interface, reducing catastrophic failure when models hallucinate arguments." Full citations in `Evidence & prior art`.

## Risks & what would make this wrong

**Pre-mortem.**
- *Scope sprawl — six families is too much for one RFC.* Mitigation: they share one thesis (memory/proxy vs. ground truth) and one delivery (references on the existing infra-flavored load path); each family is a small prose addition, and the field report's MECE argument (bounded by the inner-loop-stage axis) is what makes the set finite rather than open-ended.
- *Tool/vendor-specific drift.* Mitigation: A1 is tool-keyed with the spectrum table first-class; an adversarial pass checks for vendor-binding; the weak-oracle row + tier-honesty are mandatory.
- *Box-ticking* (contract "acquired: yes"; craft "considered: yes"). Mitigation: A1's output is a *cited contract slice* the build references; A2 is reviewed by `quality-engineer` against the deployed reality, not a checklist tick.
- *`quality-engineer` window bloat* from oracle output + a second module. Mitigation: contract fetched in slices; Decision 8 names the dedicated-agent escape hatch on evidence.
- *V2 probe becomes standing infrastructure / cost leak.* Mitigation: probe self-teardown is part of the decision and rides RFC-0041 P1's teardown artifact.
- *A2 craft overlaps `security-checklists` (permissions) and the P1 scanner.* Mitigation: stated carve — `security-checklists` owns *over*-permissioning (is this too open?); `cloud-implementation-craft` owns *under*-permissioning + timing/retry (will the call path even succeed?); the scanner checks config-against-policy; contract-acquisition checks IaC-against-the-platform's-structural-contract. Four distinct questions, named in front matter.

**Key assumptions (falsifiable).**
- *Infra work reliably routes to full mode and is detected as infra-flavored* — else the references don't load. (Believed true; RFC-0041 established both.)
- *Mature toolchains expose a drivable contract oracle* — else T1 is empty. (True for strong/medium tiers; the weak-oracle row + runtime-probe fallback is the explicit answer when false.)
- *Loading `operational-safety` into the EXECUTE brief helps the implementer, not just inflates context* — if craft prose doesn't change build behavior, A2 is dead weight. (Believed true; the field-report failures are exactly what the prose names — but this is the assumption most worth watching post-ship.)
- *`quality-engineer` can hold contract + craft without a dedicated window.* (Believed true given slice-fetching; Decision 8 is the pre-named escape hatch.)

**Drawbacks.**
- A third orchestrator-loaded reference area to maintain, EXECUTE-brief growth on infra work, and longer `work-loop` prose — real surface-area cost, justified by a failure class confirmed across two independent builds.
- The new skill is protocol-shaped where the two existing libraries are checklist-shaped — a small family inconsistency.
- For single-familiar-platform adopters, the tiering and some craft is partly dormant (mitigated: loads only on infra-flavored work; T1/V2 run the validator/probe they'd run anyway).

## Evidence & prior art

- **Spike / de-risk result.** The riskiest *factual* assumption — "the toolchain exposes the immutable-field contract machine-readably" — was checked and found **true but heterogeneous**, shaping A1: CFN `createOnlyProperties` and Pulumi `replaceOnChanges` are in their schemas, but **Terraform's `providers schema -json` does not expose force-new** — *verified by fetching* the HashiCorp `terraform providers schema` reference, which documents only `type`/`description`/`required`/`optional`/`computed`/`sensitive`. So the protocol reads Terraform's replace signal from `plan` + docs (encoded in T1). The riskiest *design* assumption — "this is a habit, not tooling" — holds: prose + a command reference reusing RFC-0041's already-firing module-routing mechanism clears Principle 3 as `operational-safety` did.
- **Repo precedent.** RFC-0041 (the scaffold this grounds; P1 preflight extended by a fifth artifact (durable-credentials-once), P2 smoke refined by V1/V2, `observability-and-smoke` strengthened by D, no-new-reviewer ceiling honored); RFC-0029 + `security-checklists`/`operational-safety` (the orchestrator-loaded depth-library pattern); AGENTS.md "Grep to verify a function exists before importing it" (A1's grounding gate generalizes it); the 3-tier dependency policy (T2's detect-and-recommend); `design-reviewer` RFC-0032 (the spec-stage home for the contradiction class); CHARTER Principles 1–4.
- **External prior art.** *Deliberately drawn beyond the two mined builds, across stacks and against industry research, so the taxonomy isn't over-fit to two cases.*
  - *Contract oracles (A1).* [Terraform `providers schema`](https://developer.hashicorp.com/terraform/cli/commands/providers/schema) (fetched: six attribute fields, no force-new); [Terraform `lifecycle` / forces-replacement](https://developer.hashicorp.com/terraform/language/meta-arguments/lifecycle); CloudFormation [`createOnlyProperties`/`primaryIdentifier` resource-type schema](https://docs.aws.amazon.com/cloudformation-cli/latest/userguide/resource-type-schema.html); [Pulumi `package get-schema`](https://www.pulumi.com/docs/iac/cli/commands/pulumi_package_get-schema/) (`replaceOnChanges`).
  - *MECE cross-check (independent axis).* [A Defect Taxonomy for Infrastructure as Code: A Replication Study](https://arxiv.org/abs/2505.01568) (fetched: a replication of a Puppet-based original taxonomy across PL-IaC tools — Pulumi, Terraform CDK, AWS CDK; **3,364 defect-related commits / 285 OSS repos** qualitative, plus an expanded **447 OSS + 94 proprietary** projects; **eight** defect categories). The abstract does not enumerate all eight, but the categories it names map cleanly onto the six families — *Configuration Data* (the most frequent) → A1/A2 (contract + craft); *Idempotency* → P (provision/convergence); *Security* → adjacent (`security-checklists`) — an independent-axis cross-check that the inner-loop families capture the same failure space, none of the named categories falling outside.
  - *The convergence loop is recognized, not invented (V1/V2).* [Deployability-Centric IaC Generation: Fail, Learn, Refine, and Succeed through LLM-Empowered DevOps Simulation](https://arxiv.org/abs/2506.05623) frames LLM-IaC generation as a fail→learn→refine→succeed loop driven by **deployment-simulation feedback** rather than static linting alone — external prior art for the runtime-truth-over-cheap-proxy thesis. ## Errata

This section is append-only; it records refinements made after acceptance that the spec (`docs/specs/infra-grounding/`) implements, so the spec does not silently diverge from the frozen RFC body above.

- **2026-06-24 — packaging / entrypoint-import model is named explicitly (A1 + A2). Approver: eugenelim.** Re-reading the field report's **Build 1** (the managed agent-runtime + tool-function-gateway build, per the field report's scrubbed phrasing) found that its dominant Author·structural failure was the managed runtime's **deployment-artifact packaging / entrypoint-import model** — the agent authored package-relative and project-root-rooted imports, not knowing the runtime loads the entrypoint from a *flat package root as a script*, which failed at runtime (`ModuleNotFoundError`). The RFC's A1 EXECUTE gate ("before generating a CLI invocation or IaC resource") and A2's craft enumeration did not name this class explicitly. **Two additive refinements (no contradiction with the decisions above):** (1) the A1 EXECUTE contract-grounding gate also fires on **authoring application code that runs on a managed runtime** (a function handler whose packaging / import model the platform dictates), not only on a CLI invocation or IaC resource; (2) the A2 `cloud-implementation-craft` module names the **deployment-artifact packaging & entrypoint / module-resolution model** as a craft area (general habit: confirm how the runtime packages and imports the entrypoint before writing imports — platform specifics still defer to the **T2 curated platform skill**, e.g. `aws-bedrock-agentcore-skill`, never bundled per-vendor data, preserving Principle 1 and Decision 8). This sharpens the existing families; it adds no new family, skill, module, or reviewer.
- **2026-06-24 — infra build & verification scripts externalize their configuration (A2 + the P reusable-script corollary). Approver: eugenelim.** The infra build scripts and the verify / probe / teardown scripts the loop produces take their environment-specific values — **resource name prefixes, region, account / project, tags, stage, and any naming-convention tokens** — from **external configuration** (e.g. Terraform `*.tfvars` / `TF_VAR_*`, CDK context / `cdk.json` / env, a sourced env file, a parameters file), **never hardcoded inline**. The named per-tool mechanisms are illustrative, not normative (Principle 1). Two benefits make this load-bearing, not cosmetic: (1) it lets a build honour an **organization's naming and tagging conventions** without editing script bodies, and is the construction-side complement to A1's naming-regex contract; and (2) it makes the reusable-script harness (Decision 4) able to **stand up like-for-like environments** — an identical stack instantiated under a different prefix / config for an ephemeral test run, a per-developer sandbox, or a staging mirror — the property V2's *ephemeral, uniquely-named* probe target relies on and that `operational-safety`'s `environment-isolation` per-PR ephemeral harness builds on. Additive: no new family / skill / module / reviewer.
