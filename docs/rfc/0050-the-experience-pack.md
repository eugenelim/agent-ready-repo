## The ask

**Recommendation (BLUF).** Grow the catalogue's design/UX seat into the **`experience` pack** ‚Äî rename `design-craft ‚Üí experience`, add the **connective** UX skills (`map-customer-journey`, `blueprint-service`, `inventory-screens`) that turn outcomes into journeys into screens into backing services (and, per D9, `map-internal-process` for internal business process flows), **ground** `aesthetic-direction` and add a **taste mode** to `design-critique`, and **wire `voice-and-microcopy`** (which stays in `product-engineering`) to the screen inventory. All **pure-markdown skills, no runtime** ‚Äî the `design-craft` anti-infrastructure posture (ADR-0024) preserved. This is **child-1 of RFC-0048**; that RFC already *decided* this set (Decision 3), so this RFC models it to buildable depth and runs the series lifecycle, it does not re-open those choices.

**Three questions were held live for this RFC's review (its Open status was held for exactly this ‚Äî pressure-test before convergence); all three are now resolved:** **(D7 ‚Äî adopted)** ship a forked-context **UX/design-lens reviewer agent** (`experience-reviewer`) ‚Äî admissible under the loop/work-type agent-addition policy (**ADR-0042**, Proposed ‚Äî supersedes ADR-0023 on its acceptance), the resolution of **ADR-0024's deferred OQ#2**, and the correction of RFC-0048's roster asymmetry (security + reliability each got a forked design-time reviewer; UX/design did not ‚Äî RFC-0048:235). Adopted because the operating model is **agent-autonomous where possible, with human review reserved for the human-value-add judgment gates** ‚Äî so the design review between those gates must run without a human, and a same-session skill cannot (it marks its own homework). **(D8 ‚Äî adopted)** `inventory-screens` emits an optional **design-tool handover** for generative design tools (Figma AI, Claude, v0, ‚Ä¶). **(D9 ‚Äî adopted)** add an **internal business process-flow** capability (current/target-state, L1/L2/L3) as a **separate skill `map-internal-process`** in `experience`; the journey skill is named **`map-customer-journey`** so the two read as one outside-in / inside-out mapping family ‚Äî pressure-tested and confirmed in ¬ß D9. The **autonomous-operation substrate** (a per-initiative agent environment triggered off a backlog ‚Äî Hermes is *one example*, not the codified target) is the *deployment target the skills are authored to run in*, not something this pack builds (Principle 3, RFC-0048:38).

**Why now (SCQA).** *Situation:* `design-craft` (RFC-0033) ships four framework-agnostic craft skills ‚Äî `aesthetic-direction`, `design-system-foundations`, `layout-and-information-architecture`, `design-critique` ‚Äî that design *a* screen. *Complication:* RFC-0048 traced a worked example end-to-end and found the **connective UX layer missing** ‚Äî nothing turns an outcome into a journey, a journey into a screen inventory, or a screen into its backing service ([`0048-notes/04`](0048-notes/04-artifact-inventory.md)); `voice-and-microcopy` is words-only and not tied to any screen list; and the pack's name undersells the seat. *Question:* can we add the connective layer (and rename the seat) while honoring the pack's **stack-neutrality** guardrail (no values tables) and its **anti-infrastructure posture** (ADR-0024 ‚Äî no hook/engine/validator), and keep the pack standalone-useful? *(ADR-0024's "all-skills-zero-agents" v1 minimalism is the one posture this RFC deliberately revisits ‚Äî see D7, decided within ADR-0042's agent-addition policy.)*

## Problem & goals

**Diagnosis.** `design-craft` designs *a* screen well but cannot answer the questions a product team asks *before* a screen exists:

- **No journey.** Nothing maps the outcome to the stages and actions a user moves through ‚Äî so screens get invented, not derived.
- **No screen‚Üîservice tie.** Nothing connects a screen's actions to the backing services that fulfil them, so a screen can be designed that nothing can build (RFC-0048's reconciliation gap).
- **No enumerated, sequenced screen set.** `design-craft` has no notion of "the screens this product needs," their order, or their per-screen states ‚Äî so `voice-and-microcopy` has nothing to attach copy to (GAP-C1), and there is no unit a per-screen generation step can consume.
- **The name undersells the seat.** "design-craft" reads as visual polish; the seat is the whole **experience** ‚Äî flow, service, screen, taste, and words.

**Goals.**
- Add the **connective layer** (journey ‚Üí screen inventory ‚Üí service blueprint) as pure-markdown intent specs the build and the tech packs consume.
- **Ground taste** so aesthetic judgment runs against a stable referent (persona + precedent + standards + platform), not fresh opinion (Kahneman: valid intuition needs a stable referent ‚Äî [`0048-notes/05`](0048-notes/05-judgment-decomposition-and-phases.md); primary citation in RFC-0048 ¬ß Evidence).
- Emit a **per-screen brief** self-contained enough to generate one screen in isolation, yet carrying the connective context so the whole stays coherent ([`0048-notes/07`](0048-notes/07-screen-brief-format.md)).
- Keep every skill **standalone-useful** (detect-and-degrade on the upstream product artifacts) and **stack-neutral**. The connective skills (D2) add **zero agents** ‚Äî they are pure-markdown; *whether the pack adds one forked-context reviewer agent is the separate, live D7 call* (decided within ADR-0042's agent-addition policy, not foreclosed by a blanket no-agents goal).

**Non-goals** (could-have-been-goals, deliberately dropped):
- ~~**A new reviewer agent.**~~ **Promoted to live decision D7 (pressure-test, Finding A).** The original draft listed "no new reviewer agent" as a non-goal, reasoning it would offend ADR-0024's posture and the reviewer ceiling. That reasoning was unsound: **ADR-0023 scopes the ceiling to *core-loop* code-review lenses only** and explicitly admits opt-in, different-surface, different-cadence reviewers (architect's `design-reviewer` is the first one admitted). A UX reviewer is therefore *not forbidden* ‚Äî the real question is *worth-it-and-at-what-autonomy-level* (D7). What stays out regardless: a **fourth core `work-loop` code-review lens** ‚Äî that remains capped (and the superseding ADR keys the cap explicitly to the loop type).
- **An install-time pack alias.** No pack-level rename/alias field exists (grep-confirmed, RFC-0048); inventing one is a distribution-mechanism RFC, not this one. We follow the `contract-acquisition` precedent: rename + erratum, no alias.
- **Pixel comps / Figma files *authored by this pack*.** Agents author design **intent specs** (markdown/mermaid/tables); a designer or UI-codegen realizes them (the `design-craft` framing, unchanged). **D8 narrows this**, it does not breach it: the pack may author a *handover instruction set* that a generative design tool consumes, but never the comp/pixels itself ‚Äî realization stays the tool's job.
- **The autonomous-agent runtime.** RFC-0048's "no new orchestration runtime / daemon / service" (RFC-0048:38) stands. An autonomous-agent runtime (one example is a Hermes-style per-initiative sandbox; a workflow engine or a CI matrix are others) is a *deployment target* the adopter chooses ‚Äî never codified here. This pack is **authored to run inside one** (self-contained artifacts, layout-resolved paths, detect-and-degrade, and ‚Äî via D7 ‚Äî an agent-runnable coherence check that doesn't need a human in the loop), but builds none of it (Principle 3).
- **Values tables.** No fixed breakpoints, HIG spacing, or Material token sets reprinted ‚Äî the platform axis *points to* the conventions, the method derives the values (RFC-0033's guardrail).
- **The traceability lint / Domain Framing + Scope Boundary / self-coverage gate.** Those are RFC-0048's sibling child efforts (Decisions 4‚Äì6); this RFC *consumes* their contracts (the per-screen brief's traceability columns) but builds none of them.
- **Building the discovery loop or `discovery-lead`.** The connective skills are invoked by a human designer today and by `discovery-lead` once it ships (RFC-0048 D8) ‚Äî this RFC makes them work either way.

### D2 ‚Äî the three connective skills

All three are **pure-markdown SKILL.md + `references/` + `assets/`**, no adapter primitives, no agents ‚Äî so they project to every adapter `design-craft` already supports. Each is **standalone-useful**: it elicits its inputs inline when the upstream product artifacts (persona, outcomes, intents) are absent, and **degrades** when downstream packs (`architect`, `contracts`) are not installed (it names services/components textually rather than emitting C4/contracts).

| Skill | Produces | Grounded in | Consumed by |
| --- | --- | --- | --- |
| **`map-customer-journey`** | a **customer/end-user journey map** ‚Äî stages √ó actions / emotions / pains / opportunities (frontstage, outside-in), with the **platform/surface axis** | NN/g journey mapping; Patton user-story mapping; Torres opportunity-solution tree ([`0048-notes/01`](0048-notes/01-research-consolidation.md)) | `inventory-screens`, `blueprint-service` |
| **`blueprint-service`** | a **service blueprint** ‚Äî frontstage / line-of-visibility / backstage / support ‚Äî the screen‚Üîservice tie | NN/g service blueprints ([verified](https://www.nngroup.com/articles/service-blueprints-definition/)) | `architect` (backstage ‚Üí C4 + contracts); the spec LLD |
| **`inventory-screens`** | a **screen inventory + per-screen state matrix** (states **defer to the shared handle-all-states floor**), the **platform/surface axis**, and **one per-screen brief per screen** | the screen briefs ride the `0048-notes/07` template; states from the shared handle-all-states floor (today `design-critique/references/quality-floor.md`) | `aesthetic-direction` / `design-system-foundations` / `layout-and-information-architecture`; `voice-and-microcopy` |

`map-customer-journey` ‚Üí `inventory-screens` ‚Üí `blueprint-service` is the natural order, but each runs alone. `blueprint-service`'s backstage column is the **slicing instrument** RFC-0048 names (each backstage service ‚Üí a component); its hand-off to `architect`/`contracts` is by-reference (a named service), not an import. `map-internal-process` (D9) is the **inside-out** sibling of this outside-in chain ‚Äî it shares `blueprint-service`'s backstage seam by-reference but runs on its own ontology (see ¬ß D9).

**Shared handle-all-states floor.** The canonical state set lives today in `design-critique`'s `references/quality-floor.md` (the floor RFC-0033 shipped: empty / loading / error / success / partial / disabled). With three more skills now consuming it, the implementing spec **promotes it to a pack-shared reference** (the natural home is a pack-level `references/`, the existing cross-skill-reference pattern `aesthetic-direction` already uses), so `inventory-screens` and `voice-and-microcopy` defer to one floor rather than a private sibling reference. `permission/denied` is an **additional gated-screen state** (named in `0048-notes/07`), not a replacement list ‚Äî it extends the floor, it does not fork it.

### D3 ‚Äî the platform/surface axis

`map-customer-journey`, `inventory-screens`, and `aesthetic-direction` each carry a **surface** value: `responsive-web | iOS | Android | cross-platform`. (`map-internal-process` does **not** carry this axis ‚Äî internal process flows are actor/swimlane-shaped, not device-surface-shaped.) The axis changes *what the method asks*, not *what values it prints*:
- **iOS** ‚Üí consult **Apple HIG** ([verified](https://developer.apple.com/design/human-interface-guidelines)) for navigation patterns, gestures, and platform affordances.
- **Android** ‚Üí consult **Material 3** ([verified](https://m3.material.io/)) for its component vocabulary and adaptive-layout guidance.
- **responsive-web** ‚Üí consult **responsive breakpoints / media queries** ([MDN, verified](https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/CSS_layout/Responsive_Design)) and, where installable, **PWA** conventions.
- **cross-platform** ‚Üí design the shared intent once, then name the per-surface adaptations (the journey and screen list are shared; chrome/navigation/gestures adapt per surface).

The skills **point to** these standards (the way `design-craft` already points to WCAG / W3C tokens) and **never reprint** breakpoint tables or HIG spacing ‚Äî the stack-neutral guardrail (RFC-0033) is preserved by construction.

### D4 ‚Äî the per-screen brief

`inventory-screens` emits, per screen, the brief in [`0048-notes/07`](0048-notes/07-screen-brief-format.md) (its `assets/` carries the template). The brief is a **split**:
- a **shared design contract** ‚Äî authored once per product (by `aesthetic-direction` + `design-system-foundations` + the navigation model + the shared handle-all-states floor) and **referenced**, never copied, by every screen brief; this is what keeps N independently-generated screens coherent;
- a **per-screen spec** ‚Äî this screen only (job, states, data, actions, copy pointer).

Coherence is enforced by three things, only one of which this RFC ships: (a) every brief references the *same* contract (shipped here); (b) the **traceability lint** (RFC-0048 D6, sibling effort) checks every action ‚Üí a named service and every screen ‚Üí a journey step; (c) `design-critique` reviews each generated screen against the contract and its neighbours (the taste mode, D5).

**Forward idea picked up (notes/07 ¬ß Consistency & prototyping; sample-bank notes/09).** `inventory-screens`' procedure ends not at "briefs emitted" but at a **cross-brief consistency pass** (shared components reused, states uniform, copy voice aligned, no contradictory navigation) and then **reaches for the cheapest whole-journey verification available** *before* G4: trigger a wireframe/prototyping tool via MCP to assemble a low-fi clickable prototype and walk the journey end-to-end, **or**, when no such tool is present, build a **text-only steel thread** (a scripted walk through the briefs in journey order asserting every transition resolves and every action has a backing service). The skill body teaches the model to reach for this unprompted ‚Äî the modelling discipline RFC-0048's sample-bank logs, not just a feature. **Who runs the consistency pass (pressure-test, Finding D ‚Äî resolved):** under agent-autonomous operation the in-session author cannot independently check its own N briefs, so the fresh-context cross-brief review is **D7's `experience-reviewer`** (now adopted); at a human-value-add gate, the human reviews. The pass has a named runner.

### D5 ‚Äî grounded `aesthetic-direction`, taste-mode `design-critique`

**`aesthetic-direction` (‚äï).** Today it converges a felt vibe into ranked named goals. The enhancement **grounds** those goals in a stable referent ‚Äî the **persona** (from the Domain Framing, or elicited inline if absent), **precedent** (a brief survey of comparable products' taste, refreshed only when the loop needs it), recognized **standards**, and **platform conventions** (D3). The recorded direction names *what grounds each goal*, so a later choice points back to a referent, not a fresh opinion. It stays method-not-values (no palette/font/value printed).

**`design-critique` (‚äï).** Today it runs a heuristic + quality-floor evaluation. The enhancement adds a **taste mode**: an evidence-grounded critique of a screen against the **grounded aesthetic reference** + **platform fit**. **Correction (pressure-test, Finding C):** an interactive skill invoked in the *authoring* session is **authoring-time self-review** ‚Äî it is **not** "fresh-context" and **not** an adversarial pass; a same-session critique marks its own homework (the standing anti-pattern named in architect's `design-reviewer` preamble). The genuine fresh-context UX review is **D7's forked-context `experience-reviewer`**, if adopted; `design-critique`'s taste rubric is exactly what that agent (and RFC-0048 O5) reuses. **Absent D7**, the fresh-context guarantee comes only from a human at G2 or a deliberate fresh-session invocation ‚Äî not from this skill running inline. `design-critique` itself stays an **interactive authoring-time skill**; the reviewer question is D7.

### D6 ‚Äî path resolution (config ‚Üí default ‚Üí discover-by-marker)

The connective skills write durable artifacts; their paths resolve in **three tiers, never hardcoded** (RFC-0040 / ADR-0030, mirroring `product-engineering`'s `frame-intent`):

1. **Config** ‚Äî the `[experience]` table of the adopter-owned `agentbundle-layout.toml` (repo-root overrides user-profile), `parent = "<base>"`.
2. **Designed default** ‚Äî `parent = "docs/design"` (the pack's `[pack.layout.repo]` default; committed design docs, the natural home, paralleling `product-engineering`'s `docs/product`). No `[pack.layout.user]` default ‚Äî output is per-repo, like `product-engineering`.
3. **Discover-by-marker** ‚Äî if neither resolves and the adopter chose a different layout, search the workspace for the artifact's canonical filename + frontmatter `type:` (`type: customer-journey` / `service-blueprint` / `screen-inventory` / `process-flow`).

File-per-slug shapes under `<parent>`:
- `map-customer-journey` ‚Üí `<parent>/journeys/<slug>.md`
- `blueprint-service` ‚Üí `<parent>/blueprints/<slug>.md`
- `inventory-screens` ‚Üí `<parent>/screens/<slug>-inventory.md` + per-screen briefs `<parent>/screens/<slug>/<screen-name>.md`
- `map-internal-process` (D9) ‚Üí `<parent>/processes/<slug>.md` (carries both as-is and to-be, frontmatter `type: process-flow`)

A `references/agentbundle-layout.md` in each new skill documents the table (the `product-engineering` pattern); the pack ships the `[pack.layout.repo]` default in `pack.toml`; `init-project`/`adapt-to-project` seed the config; each skill creates its dir lazily on write and surfaces the resolved path before the first write.

### Cross-pack edit ‚Äî `voice-and-microcopy` (D5/GAP-C1)

`voice-and-microcopy` **stays in `product-engineering`** (its home; RFC-0030). The edit: its procedure learns to **consume the screen inventory's per-screen state matrix** ‚Äî when a screen inventory is present, it writes copy **per screen √ó state** keyed to the inventory, rather than ad-hoc; absent one, it behaves as today (detect-and-degrade). The `experience` README cross-links to `voice-and-microcopy` (and vice-versa), so the seat reads as one even though words live in PE. `product-engineering` takes a **minor bump (`0.6.0 ‚Üí 0.7.0`)** for this.

### D7 ‚Äî the UX/design-lens reviewer agent (ADOPTED)

*Surfaced by the pressure-test; resolved in this RFC's review ‚Äî adopt.*

**The roster asymmetry.** RFC-0048's lens-team roster (RFC-0048:230‚Äì239, "authoritative for the roster") gives every design-time discipline *except one* an independent reviewer:

| Design-time discipline | Reviewer primitive | Source |
| --- | --- | --- |
| Security / compliance | `discovery-threat-reviewer` ‚Äî **new forked agent** | RFC-0048 |
| Quality / reliability | `discovery-reliability-reviewer` ‚Äî **new forked agent** | RFC-0048 |
| Architecture | `design-reviewer` ‚Äî **forked agent (reused)** | architect |
| **UX / design** | **a skill, *not* a reviewer agent** (`design-critique`, ‚Ä¶) | RFC-0048:235 |

UX/design is the **only** design-time lens with no forked-context reviewer. The justification was ADR-0024's posture + "the ceiling." But the threat and reliability lenses got forked agents *for exactly the reason a UX lens needs one* ‚Äî design-time independence, a critique that doesn't mark its own homework.

**Admissibility.** ADR-0023 (and its superseding ADR) scopes the ceiling to **core `work-loop` code-review lenses**. An `experience-reviewer` is opt-in (pack-installed), reviews a **different surface** (journeys/screens/briefs, not code diffs), and runs at a **different cadence** (design-time, when the pack is invoked). By ADR-0023's own three-part test it is **admissible** ‚Äî the same category as architect's `design-reviewer`. It is also the resolution of **ADR-0024 OQ#2** ("whether design-craft eventually ships a `design-reviewer` subagent ‚Ä¶ a later RFC if ever") ‚Äî and *this is that later RFC*.

**The crux was autonomy level, not permission ‚Äî and it is now settled.** The operating model is **agent-autonomous where possible, with human review reserved for the human-value-add judgment gates** (the value/conflict calls ‚Äî G0 vision, G1.5 MVP boundary, G2 "what" + conflict adjudication, and one-way-door decisions). The design review that runs *between* those gates therefore executes with **no human in the loop**:
- A same-session `design-critique` cannot be that review ‚Äî it runs in the authoring context and marks its own homework; the cross-brief consistency pass (D4) would have no independent runner.
- So the independent design check is **the forked-context `experience-reviewer`** ‚Äî load-bearing, not optional, exactly as the discovery loop's threat and reliability reviewers are load-bearing for *their* autonomous review.

`design-critique` stays the authoring-time skill; the `experience-reviewer` is the fresh-context reviewer that lets the design step run autonomously between human-value-add gates.

**Name ‚Äî collision-hardened (pressure-test, naming-collision check).** The obvious `experience-design-reviewer` **embeds `design-reviewer`** (architect's) as a substring ‚Äî the exact trap RFC-0048:246 names, because reviewers are dispatched by **role-match** and exact-name dispatch is not available on every adapter. The name is therefore **`experience-reviewer`**: a distinct head (`experience`, the seat name ‚Äî mirroring why architect's is pack-keyed `design-reviewer`) that is not a substring of any core or discovery reviewer. That rules out the *substring* trap; it does **not** by itself rule out **role-match attraction on the shared `reviewer` token** (the second collision mode RFC-0048:261 names ‚Äî every reviewer shares "reviewer"). That mode is handled by the **name-independent guard** (RFC-0048:249): the `description:` **leads with a design-time-only cue** ‚Äî *"Design-time / experience ONLY ‚Äî reviews journey / screen-inventory / per-screen-brief / aesthetic artifacts; never code diffs (use `core`'s reviewers) and never architecture design docs (use architect's `design-reviewer`)"* ‚Äî exactly as `discovery-threat-reviewer` disambiguates itself. Both guards, per ADR-0042 conjunct 4.

**What it reviews / degrades to.** Reviews the journey, screen inventory + per-screen briefs, and a generated screen against the **grounded aesthetic reference** (D4's shared design contract) + **platform fit** (D3) + **cross-brief coherence** (the D4 consistency pass). Forked-context, read-only, flags-never-rewrites (the `design-reviewer` shape). Absent the pack, the discipline degrades to `discovery-lead`/the human holding the lens ‚Äî exactly how RFC-0048's roster degrades the architecture lens.

**Cross-integrity.** Adopting D7 **amends RFC-0048's roster** (line 235: UX/design moves from *skill, not reviewer* to *skill + an opt-in forked reviewer*) ‚Äî logged as an Amendment in RFC-0048 (both RFCs are Open; reconciled in this same effort).

**Decision-weight impact.** D7 alone moves this RFC from *standard* toward *heavy*: a new shipped agent (a public primitive), an eval surface (trigger + judge, per RFC-0037), and a foundation-RFC amendment.

**Decision ‚Äî adopted.** The `experience-reviewer` clears **all four conjuncts of ADR-0042's test**: (1) a different loop/work-type from the core code-review gate (design-time review, not code diffs); (2) the unique-value test ‚Äî forked-context **independence** (the same-session skill marks its own homework); (3) the **charter's four principles** ‚Äî it stays pure-markdown method, standalone-useful, agnostic, and adds no runtime/hook/validator (the pack's existing posture, unchanged); (4) **collision-hardened** by construction (name + `description` cue, above). It resolves ADR-0024 OQ#2.

**One honest dependency.** ADR-0042 makes the reviewer *admissible*; what makes it *load-bearing* (rather than admissible-but-optional) is the **agent-autonomous-where-possible posture** ‚Äî and that posture lives in RFC-0048 + this RFC, **both Open**. So precisely: D7 is **adopted as load-bearing given that posture**; if the posture is not ratified at acceptance (the design gate stays human-ratified every time), D7 falls back to *admissible-but-optional* per ADR-0042 ‚Äî still permitted, just not required. The pre-mortem's falsifier (¬ß Risks) carries this. Remaining work is mechanical: the eval surface and the RFC-0048 roster amendment (flipped, below).

### D8 ‚Äî the optional design-tool handover (ADOPTED)

*New thread ‚Äî the experience‚Üírealization seam.*

`inventory-screens` (and its per-screen briefs) can emit, **optionally**, a **design-tool handover**: a structured, machine-consumable instruction set ‚Äî keyed to each per-screen brief (job ¬∑ states ¬∑ layout intent ¬∑ navigation ¬∑ copy pointer ¬∑ platform surface ¬∑ the grounded aesthetic reference) ‚Äî that a **generative design tool** consumes to realize a low- or high-fidelity design.

- **Detect-and-degrade.** If a design-tool MCP is connected (e.g. Figma AI, Claude, v0, or another generative-UI tool), the skill **triggers it** and walks the result (extending D4's "trigger a wireframe/prototyping tool via MCP" note from low-fi prototyping into realization). If none is present, it **emits the handover file** for the adopter to paste into whichever tool they use. No tool is required, none is endorsed ‚Äî the skill names tool *categories*, not a winner.
- **Stays method, never pixels.** The handover is **instructions**, not a comp. This narrows the pixel-comps non-goal without breaching it (and without touching the agnosticism guardrail ‚Äî it points to the brief and the grounded reference, reprints no values).
- **Why it matters under autonomy.** In an autonomous environment there is no human to open Figma; the handover is the machine-readable seam that lets the realization step run (by an MCP tool) or queue (as a file) without a human in the loop.
- **Path.** Rides the per-screen brief ‚Äî `<parent>/screens/<slug>/<screen-name>.handover.md` (or a `handovers/` sibling), resolved through the same D6 layout contract.

**Recommendation.** Adopt ‚Äî it is low-cost (one optional artifact + a detect-and-degrade branch), method-preserving, and it is the seam the autonomous operating model needs.

### D9 ‚Äî internal business process flows: current-state / target-state (ADOPTED)

*New thread ‚Äî the pack maps customer-facing journeys but had no **internal business** process view. Main-thread research (fetched-and-confirmed, see ¬ß Evidence) settled that this is a real, distinct gap. Decided: a separate skill **`map-internal-process`** in **`experience`**, with the journey skill named **`map-customer-journey`** so the pair reads as one outside-in / inside-out family.*

**The gap is real, with a boundary condition.** A **service blueprint**'s backstage/support rows (`blueprint-service`, D2) describe internal activity *anchored to a customer touchpoint* ‚Äî they cover a process **triggered by and resolved within** a single customer interaction. They are **not** an internal business process flow when the process **spans many customer touchpoints** (e.g. "manage a claim" over weeks), **has no customer-facing layer** (e.g. "run the monthly close", "process supplier invoices"), or is being captured **as a constraint on the solution** rather than as a CX output. The repo already names this artifact and lacks a producer for it: `product-engineering`'s `frame-intent/references/current-state-inputs.md` lists *"Process map (inside-out, L3)"* as a brownfield constraint input ‚Äî distinct from the *"Journey map (outside-in)"* ‚Äî with no skill that authors it.

**The skill (`map-internal-process`).** Pure-markdown, agnostic ‚Äî points to standards, reprints none:
- **Levels** ‚Äî APQC PCF vocabulary (points to `apqc.org/pcf`, never reprints the framework). *Correction from the main-thread research:* APQC numbers **Category as L1** (13 of them), **Process Group L2**, **Process L3**, **Activity L4**, **Task L5**. The skill anchors on the **L3 process** (the named flow: trigger ‚Üí outcome ‚Üí roles ‚Üí steps) and decomposes to **L4 activities** (where cross-functional handoffs and decision gates ‚Äî the swimlane content ‚Äî live); **L5 tasks are SOP/work-instruction territory, out of scope**.
- **Two states** ‚Äî **as-is** ("current") and **to-be** ("target"), with an as-is‚Üíto-be **delta table** (`| Step | As-is | To-be | Rationale |`) ‚Äî the analytical output, not just the diagram.
- **Capture methods** ‚Äî synthesize from **job aids / SOPs / work instructions** first (document analysis, flagging cross-source conflicts), then validate via **SME workshop**: process walkthrough (default), JAD session (multi-stakeholder/cross-department), gemba walk (high-exception processes). These are the standards-pointing equivalents of `map-customer-journey`'s NN/g citation.
- **Artifacts** ‚Äî a **SIPOC** scoping table (bounds the process first), a **swimlane** flow in mermaid (`flowchart` with `subgraph` lanes per actor ‚Äî BPMN *vocabulary* at markdown fidelity), and a **pain/waste register** (the bridge from as-is to to-be).
- **Notation pointer** ‚Äî points to **BPMN 2.0** (OMG / ISO 19510) and cross-functional-flowchart conventions; authors in mermaid; names full BPMN XML and metric-rich VSM as *BPM-tool territory* (not agent-producible as markdown), per the research.
- **Seams** ‚Äî cross-references the service blueprint by-name (when a process is customer-triggered) and is the producer of the `frame-intent` "current-state process map" input.

**Separate skill ‚Äî decided (pressure-tested).** It is not a *mode* of the journey skill: the inputs differ (personas + outcomes vs. job aids / SOPs + SME workshops), the output differs (outside-in stages √ó emotions/pains vs. inside-out swimlane √ó decision gates, as-is/to-be), and the grounding differs (NN/g journey mapping vs. APQC/BPMN/BABOK). Folding them into one skill would conflate outside-in CX with inside-out operations ‚Äî the [ValueFlow "why you need all three"](https://valueflow.co/customer-lifecycle-design/) framing (customer journey ¬∑ service blueprint ¬∑ process engineering as three distinct artifacts) is the practitioner argument for keeping them separate.

**The paired naming ‚Äî decided.** The journey skill is **`map-customer-journey`** and the new one is **`map-internal-process`**. It signals the **outside-in / inside-out axis** as a discoverable family, disambiguates "journey," and costs nothing now (both unshipped ‚Äî no migration/alias/frozen-governance bridge, unlike the D1 pack rename). **Deliberate scoping decision (the caveat, now resolved):** `map-customer-journey` **scopes journey-mapping to the customer/end-user** ‚Äî employee-journey mapping is explicitly out of this skill's v1 (a later mode if ever), so the customer/internal pairing is the clean axis rather than the looser experience/operations one.

**Placement ‚Äî `experience` (the *belongs-here* argument, separate from the *distinct-skill* one).** That it is a distinct artifact (above) proves only that it is a *separate skill*, not *which pack* owns it ‚Äî `product-engineering` is the live alternative (its `frame-intent/current-state-inputs.md` already names the L3 process map). The reason it lands in `experience`: **mapping-family cohesion** ‚Äî it is the inside-out sibling of `map-customer-journey` and shares `blueprint-service`'s backstage seam, so the three read as one mapping family in one seat; splitting it to PE fractures that family across two packs for no gain, since the **by-name seam** (PE's `frame-intent` *consumes* the process map `experience` *produces*) gives PE what it needs without ownership. **This is a deliberate widening of the seat's identity** ‚Äî `experience` now means *customer-facing **and** internal-operations mapping*, not UX-only ‚Äî and is recorded as such, not slid in: it matches service-design practice (journey ¬∑ blueprint ¬∑ process as one discipline) and the agnosticism/anti-infrastructure guardrails are unchanged by it.

**Evidence ‚Äî fetched and confirmed (main thread).** The APQC five-level hierarchy, BPMN-as-`to-be`-notation + swimlanes, the as-is/to-be pair, and the service-blueprint-vs-process-map distinction were verified against `apqc.org`, `iiba.org` (BABOK), `nngroup.com`, and practitioner sources (see ¬ß Evidence). The earlier subagent run could not fetch; this round did, so the D9 claims meet the no-fabricated-sources standard.

**Decision ‚Äî adopted.** A separate skill **`map-internal-process`** in **`experience`**, with the journey skill named **`map-customer-journey`** (customer/end-user-scoped). It carries the platform/surface axis **not** (it is actor/swimlane-shaped); it joins the pack's eval + layout surface (D6 path `<parent>/processes/<slug>.md`).

## Risks & what would make this wrong

**Pre-mortem.**
- *The rename breaks adopters.* `design-craft` is user-scope-default, so a rename touches every repo an adopter opened. **Mitigation:** the `contract-acquisition` precedent (rename + erratum, no alias) is proven; v0.1.1 + pre-stable bounds blast radius; the implementing spec re-aggregates the marketplace and bridges frozen governance in one PR.
- *The new skills become checklist ceremony* ‚Äî a journey map that's a table nobody reads, a brief that's boilerplate. **Mitigation:** the skills require *substantive* per-stage / per-screen content (the RFC-0048 anti-ceremony rule), and `inventory-screens` ends in a real verification step (the consistency pass + steel-thread, D4), not "emitted."
- *The platform axis leaks values* ‚Äî a skill starts printing breakpoints or HIG spacing. **Mitigation:** the stack-neutral guardrail (RFC-0033) is restated in each new skill's anti-patterns; the axis points to standards, never reprints them.
- *We mis-call the autonomy level on D7.* Ship the `experience-reviewer` for a gate a human still ratifies ‚Üí redundant ceremony (and an agent to maintain); skip it for an agent-autonomous gate ‚Üí design ships with **no** independent review (the in-session author marks its own homework). **Mitigation:** tie the D7 decision to the *actual* operating-model autonomy posture, not to a default ‚Äî and note the cost is asymmetric (an unused agent is cheaper than unreviewed autonomous design).
- *D8's handover drifts into authoring pixels / values.* A handover that starts embedding palettes, breakpoints, or a comp breaches the agnosticism guardrail. **Mitigation:** the handover is *instructions keyed to the brief*, names tool categories not values, and is covered by the same agnosticism CI lint as the rest of the pack.
- *D9 widens the seat's identity by accretion.* Folding internal business-process / BA work into `experience` without deciding it stretches a UX/design seat into operations. **Mitigation:** D9's placement (`experience` vs. `product-engineering`) is an explicit, surfaced call ‚Äî not a silent ride-along on the rename.

**Key assumptions (falsifiable).**
- *The connective artifacts are agent-producible as intent specs* (markdown/mermaid/tables), not pixel comps ‚Äî so a skill can author them. If journeys/blueprints genuinely need visual fidelity to be useful, the intent-spec framing is too thin.
- *Taste grounded in persona + precedent + standards is better than ungrounded taste* (Kahneman's stable-referent condition). If the persona/precedent referent is usually absent or unreliable, the grounding is theatre and `aesthetic-direction` should stay as-is.
- *The rename is low-cost at this stage.* If `design-craft` has meaningful install depth already, the no-alias migration is too disruptive and a pack-alias mechanism (deferred) becomes the real blocker.
- *(D7 ‚Äî premise now confirmed) The operating model is agent-autonomous where possible, with human review reserved for the human-value-add judgment gates.* This is the stated direction, so the `experience-reviewer` is load-bearing (it runs the design review between those gates). Falsifier: if in practice a human ends up ratifying *every* design at G2 anyway, the agent becomes redundant ceremony ‚Äî but that would contradict the stated autonomy posture, not confirm a different one.

**Drawbacks.** Three more skills to maintain; two enhanced skills; a cross-pack edit in `product-engineering`; the rename's migration tail and the frozen-governance bridge; added surface in the `experience` README. Mitigated by the standalone-useful + detect-and-degrade posture (no skill *requires* the others) and the one-spec migration.

## Evidence & prior art
