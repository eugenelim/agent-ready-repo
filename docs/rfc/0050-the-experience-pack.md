## The ask

**Recommendation (BLUF).** Grow the catalogue's design/UX seat into the **`experience` pack** — rename `design-craft → experience`, add the **connective** UX skills (`map-customer-journey`, `blueprint-service`, `map-screen-flow`) that turn outcomes into journeys into screens into backing services (and, per D9, `map-internal-process` for internal business process flows), add the **`interaction-design`** craft skill (the behavioral pillar, D10), **ground** `aesthetic-direction` and add a **taste mode** to `design-critique`, and **wire `voice-and-microcopy`** (which stays in `product-engineering`) to the screen inventory. All **pure-markdown skills, no runtime** — the `design-craft` anti-infrastructure posture (ADR-0024) preserved. This is **child-1 of RFC-0048**; that RFC already *decided* this set (Decision 3), so this RFC models it to buildable depth and runs the series lifecycle, it does not re-open those choices.

**Four questions were held live for this RFC's review (its Open status was held for exactly this — pressure-test before convergence); all four are now resolved:** **(D7 — adopted)** ship a forked-context **UX/design-lens reviewer agent** (`experience-reviewer`) — admissible under the loop/work-type agent-addition policy (**ADR-0042**, Proposed — supersedes ADR-0023 on its acceptance), the resolution of **ADR-0024's deferred OQ#2**, and the correction of RFC-0048's roster asymmetry (security + reliability each got a forked design-time reviewer; UX/design did not — RFC-0048:236). Adopted because the operating model is **agent-autonomous where possible, with human review reserved for the human-value-add judgment gates** — so the design review between those gates must run without a human, and a same-session skill cannot (it marks its own homework); its review list carries the full quality floor incl. accessibility (§ D7). **(D8 — adopted)** `map-screen-flow` emits an optional **design-tool handover** for generative design tools (Figma AI, Claude, v0, …). **(D9 — adopted)** add an **internal business process-flow** capability (current/target-state, L1/L2/L3) as a **separate skill `map-internal-process`** in `experience`; the journey skill is named **`map-customer-journey`** so the two read as one outside-in / inside-out mapping family — pressure-tested and confirmed in § D9. **(D10 — adopted)** add the **`interaction-design`** craft skill — the behavioral pillar (feedback + timing, input/validation flow, component state machines, motion & micro-animations, navigation behavior, gesture affordances, cognitive-law fit, perceived responsiveness; references onboarding/search patterns) the visual/structural craft skills don't carry; pure-markdown, points to standards, reprints no values (§ D10). The **autonomous-operation substrate** (a per-initiative agent environment triggered off a backlog — Hermes is *one example*, not the codified target) is the *deployment target the skills are authored to run in*, not something this pack builds (Principle 3, RFC-0048:38).

**Why now (SCQA).** *Situation:* `design-craft` (RFC-0033) ships four framework-agnostic craft skills — `aesthetic-direction`, `design-system-foundations`, `layout-and-information-architecture`, `design-critique` — that design *a* screen. *Complication:* RFC-0048 traced a worked example end-to-end and found the **connective UX layer missing** — nothing turns an outcome into a journey, a journey into a screen inventory, or a screen into its backing service ([`0048-notes/04`](0048-notes/04-artifact-inventory.md)); `voice-and-microcopy` is words-only and not tied to any screen list; and the pack's name undersells the seat. *Question:* can we add the connective layer (and rename the seat) while honoring the pack's **stack-neutrality** guardrail (no values tables) and its **anti-infrastructure posture** (ADR-0024 — no hook/engine/validator), and keep the pack standalone-useful? *(ADR-0024's "all-skills-zero-agents" v1 minimalism is the one posture this RFC deliberately revisits — see D7, decided within ADR-0042's agent-addition policy.)*

## Problem & goals

**Diagnosis.** `design-craft` designs how *a* screen looks and is structured, but leaves two things unaddressed — the questions a product team asks *before* a screen exists, and how the screen *behaves* once it does:

- **No journey.** Nothing maps the outcome to the stages and actions a user moves through — so screens get invented, not derived.
- **No screen↔service tie.** Nothing connects a screen's actions to the backing services that fulfil them, so a screen can be designed that nothing can build (RFC-0048's reconciliation gap).
- **No enumerated, sequenced screen set.** `design-craft` has no notion of "the screens this product needs," their order, or their per-screen states — so `voice-and-microcopy` has nothing to attach copy to (GAP-C1), and there is no unit a per-screen generation step can consume.
- **No behavioral pillar.** The craft skills cover visual taste, system foundations, and structure/wayfinding — but nothing designs *how an interaction behaves*: feedback and its timing, input/validation flow, component state machines, motion & micro-animations, navigation behavior, gesture affordances, cognitive-law fit. Interaction design is a recognized craft pillar with no home in the pack (D10).
- **The name undersells the seat.** "design-craft" reads as visual polish; the seat is the whole **experience** — flow, service, screen, behavior, taste, and words.

**Goals.**
- Add the **connective layer** (journey → screen inventory → service blueprint) as pure-markdown intent specs the build and the tech packs consume.
- **Ground taste** so aesthetic judgment runs against a stable referent (persona + precedent + standards + platform), not fresh opinion (Kahneman: valid intuition needs a stable referent — [`0048-notes/05`](0048-notes/05-judgment-decomposition-and-phases.md); primary citation in RFC-0048 § Evidence).
- Emit a **per-screen brief** self-contained enough to generate one screen in isolation, yet carrying the connective context so the whole stays coherent ([`0048-notes/07`](0048-notes/07-screen-brief-format.md)).
- Add the **behavioral pillar** (D10) — design *how* the interaction behaves (feedback, timing, input flow, component state machines, motion & micro-animations, navigation, gestures, cognitive-law fit), the one craft lens the visual/structural skills don't carry — as pure-markdown method that enriches the per-screen brief.
- Keep every skill **standalone-useful** (detect-and-degrade on the upstream product artifacts) and **stack-neutral**. The connective skills (D2) add **zero agents** — they are pure-markdown; *whether the pack adds one forked-context reviewer agent is the separate, live D7 call* (decided within ADR-0042's agent-addition policy, not foreclosed by a blanket no-agents goal).

**Non-goals** (could-have-been-goals, deliberately dropped):
- ~~**A new reviewer agent.**~~ **Promoted to live decision D7 (pressure-test, Finding A).** The original draft listed "no new reviewer agent" as a non-goal, reasoning it would offend ADR-0024's posture and the reviewer ceiling. That reasoning was unsound: **ADR-0023 scopes the ceiling to *core-loop* code-review lenses only** and explicitly admits opt-in, different-surface, different-cadence reviewers (architect's `design-reviewer` is the first one admitted). A UX reviewer is therefore *not forbidden* — the real question is *worth-it-and-at-what-autonomy-level* (D7). What stays out regardless: a **fourth core `work-loop` code-review lens** — that remains capped (and the superseding ADR keys the cap explicitly to the loop type).
- **An install-time pack alias.** No pack-level rename/alias field exists (grep-confirmed, RFC-0048); inventing one is a distribution-mechanism RFC, not this one. We follow the `contract-acquisition` precedent: rename + erratum, no alias.
- **Pixel comps / Figma files *authored by this pack*.** Agents author design **intent specs** (markdown/mermaid/tables); a designer or UI-codegen realizes them (the `design-craft` framing, unchanged). **D8 narrows this**, it does not breach it: the pack may author a *handover instruction set* that a generative design tool consumes, but never the comp/pixels itself — realization stays the tool's job.
- **Frontend/code realization** — component architecture, state management, the actual UI build. Comparable design+build skill sets fold "turn the design into running frontend code" into the same seat; this pack does **not**, by construction. The realization seam is the **D8 handover** (design → a generative tool) and, for code, the adopter's tech stack / a future tech-realization pack — stack-specific build is the exact `no-values` / `no-runtime` territory the pack's guardrails exclude (it would print breakpoints, framework idioms, and component code). `experience` authors the brief + handover; the build lives on the far side of that seam. *(Surfaced by the pressure-test against an engineering-first skill catalogue that bundles design and build — recording the boundary as a decided non-goal, not an oversight.)*
- **The autonomous-agent runtime.** RFC-0048's "no new orchestration runtime / daemon / service" (RFC-0048:38) stands. An autonomous-agent runtime (one example is a Hermes-style per-initiative sandbox; a workflow engine or a CI matrix are others) is a *deployment target* the adopter chooses — never codified here. This pack is **authored to run inside one** (self-contained artifacts, layout-resolved paths, detect-and-degrade, and — via D7 — an agent-runnable coherence check that doesn't need a human in the loop), but builds none of it (Principle 3).
- **Values tables.** No fixed breakpoints, HIG spacing, or Material token sets reprinted — the platform axis *points to* the conventions, the method derives the values (RFC-0033's guardrail).
- **The traceability lint / Domain Framing + Scope Boundary / self-coverage gate.** Those are RFC-0048's sibling child efforts (Decisions 4–6); this RFC *consumes* their contracts (the per-screen brief's traceability columns) but builds none of them.
- **Building the discovery loop or `discovery-lead`.** The connective skills are invoked by a human designer today and by `discovery-lead` once it ships (RFC-0048 D8) — this RFC makes them work either way.

### D2 — the three connective skills

All three are **pure-markdown SKILL.md + `references/` + `assets/`**, no adapter primitives, no agents — so they project to every adapter `design-craft` already supports. Each is **standalone-useful**: it elicits its inputs inline when the upstream product artifacts (persona, outcomes, intents) are absent, and **degrades** when downstream packs (`architect`, `contracts`) are not installed (it names services/components textually rather than emitting C4/contracts).

| **`map-screen-flow`** | a **screen flow** — the journey's screens **sequenced**, with the transitions between them and the **error/edge flows** (an action fails → which screen/state the user lands in); i.e. the *interaction flow across screens*, not a bare list. It carries the **per-screen state matrix** (states **defer to the shared quality floor**), the **platform/surface axis**, and **one per-screen brief per screen**. *(The screen inventory — the enumerated set — is the spine the flow is drawn over.)* | the screen briefs ride the `0048-notes/07` template; states from the shared quality floor's handle-all-states section (today `design-critique/references/quality-floor.md`) | `aesthetic-direction` / `design-system-foundations` / `layout-and-information-architecture` / `interaction-design`; `voice-and-microcopy` |

`map-customer-journey` → `map-screen-flow` → `blueprint-service` is the natural order, but each runs alone. `blueprint-service`'s backstage column is the **slicing instrument** RFC-0048 names (each backstage service → a component); its hand-off to `architect`/`contracts` is by-reference (a named service), not an import. `map-internal-process` (D9) is the **inside-out** sibling of this outside-in chain — it shares `blueprint-service`'s backstage seam by-reference but runs on its own ontology (see § D9).

**`map-screen-flow` (not `inventory-screens` / `map-screens`).** Earlier names (`inventory-screens`, then `map-screens`) named the thinnest output — the *list* of screens — and hid the actual work. The skill designs the **interaction flow across screens**: the screen sequence, the transitions, the **per-screen states**, and the **error/edge flows** (where a failed action routes). `map-screen-flow` names that, and joins the **`map-*` connective family** (`map-customer-journey`, `map-internal-process`). The artifact is a **screen flow** (`type: screen-flow`) whose spine is the enumerated *screen inventory*.

**The two interaction levels — `map-screen-flow` (macro) vs `interaction-design` (micro).** These must not be conflated, because both touch states, transitions, and errors:
- **`map-screen-flow`** owns the **macro flow *across* screens** — which screens, in what sequence, each screen's state set, and the cross-screen error/edge routing. It is a **connective** skill (derived from the journey), the journey→screens bridge.
- **`interaction-design`** (D10) owns the **micro behavior *within* a screen** — feedback and its timing, inline-validation flow, a transition's back/undo semantics, gesture, cognitive-law fit. It is a **craft** skill that enriches one brief.
- The **shared quality floor** owns the **enumeration** of states (empty / loading / error / success / partial / disabled) both defer to.

So the design discipline is split across three homes by level (enumeration / macro flow / micro behavior), not folded into one `interaction-design` mega-skill — that would collapse the connective→craft pipeline the pack rests on. We stay **deliberately coarser** than maximalist catalogues that fan interaction design into dozens of skills; the bar is design-flow completeness (below), not skill count.

**Design-flow completeness is the bar — not skill count.** This pack is deliberately **coarser** than maximalist design-skill catalogues (which fan one discipline into dozens of skills); our four-question add-a-skill test (RFC-0033) keeps it minimal. The non-negotiable is not granularity but that the **design flow is a complete, walkable thread with no broken link**: outcome/persona (upstream, consumed) → **journey** (`map-customer-journey`) → **screen set + per-screen briefs** (`map-screen-flow`) → **backing services** (`blueprint-service`) → **per-screen design** (the craft skills + `interaction-design`, applied to each brief) → **copy** (`voice-and-microcopy`) → **review** (`design-critique` + `experience-reviewer`) → **realization** (the D8 handover), with `map-internal-process` the inside-out sibling. The thread is *proved complete* by `map-screen-flow`' **steel thread** (§ D4) — the whole-journey walk that asserts every transition resolves and every action has a backing service; that verification is **load-bearing, not optional** (it degrades from an MCP prototype to a text-only walk, never to nothing). The one *upstream* input the pack does not produce — **generative user research** (personas, usability testing) — is a named deferral to its owning effort (§ Open questions), consumed by detect-and-degrade; design-ops and soft-skills are out of charter (Principle 3). Neither is a gap *in the design flow itself*.

**Shared quality floor.** The canonical floor lives today in `design-critique`'s `references/quality-floor.md` — and it is **three commitments, not one**: (1) **handle-all-states** (empty / loading / error / success / partial / disabled), (2) the **accessibility floor** (WCAG pointed-to: perceivable contrast, operable without a pointer, meaning never on one channel, named-for-assistive-tech, forgiving targets/timing), and (3) **motion communicates state — honor reduced-motion**. With three more skills now consuming it, the implementing spec **promotes the whole file to a pack-shared reference** (the natural home is a pack-level `references/`, the existing cross-skill-reference pattern `aesthetic-direction` already uses), so `map-screen-flow`, `interaction-design` (D10), and `voice-and-microcopy` defer to one floor rather than a private sibling reference. **Promote all three sections, not just the state set** — the earlier framing called this the "handle-all-states floor," which silently narrowed a three-part floor to its first part; the accessibility and reduced-motion commitments must reach across the new skills (and the D7 reviewer's review list) exactly as the state set does, or autonomous design loses its only independent accessibility check. `permission/denied` is an **additional gated-screen state** (named in `0048-notes/07`), not a replacement list — it extends the floor, it does not fork it.

### D3 — the platform/surface axis

`map-customer-journey`, `map-screen-flow`, and `aesthetic-direction` each carry a **surface** value: `responsive-web | iOS | Android | cross-platform`. (`map-internal-process` does **not** carry this axis — internal process flows are actor/swimlane-shaped, not device-surface-shaped.) The axis changes *what the method asks*, not *what values it prints*:
- **iOS** → consult **Apple HIG** ([verified](https://developer.apple.com/design/human-interface-guidelines)) for navigation patterns, gestures, and platform affordances.
- **Android** → consult **Material 3** ([verified](https://m3.material.io/)) for its component vocabulary and adaptive-layout guidance.
- **responsive-web** → consult **responsive breakpoints / media queries** ([MDN, verified](https://developer.mozilla.org/en-US/docs/Learn_web_development/Core/CSS_layout/Responsive_Design)) and, where installable, **PWA** conventions.
- **cross-platform** → design the shared intent once, then name the per-surface adaptations (the journey and screen list are shared; chrome/navigation/gestures adapt per surface).

The skills **point to** these standards (the way `design-craft` already points to WCAG / W3C tokens) and **never reprint** breakpoint tables or HIG spacing — the stack-neutral guardrail (RFC-0033) is preserved by construction.

### D4 — the per-screen brief

`map-screen-flow` emits, per screen, the brief in [`0048-notes/07`](0048-notes/07-screen-brief-format.md) (its `assets/` carries the template). The brief is a **split**:
- a **shared design contract** — authored once per product (by `aesthetic-direction` + `design-system-foundations` + the navigation model + `interaction-design`'s behavioral conventions + the shared quality floor) and **referenced**, never copied, by every screen brief; this is what keeps N independently-generated screens coherent;
- a **per-screen spec** — this screen only (job, states, data, actions, copy pointer).

Coherence is enforced by three things, only one of which this RFC ships: (a) every brief references the *same* contract (shipped here); (b) the **traceability lint** (RFC-0048 D6, sibling effort) checks every action → a named service and every screen → a journey step; (c) `design-critique` reviews each generated screen against the contract and its neighbours (the taste mode, D5).

**Forward idea picked up (notes/07 § Consistency & prototyping; sample-bank notes/09).** `map-screen-flow`' procedure ends not at "briefs emitted" but at a **cross-brief consistency pass** (shared components reused, states uniform, copy voice aligned, no contradictory navigation) and then **reaches for the cheapest whole-journey verification available** *before* G4: trigger a wireframe/prototyping tool via MCP to assemble a low-fi clickable prototype and walk the journey end-to-end, **or**, when no such tool is present, build a **text-only steel thread** (a scripted walk through the briefs in journey order asserting every transition resolves and every action has a backing service). The skill body teaches the model to reach for this unprompted — the modelling discipline RFC-0048's sample-bank logs, not just a feature. **Who runs the consistency pass (pressure-test, Finding D — resolved):** under agent-autonomous operation the in-session author cannot independently check its own N briefs, so the fresh-context cross-brief review is **D7's `experience-reviewer`** (now adopted); at a human-value-add gate, the human reviews. The pass has a named runner.

### D5 — grounded `aesthetic-direction`, taste-mode `design-critique`

**`aesthetic-direction` (⊕).** Today it converges a felt vibe into ranked named goals. The enhancement **grounds** those goals in a stable referent — the **persona** (from the Domain Framing, or elicited inline if absent), **precedent** (a brief survey of comparable products' taste, refreshed only when the loop needs it), recognized **standards**, and **platform conventions** (D3). The recorded direction names *what grounds each goal*, so a later choice points back to a referent, not a fresh opinion. It stays method-not-values (no palette/font/value printed).

**`design-critique` (⊕).** Today it runs a heuristic + quality-floor evaluation. The enhancement adds a **taste mode**: an evidence-grounded critique of a screen against the **grounded aesthetic reference** + **platform fit**. **Correction (pressure-test, Finding C):** an interactive skill invoked in the *authoring* session is **authoring-time self-review** — it is **not** "fresh-context" and **not** an adversarial pass; a same-session critique marks its own homework (the standing anti-pattern named in architect's `design-reviewer` preamble). The genuine fresh-context UX review is **D7's forked-context `experience-reviewer`**, if adopted; `design-critique`'s taste rubric is exactly what that agent (and RFC-0048 O5) reuses. **Absent D7**, the fresh-context guarantee comes only from a human at G2 or a deliberate fresh-session invocation — not from this skill running inline. `design-critique` itself stays an **interactive authoring-time skill**; the reviewer question is D7.

### D6 — path resolution (config → default → discover-by-marker)

The connective skills write durable artifacts; their paths resolve in **three tiers, never hardcoded** (RFC-0040 / ADR-0030, mirroring `product-engineering`'s `frame-intent`):

1. **Config** — the `[experience]` table of the adopter-owned `agentbundle-layout.toml` (repo-root overrides user-profile), `parent = "<base>"`.
2. **Designed default** — `parent = "docs/design"` (the pack's `[pack.layout.repo]` default; committed design docs, the natural home, paralleling `product-engineering`'s `docs/product`). No `[pack.layout.user]` default — output is per-repo, like `product-engineering`.
3. **Discover-by-marker** — if neither resolves and the adopter chose a different layout, search the workspace for the artifact's canonical filename + frontmatter `type:` (`type: customer-journey` / `service-blueprint` / `screen-flow` / `process-flow`).

File-per-slug shapes under `<parent>`:

- `blueprint-service` → `<parent>/blueprints/<slug>.md`
- `map-screen-flow` → `<parent>/screens/<slug>-flow.md` (the screen flow + inventory + state matrix, frontmatter `type: screen-flow`) + per-screen briefs `<parent>/screens/<slug>/<screen-name>.md`

A `references/agentbundle-layout.md` in each new skill documents the table (the `product-engineering` pattern); the pack ships the `[pack.layout.repo]` default in `pack.toml`; `init-project`/`adapt-to-project` seed the config; each skill creates its dir lazily on write and surfaces the resolved path before the first write.

### D7 — the UX/design-lens reviewer agent (ADOPTED)

*Surfaced by the pressure-test; resolved in this RFC's review — adopt.*

**The roster asymmetry** (as it stood when D7 was first surfaced). RFC-0048's lens-team roster (RFC-0048:230–236, "authoritative for the roster") gave every design-time discipline *except one* an independent reviewer:

| Design-time discipline | Reviewer primitive | Source |
| --- | --- | --- |
| Security / compliance | `discovery-threat-reviewer` — **new forked agent** | RFC-0048 |
| Quality / reliability | `discovery-reliability-reviewer` — **new forked agent** | RFC-0048 |
| Architecture | `design-reviewer` — **forked agent (reused)** | architect |
| **UX / design** | **a skill, *not* a reviewer agent** (`design-critique`, …) — *the asymmetry D7 corrects; RFC-0048:236 now records the forked reviewer* | RFC-0048:236 |

UX/design is the **only** design-time lens with no forked-context reviewer. The justification was ADR-0024's posture + "the ceiling." But the threat and reliability lenses got forked agents *for exactly the reason a UX lens needs one* — design-time independence, a critique that doesn't mark its own homework.

**Admissibility.** ADR-0023 (and its superseding ADR) scopes the ceiling to **core `work-loop` code-review lenses**. An `experience-reviewer` is opt-in (pack-installed), reviews a **different surface** (journeys/screens/briefs, not code diffs), and runs at a **different cadence** (design-time, when the pack is invoked). By ADR-0023's own three-part test it is **admissible** — the same category as architect's `design-reviewer`. It is also the resolution of **ADR-0024 OQ#2** ("whether design-craft eventually ships a `design-reviewer` subagent … a later RFC if ever") — and *this is that later RFC*.

**The crux was autonomy level, not permission — and it is now settled.** The operating model is **agent-autonomous where possible, with human review reserved for the human-value-add judgment gates** (the value/conflict calls — G0 vision, G1.5 MVP boundary, G2 "what" + conflict adjudication, and one-way-door decisions). The design review that runs *between* those gates therefore executes with **no human in the loop**:
- A same-session `design-critique` cannot be that review — it runs in the authoring context and marks its own homework; the cross-brief consistency pass (D4) would have no independent runner.
- So the independent design check is **the forked-context `experience-reviewer`** — load-bearing, not optional, exactly as the discovery loop's threat and reliability reviewers are load-bearing for *their* autonomous review.

`design-critique` stays the authoring-time skill; the `experience-reviewer` is the fresh-context reviewer that lets the design step run autonomously between human-value-add gates.

**Name — collision-hardened (pressure-test, naming-collision check).** The obvious `experience-design-reviewer` **embeds `design-reviewer`** (architect's) as a substring — the exact trap RFC-0048 § *Names are collision-hardened by construction* (254–262, substring trap at 259) names, because reviewers are dispatched by **role-match** and exact-name dispatch is not available on every adapter. The name is therefore **`experience-reviewer`**: a distinct head (`experience`, the seat name — mirroring why architect's is pack-keyed `design-reviewer`) that is not a substring of any core or discovery reviewer. That rules out the *substring* trap; it does **not** by itself rule out **role-match attraction on the shared `reviewer` token** (the second collision mode the same RFC-0048 section names at ~261 — every reviewer shares "reviewer"). That mode is handled by the **name-independent guard** (RFC-0048:262): the `description:` **leads with a design-time-only cue** — *"Design-time / experience ONLY — reviews journey / screen-inventory / per-screen-brief / aesthetic artifacts; never code diffs (use `core`'s reviewers) and never architecture design docs (use architect's `design-reviewer`)"* — exactly as `discovery-threat-reviewer` disambiguates itself. Both guards, per ADR-0042 conjunct 4.

**What it reviews / degrades to.** Reviews the journey, screen inventory + per-screen briefs, and a generated screen against the **grounded aesthetic reference** (D4's shared design contract) + **platform fit** (D3) + **cross-brief coherence** (the D4 consistency pass) + **the shared quality floor** — its **handle-all-states**, **accessibility** (WCAG pointed-to), and **reduced-motion** sections (the promoted three-part floor, § D2). The accessibility section is load-bearing here: under the agent-autonomous posture this forked reviewer is the **only independent accessibility check** between human-value-add gates — `design-critique`'s authoring-time floor pass marks its own homework, so a reviewer that omitted accessibility would let autonomous design ship unaudited on the one axis the discipline treats as non-negotiable. Forked-context, read-only, flags-never-rewrites (the `design-reviewer` shape). Absent the pack, the discipline degrades to `discovery-lead`/the human holding the lens — exactly how RFC-0048's roster degrades the architecture lens.

**Cross-integrity.** Adopting D7 **amends RFC-0048's roster** (line 236: UX/design moves from *skill, not reviewer* to *skill + an opt-in forked reviewer*) — already logged as an Amendment in RFC-0048 (both RFCs are Open; RFC-0048:236 records the change as binding on this RFC's acceptance, reconciled in this same effort).

**Decision-weight impact.** D7 alone moves this RFC from *standard* toward *heavy*: a new shipped agent (a public primitive), an eval surface (trigger + judge, per RFC-0037), and a foundation-RFC amendment.

**Decision — adopted.** The `experience-reviewer` clears **all four conjuncts of ADR-0042's test**: (1) a different loop/work-type from the core code-review gate (design-time review, not code diffs); (2) the unique-value test — forked-context **independence** (the same-session skill marks its own homework); (3) the **charter's four principles** — it stays pure-markdown method, standalone-useful, agnostic, and adds no runtime/hook/validator (the pack's existing posture, unchanged); (4) **collision-hardened** by construction (name + `description` cue, above). It resolves ADR-0024 OQ#2.

**One honest dependency.** ADR-0042 makes the reviewer *admissible*; what makes it *load-bearing* (rather than admissible-but-optional) is the **agent-autonomous-where-possible posture** — and that posture lives in RFC-0048 + this RFC, **both Open**. So precisely: D7 is **adopted as load-bearing given that posture**; if the posture is not ratified at acceptance (the design gate stays human-ratified every time), D7 falls back to *admissible-but-optional* per ADR-0042 — still permitted, just not required. The pre-mortem's falsifier (§ Risks) carries this. Remaining work is mechanical: the eval surface and the RFC-0048 roster amendment (flipped, below).

### D8 — the optional design-tool handover (ADOPTED)

*New thread — the experience→realization seam.*

`map-screen-flow` (and its per-screen briefs) can emit, **optionally**, a **design-tool handover**: a structured, machine-consumable instruction set — keyed to each per-screen brief (job · states · layout intent · navigation · copy pointer · platform surface · the grounded aesthetic reference) — that a **generative design tool** consumes to realize a low- or high-fidelity design.

- **Detect-and-degrade.** If a design-tool MCP is connected (e.g. Figma AI, Claude, v0, or another generative-UI tool), the skill **triggers it** and walks the result (extending D4's "trigger a wireframe/prototyping tool via MCP" note from low-fi prototyping into realization). If none is present, it **emits the handover file** for the adopter to paste into whichever tool they use. No tool is required, none is endorsed — the skill names tool *categories*, not a winner.
- **Stays method, never pixels.** The handover is **instructions**, not a comp. This narrows the pixel-comps non-goal without breaching it (and without touching the agnosticism guardrail — it points to the brief and the grounded reference, reprints no values).
- **Why it matters under autonomy.** In an autonomous environment there is no human to open Figma; the handover is the machine-readable seam that lets the realization step run (by an MCP tool) or queue (as a file) without a human in the loop.
- **Path.** Rides the per-screen brief — `<parent>/screens/<slug>/<screen-name>.handover.md` (or a `handovers/` sibling), resolved through the same D6 layout contract.

**Recommendation.** Adopt — it is low-cost (one optional artifact + a detect-and-degrade branch), method-preserving, and it is the seam the autonomous operating model needs.

### D9 — internal business process flows: current-state / target-state (ADOPTED)

*New thread — the pack maps customer-facing journeys but had no **internal business** process view. Main-thread research (fetched-and-confirmed, see § Evidence) settled that this is a real, distinct gap. Decided: a separate skill **`map-internal-process`** in **`experience`**, with the journey skill named **`map-customer-journey`** so the pair reads as one outside-in / inside-out family.*

**The gap is real, with a boundary condition.** A **service blueprint**'s backstage/support rows (`blueprint-service`, D2) describe internal activity *anchored to a customer touchpoint* — they cover a process **triggered by and resolved within** a single customer interaction. They are **not** an internal business process flow when the process **spans many customer touchpoints** (e.g. "manage a claim" over weeks), **has no customer-facing layer** (e.g. "run the monthly close", "process supplier invoices"), or is being captured **as a constraint on the solution** rather than as a CX output. The repo already names this artifact and lacks a producer for it: `product-engineering`'s `frame-intent/references/current-state-inputs.md` lists *"Process map (inside-out, L3)"* as a brownfield constraint input — distinct from the *"Journey map (outside-in)"* — with no skill that authors it.

**The skill (`map-internal-process`).** Pure-markdown, agnostic — points to standards, reprints none:
- **Levels** — APQC PCF vocabulary (points to `apqc.org/pcf`, never reprints the framework). *Correction from the main-thread research:* APQC numbers **Category as L1** (13 of them), **Process Group L2**, **Process L3**, **Activity L4**, **Task L5**. The skill anchors on the **L3 process** (the named flow: trigger → outcome → roles → steps) and decomposes to **L4 activities** (where cross-functional handoffs and decision gates — the swimlane content — live); **L5 tasks are SOP/work-instruction territory, out of scope**.
- **Two states** — **as-is** ("current") and **to-be** ("target"), with an as-is→to-be **delta table** (`| Step | As-is | To-be | Rationale |`) — the analytical output, not just the diagram.
- **Capture methods** — synthesize from **job aids / SOPs / work instructions** first (document analysis, flagging cross-source conflicts), then validate via **SME workshop**: process walkthrough (default), JAD session (multi-stakeholder/cross-department), gemba walk (high-exception processes). These are the standards-pointing equivalents of `map-customer-journey`'s NN/g citation.
- **Artifacts** — a **SIPOC** scoping table (bounds the process first), a **swimlane** flow in mermaid (`flowchart` with `subgraph` lanes per actor — BPMN *vocabulary* at markdown fidelity), and a **pain/waste register** (the bridge from as-is to to-be).
- **Notation pointer** — points to **BPMN 2.0** (OMG / ISO 19510) and cross-functional-flowchart conventions; authors in mermaid; names full BPMN XML and metric-rich VSM as *BPM-tool territory* (not agent-producible as markdown), per the research.
- **Seams** — cross-references the service blueprint by-name (when a process is customer-triggered) and is the producer of the `frame-intent` "current-state process map" input.

**Separate skill — decided (pressure-tested).** It is not a *mode* of the journey skill: the inputs differ (personas + outcomes vs. job aids / SOPs + SME workshops), the output differs (outside-in stages × emotions/pains vs. inside-out swimlane × decision gates, as-is/to-be), and the grounding differs (NN/g journey mapping vs. APQC/BPMN/BABOK). Folding them into one skill would conflate outside-in CX with inside-out operations — the [ValueFlow "why you need all three"](https://valueflow.co/customer-lifecycle-design/) framing (customer journey · service blueprint · process engineering as three distinct artifacts) is the practitioner argument for keeping them separate.

**The paired naming — decided.** The journey skill is **`map-customer-journey`** and the new one is **`map-internal-process`**. It signals the **outside-in / inside-out axis** as a discoverable family, disambiguates "journey," and costs nothing now (both unshipped — no migration/alias/frozen-governance bridge, unlike the D1 pack rename). **Deliberate scoping decision (the caveat, now resolved):** `map-customer-journey` **scopes journey-mapping to the customer/end-user** — employee-journey mapping is explicitly out of this skill's v1 (a later mode if ever), so the customer/internal pairing is the clean axis rather than the looser experience/operations one.

**Placement — `experience` (the *belongs-here* argument, separate from the *distinct-skill* one).** That it is a distinct artifact (above) proves only that it is a *separate skill*, not *which pack* owns it — `product-engineering` is the live alternative (its `frame-intent/current-state-inputs.md` already names the L3 process map). The reason it lands in `experience`: **mapping-family cohesion** — it is the inside-out sibling of `map-customer-journey` and shares `blueprint-service`'s backstage seam, so the three read as one mapping family in one seat; splitting it to PE fractures that family across two packs for no gain, since the **by-name seam** (PE's `frame-intent` *consumes* the process map `experience` *produces*) gives PE what it needs without ownership. **This is a deliberate widening of the seat's identity** — `experience` now means *customer-facing **and** internal-operations mapping*, not UX-only — and is recorded as such, not slid in: it matches service-design practice (journey · blueprint · process as one discipline) and the agnosticism/anti-infrastructure guardrails are unchanged by it.

**Evidence — fetched and confirmed (main thread).** The APQC five-level hierarchy, BPMN-as-`to-be`-notation + swimlanes, the as-is/to-be pair, and the service-blueprint-vs-process-map distinction were verified against `apqc.org`, `iiba.org` (BABOK), `nngroup.com`, and practitioner sources (see § Evidence). The earlier subagent run could not fetch; this round did, so the D9 claims meet the no-fabricated-sources standard.

**Decision — adopted.** A separate skill **`map-internal-process`** in **`experience`**, with the journey skill named **`map-customer-journey`** (customer/end-user-scoped). It carries the platform/surface axis **not** (it is actor/swimlane-shaped); it joins the pack's eval + layout surface (D6 path `<parent>/processes/<slug>.md`).

### D10 — the `interaction-design` craft skill (ADOPTED)

*New thread — surfaced by the pressure-test against external design-skill catalogues: the experience discipline has a behavioral pillar the pack does not carry.*

**The gap is real and distinct.** The pack's craft skills cover **visual taste** (`aesthetic-direction`), **system foundations** (`design-system-foundations`), and **structure / wayfinding** (`layout-and-information-architecture`); the connective skills cover the **flow** (journey → screens → service). The per-screen brief names a screen's **actions** and its **states** — but no skill designs **how the interaction behaves**: the feedback an action produces and its timing, the input/validation flow of a form, navigation as a behavior (not just a structure), gesture and pointer affordances, and the fit to the cognitive laws that govern perceived ease. Interaction design is a recognized craft pillar, separate from visual design, from information architecture, and from journey mapping — and it is the one pillar with no home in the pack.

**The skill (`interaction-design`).** Pure-markdown, agnostic — points to standards, reprints none. It is **one coarse craft skill covering the discipline's pillars**, not a fan-out of many sub-skills (the granularity choice in § D2):
- **Feedback & response** — every action produces a perceptible, proportionate response; latency is designed for, not assumed away (skeleton-vs-spinner thresholds, optimistic update, the slow path and degraded connectivity as designed states). This is the **perceived-performance** lens, grounded in the **Doherty Threshold** (sub-~400 ms keeps the user in flow) — the design-time half of responsiveness; runtime measurement (Core Web Vitals etc.) stays out of charter (engineering/runtime, Principle 3).
- **Input & forms** — input flow, inline-validation timing (on-blur vs on-submit), error recovery, progressive disclosure.
- **State machines (component/screen behavior)** — models a screen or component's **stateful behavior** as a finite state model: states, events, transitions, guards (e.g. a form: `idle → validating → invalid → submitting → success|error`). Authored as a **mermaid `stateDiagram-v2`** embedded in the brief's interaction section; points to **statechart / FSM** modeling (Harel statecharts) as the agnostic referent — never a state-management *library* (no XState/Redux code reprinted). This is the *dynamic* complement to the quality floor's static state *set*.
- **Motion & micro-animations** — purposeful motion that **communicates state and causality** (transition choreography, state-change feedback, affordance hints, orientation continuity); points to platform motion guidance (**Material motion**, **Apple HIG motion**) and motion-design principles, **honors the quality floor's reduced-motion** rule, and **reprints no durations/easing/curves** (the values guardrail — motion is designed as principle + purpose, never as a `200ms ease-in-out` table).
- **Navigation as behavior** — transitions, back/undo semantics, preserving orientation across moves (pairs with `layout-and-information-architecture`'s *structure* — this is the *behavior* over that structure).
- **Gesture & pointer affordances** — carried on the **platform/surface axis** (D3): tap/swipe/drag and pointer conventions point to **Apple HIG** / **Material 3** gesture & motion guidance, never reprinted.
- **Cognitive-law fit** — points to **Nielsen's usability heuristics** (already in `design-critique`'s `references/heuristics.md` — cross-referenced, not duplicated) and **Laws of UX** ([Fitts's, Hick's, Miller's, Doherty, Jakob's](https://lawsofux.com/)) as the referent for target sizing, choice load, chunking, and convention-matching.
- **Referenced interaction-pattern families** — names and **points to** recognized **onboarding** patterns (progressive onboarding, empty-state-as-onboarding, contextual coachmarks) and **search-interaction** patterns (typeahead/autocomplete, faceted filtering, zero-/error-result handling) as pattern *families* a per-screen brief can invoke — **referenced, not authored as standalone skills** (the deliberate coarse-granularity line vs maximalist catalogues; a brief reaches for the pattern, the skill points to where it's documented).

**Distinct from its neighbours (the honest overlap check).** It is **not** the quality floor — the floor's *handle-all-states* enumerates *which* states exist and *reduced-motion* sets the motion-honors-preference rule; `interaction-design` models the *behavioral transitions between* those states (the state machine), designs the *motion* that honors that rule, and designs the *feedback within* them. It is **not** `layout-and-information-architecture` — that places things and structures wayfinding; this designs how moving through them *behaves*. It is **not** `aesthetic-direction` — that grounds visual taste; this grounds behavioral fit. **It is *not* `map-screen-flow`** — that is the *macro* flow *across* screens (which screens, the sequence, the cross-screen error/edge routing, the state matrix); `interaction-design` is the *micro* behavior *within* a screen (the in-component state machine, feedback/timing, input-validation flow, transition semantics, motion, gesture). **Three state homes, one carve:** the quality floor owns the state *set* (enumeration), `map-screen-flow` owns the *cross-screen* routing, `interaction-design` owns the *in-component* state machine. Macro-flow vs micro-behavior is the load-bearing line, since both touch states/transitions/errors.

**Shape.** A **craft skill** alongside the existing craft skills — it **enriches the per-screen brief** (the interaction/behavior section) and **defers to the shared quality floor** (§ D2), rather than emitting its own file-per-slug artifact; so it needs **no new D6 layout entry**. It is **standalone-useful** (elicits the screen/flow inline when no inventory is present) and **degrades** (names behaviors textually when the platform axis or downstream packs are absent). It is **reviewed by** the `experience-reviewer` (D7) like the other craft output. Its eval surface follows from this shape: a **trigger eval** (`eval_queries.json`) on the cold prompt, and a **Tier-4 judge** (`evals.json`) that scores the **interaction/behavior section it contributes to the per-screen brief** — feedback + timing, input/validation flow, the component state machine, motion/micro-animation purpose, navigation behavior, gesture affordances, cognitive-law fit — not a standalone artifact.

**Decision — adopted.** It clears the pack's add-a-skill bar: **agnostic** (principles + pointed-to standards, no values), **not duplicated** (a distinct behavioral pillar the carve above isolates), a **craft discipline** (not scaffolding), and it **recurs on every interactive product** (the three-times bar). Pure-markdown, no new primitive class, no runtime — the pack's posture is unchanged.

## Risks & what would make this wrong

**Pre-mortem.**
- *The rename breaks adopters.* `design-craft` is user-scope-default, so a rename touches every repo an adopter opened. **Mitigation:** the `contract-acquisition` precedent (rename + erratum, no alias) is proven; v0.1.1 + pre-stable bounds blast radius; the implementing spec re-aggregates the marketplace and bridges frozen governance in one PR.
- *The new skills become checklist ceremony* — a journey map that's a table nobody reads, a brief that's boilerplate. **Mitigation:** the skills require *substantive* per-stage / per-screen content (the RFC-0048 anti-ceremony rule), and `map-screen-flow` ends in a real verification step (the consistency pass + steel-thread, D4), not "emitted."
- *The platform axis leaks values* — a skill starts printing breakpoints or HIG spacing. **Mitigation:** the stack-neutral guardrail (RFC-0033) is restated in each new skill's anti-patterns; the axis points to standards, never reprints them.
- *We mis-call the autonomy level on D7.* Ship the `experience-reviewer` for a gate a human still ratifies → redundant ceremony (and an agent to maintain); skip it for an agent-autonomous gate → design ships with **no** independent review (the in-session author marks its own homework). **Mitigation:** tie the D7 decision to the *actual* operating-model autonomy posture, not to a default — and note the cost is asymmetric (an unused agent is cheaper than unreviewed autonomous design).
- *D8's handover drifts into authoring pixels / values.* A handover that starts embedding palettes, breakpoints, or a comp breaches the agnosticism guardrail. **Mitigation:** the handover is *instructions keyed to the brief*, names tool categories not values, and is covered by the same agnosticism CI lint as the rest of the pack.
- *D9 widens the seat's identity by accretion.* Folding internal business-process / BA work into `experience` without deciding it stretches a UX/design seat into operations. **Mitigation:** D9's placement (`experience` vs. `product-engineering`) is an explicit, surfaced call — not a silent ride-along on the rename.
- *D10 overlaps its neighbours into redundancy.* If `interaction-design` re-states the quality floor's states/motion or `layout-and-information-architecture`'s wayfinding, it is ceremony, not a pillar. **Mitigation:** the carve is explicit and one-directional — existing skills own *what/where* (which states exist, where things sit), `interaction-design` owns *how-it-behaves* (transitions, feedback timing, input flow, gesture, cognitive-law fit); the skill cross-references the floor and `layout-and-information-architecture` rather than re-printing them, and the D7 reviewer would flag a brief that conflated them.

**Key assumptions (falsifiable).**
- *The connective artifacts are agent-producible as intent specs* (markdown/mermaid/tables), not pixel comps — so a skill can author them. If journeys/blueprints genuinely need visual fidelity to be useful, the intent-spec framing is too thin.
- *Taste grounded in persona + precedent + standards is better than ungrounded taste* (Kahneman's stable-referent condition). If the persona/precedent referent is usually absent or unreliable, the grounding is theatre and `aesthetic-direction` should stay as-is.
- *The rename is low-cost at this stage.* If `design-craft` has meaningful install depth already, the no-alias migration is too disruptive and a pack-alias mechanism (deferred) becomes the real blocker.
- *(D7 — premise now confirmed) The operating model is agent-autonomous where possible, with human review reserved for the human-value-add judgment gates.* This is the stated direction, so the `experience-reviewer` is load-bearing (it runs the design review between those gates). Falsifier: if in practice a human ends up ratifying *every* design at G2 anyway, the agent becomes redundant ceremony — but that would contradict the stated autonomy posture, not confirm a different one.
- *(D10) Interaction behavior is a separable design concern* that a skill can author distinctly from layout/structure and the state floor. If in practice authors can't draw the *what/where* vs. *how-it-behaves* line — if `interaction-design` and `layout-and-information-architecture` keep colliding — then interaction is not a separable skill and belongs folded into the existing craft skills, not standing alone.

**Drawbacks.** The new skills to maintain (the connective trio + `map-internal-process` + `interaction-design`); two enhanced skills; a cross-pack edit in `product-engineering`; the rename's migration tail and the frozen-governance bridge; added surface in the `experience` README. Mitigated by the standalone-useful + detect-and-degrade posture (no skill *requires* the others) and the one-spec migration.

### Future considerations — deferred expansions

*Surfaced 2026-06-29 by a comparison of this pack against a maximalist external design-skill catalogue (~97 skills across research / ux-strategy / design-systems / ui-design / interaction-design / prototyping-testing / design-ops / designer-toolkit / visual-critique). - **Design-ops — sprint planning / design debt / impact reporting.** Currently treated as **out of charter** (process/ops, Principle 3 — not design *method*).
