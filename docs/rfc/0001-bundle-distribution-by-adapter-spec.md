## Summary

This repo is an **agent-primitives catalogue**, organised into
four independently-installable **packs**: `core`,
`governance-extras`, `user-guide-diataxis`, and `monorepo-extras`.
Core is the
foundation. It ships the spec-driven workflow (work-loop,
new-spec, bug-fix, the reviewer agents and implementer, the
session-start and pre-pr hooks), the layer-0 governance content
adopters need to be
agentbundle (AGENTS.md template, CHARTER, CONVENTIONS, the
`docs/architecture/` and `docs/specs/` shapes with README seeds,
and the spec.md+plan.md templates), and the `adapt-to-project`
skill that customizes everything to the adopter's actual project.
The three add-ons extend core: `governance-extras` adds the
RFC/ADR ceremony (skills + `docs/rfcs/` + `docs/adrs/` + the
rfc.md+adr.md templates); `user-guide-diataxis` adds the user-
documentation structure (the four diátaxis homes under
`docs/guides/`); `monorepo-extras` adds the `new-package` skill
and `packages/` scaffolding for adopters with a monorepo layout.
Adopters install à la carte through APM, Claude
Code plugins, or our CLI; the repo is not a starting-point
project they fork. The artifact this work produces is a **per-IDE
adapter contract**, defined in TOML at
`docs/contracts/adapter.toml`, declaring how source
primitives project to per-tool outputs. A build pipeline at
`tools/build/` is the reference implementation. It reads the
contract and per-pack recipes, then produces ecosystem-native
distribution artifacts. After installing the packs they want,
adopters invoke `core`'s `adapt-to-project` skill to customize
the just-installed primitives. This is the LLM-driven, step-gated
adaptation that "Use this template" patterns can't do.

## Motivation

Four observations drive this RFC's shape:

**1. Brownfield is the primary value surface.** Enterprises have
long tails of existing code repos — services, libraries, internal
tooling — that aren't going to be rebuilt as fresh template clones.
Each of those repos is a candidate for being *upgraded* into an
agentbundle repo by installing our catalogue plus running an
LLM-driven adaptation step that customizes the inserted primitives
to the codebase's actual conventions. "Today I can superpower any
codebase" is the headline promise; the size of the addressable
surface scales with every existing repo, not with new starts.
Greenfield (fresh repo) is the easier case structurally — same
install path, less existing content to reconcile.

**2. The original drift bug is one instance of a structural class.**
The recent fix (commit `a2bc58d`) restructured `docs/_templates/spec.md`
and had to touch nine files; the underlying class is *one source
edit demands N coupled edits elsewhere*. This RFC's structural
response — source-of-truth at `.apm/`, projections regenerated by a
build pipeline — closes that class. (RFC-0002 extends the same
mechanism to validate the repo against its own build output.)

**3. The agent-packaging ecosystem converged in 2025-2026 on two
tools, and a third spec.**
- **Microsoft APM** (open-source, "like npm for agent context"):
  manifest at `apm.yml`, source primitives under `.apm/`, lockfile at
  `apm.lock.yaml`, per-tool compile via `apm compile -t <target>`,
  policy governance via `apm-policy.yml`. Targets Copilot, Claude
  Code, Cursor, OpenCode, Codex, Gemini, Windsurf.
- **Claude Code plugins + marketplaces**: manifest at
  `.claude-plugin/plugin.json`, plugin layout (`skills/`, `agents/`,
  `hooks/hooks.json`, `.mcp.json`, …), marketplace catalog at
  `.claude-plugin/marketplace.json`, official meta-bundle category.
- **Agent Skills spec at agentskills.io**: standardizes the
  single-skill format (SKILL.md + YAML frontmatter + scripts/ /
  references/ / assets/). Adopted by ~40 tools per the spec's client
  list.

Re-implementing APM's resolver or Claude's marketplace fails the
four-principle test (substantive, not duplicative). The bundle is
small enough that hand-curation has been viable so far; the
ecosystem's installers handle distribution better than we could.

**4. The one piece the ecosystem hasn't published is the per-IDE
adapter contract.** APM has adapters in code (the README points at an
`MCPClientAdapter` inheritance pattern in `CONTRIBUTING.md`) but no
documented contract. Claude plugins don't have adapters at all
(single-tool). agentskills.io stops at the single-skill level. Our
work over six rounds of adversarial review produced a rigorous
adapter contract — for Claude Code, Kiro, Copilot, Codex — that the
ecosystem doesn't have. Formalizing it as a published TOML spec is
the contribution this template makes.

## Proposal

A source tree under `packs/` (each pack containing its own `.apm/` and `seeds/`), one
adapter contract in TOML, one build pipeline that renders the
combination into ecosystem-native distribution outputs. Adopters
consume via existing installers.

Each `packs/<name>/` directory is an independently-installable
unit. Inside a pack:
- `pack.toml` — metadata (name, version, description,
  dependencies on other packs, the `[pack.adaptation]` table of
  substitution markers, the `[pack.seeds]` list naming files this
  pack drops into the adopter's repo).
- `.claude-plugin/plugin.json` — per-pack Claude Code plugin
  manifest (hand-authored).
- `.apm/` — APM-consumable agent-runtime primitives (skills,
  subagents, hooks).
- `seeds/` — files the pack drops into the adopter's repo on
  install. Each directory under `seeds/` contains a `README.md`
  explaining what goes there (every projected
  directory must be self-documenting). Templates under
  `seeds/docs/_templates/` are split across packs by relevance —
  spec.md/plan.md ship with `core`; rfc.md/adr.md ship with
  `governance-extras`. The adopter ends up with a unified
  `docs/_templates/` directory whose contents reflect the packs
  they installed.

The build pipeline produces one APM package per pack (each pack
gets its own `apm.yml` generated from `pack.toml`) and one Claude
Code plugin per pack (each carries its hand-authored
`plugin.json`), and a single shared `marketplace.json` lists them
all. The pack's `seeds/` content is included in both the APM
package and the Claude plugin so that — regardless of install
route — the adopter ends up with the expected directory layout
and README seeds in their repo.

Adopters install any subset of the four packs — never "the repo"
as a unit. There is no `apm install agent-ready-repo`; there's
`apm install agent-ready-repo/core`,
`apm install agent-ready-repo/governance-extras`,
`apm install agent-ready-repo/user-guide-diataxis`, and
`apm install agent-ready-repo/monorepo-extras`. Same for Claude
plugins: `/plugin install core@agent-ready-repo` (and so on per
pack). The
catalogue is the *collection of four packs*; the repo is the
*source of truth for them*. `core` is the only pack most adopters
need; the three add-ons are for adopters who want the RFC/ADR
ceremony, the user-documentation structure, or monorepo package
scaffolding.

#### Pack dependencies (intra- and cross-repo)

Packs declare dependencies in `pack.toml`. Dependencies can be on
other packs **in this same catalogue** (intra-repo) or on packs
**in other catalogues** (cross-repo). The install tool (APM, Claude
plugin marketplace, or our CLI) resolves dependencies before
installing the dependent pack; circular dependencies are rejected
at build time by the validator.

**Intra-repo example** — `governance-extras` *recommends* `core`
because its skills (new-rfc, new-adr, update-conventions) compose into
the work-loop when present, but they're authorable on their own
against any work-loop replacement an adopter has in place:

```toml
# packs/governance-extras/pack.toml
[pack]
name = "governance-extras"
version = "0.1.0"
description = "RFC/ADR ceremony skills (new-rfc, new-adr, update-conventions)."

[pack.dependencies]
recommended = [
  { catalogue = "agent-ready-repo", pack = "core", version = ">=0.1.0" },
]
```

`pack.toml`'s `[pack.dependencies]` table accepts three keys:
- `required = [...]` — installation fails if not satisfied.
- `recommended = [...]` — installer emits an info-level notice if
  not satisfied; install proceeds.
- `conflicts = [...]` — installer refuses if the named pack is
  already installed.

Version-range grammar follows npm/semver caret-and-tilde
conventions: `^0.1.0` (compatible with 0.1.x), `~0.1.0` (compatible
with 0.1.0 patch updates only), `>=0.1.0` (any version at or
above), `>=0.1.0 <0.3.0` (range). Pre-release versions
(`0.1.0-alpha.1`) are matched literally; semver pre-release
ordering rules apply.

Cross-tool dependency resolution is *not* unified at install time
— each install tool resolves dependencies in its own format. An
adopter who installs `core` via APM and then tries to install
`jira-defect-flow` via Claude plugins has two separately-resolved
dependency graphs, and the catalogue makes no claim about
reconciliation. `pack.toml` is the *catalogue-authoring* unified
declaration that the build pipeline reads to emit per-tool
manifests; the install tool then resolves against those.

**Cross-repo example** — a downstream Jira-integration catalogue
declares packs that depend on our `core` for its bug-fix skill:

```toml
# In some adopter's or third party's repo:
# packs/jira-defect-flow/pack.toml
[pack]
name = "jira-defect-flow"
version = "1.2.0"
description = "Workflow that pulls Jira defect tickets into the bug-fix loop."

[pack.dependencies]
required = [
  { catalogue = "agent-ready-repo", pack = "core", version = ">=0.1.0" },
  { catalogue = "internal-tooling/jira-mcp-server", pack = "jira-client", version = ">=2.0.0" },
]
```

When an adopter installs `jira-defect-flow`, the install tool
resolves the dependency graph and ensures `core` is installed (or
prompts the adopter to install it explicitly). Cross-repo
dependencies follow the install tool's native model: APM resolves
them via git references in `apm.yml`; Claude Code plugins resolve
them via `allowCrossMarketplaceDependenciesOn` in `marketplace.json`
plus the dependency declarations in each plugin. Our `pack.toml`
schema is the unified declaration the build pipeline reads to
generate both forms.

This is the load-bearing reason packs are independently
installable rather than mega-bundled: downstream catalogues
(adopter-authored, third-party, or our own future extensions) need
to declare specific dependencies on specific packs without dragging
in everything else in this repo.

### The per-IDE adapter contract

The contract is a TOML document at
`docs/contracts/adapter.toml`. TOML chosen because:

- **Typed scalars** — strings, ints, booleans, datetimes are
  distinguished by syntax. No Norway problem, no `1.10` parses as
  `1.1`, no surprise type coercions.
- **Comments allowed** — the contract is a spec document; readers
  need annotation. JSON forbids comments; YAML allows them but
  competes with whitespace semantics.
- **Section-oriented** — one `[adapter.<target>]` table per IDE,
  with `[[adapter.<target>.projection]]` arrays for each
  primitive-to-output rule. The shape matches what the spec
  describes.
- **Format-neutral relative to APM (YAML) and Claude plugins (JSON)**.
  Neither downstream packager has reason to pick our spec over its
  native manifest format, so the spec's format is independent.
- **`tomllib` is in Python stdlib (3.11+).** The reference
  implementation reads the contract without external dependencies.

The contract's schema:

```toml
[contract]
version = "0.1"
description = "Maps source primitives to per-tool projection outputs."

# Primitive definitions — what lives in .apm/
[primitive.skill]
source-path = ".apm/skills/<name>/"
schema = "agentskills.io/specification"

[primitive.agent]
source-path = ".apm/agents/<name>.md"
schema = "local:agent-frontmatter-v1"

[primitive.hook-body]
source-path = ".apm/hooks/<name>.py"
schema = "executable-python"

# Per-target adapters
[adapter.claude-code]
target-tool = "Claude Code"
schema-version = "v2026-05"

[[adapter.claude-code.projection]]
primitive = "skill"
output-path = ".claude/skills/<name>/"
mode = "direct-directory"

[[adapter.claude-code.projection]]
primitive = "agent"
output-path = ".claude/agents/<name>.md"
mode = "direct-file"

[[adapter.claude-code.projection]]
primitive = "hook-body"
output-path = "tools/hooks/<name>.py"
mode = "direct-file"

[[adapter.claude-code.projection]]
primitive = "hook-wiring"
output-path = ".claude/settings.local.json"
mode = "merge-json"

[adapter.kiro]
target-tool = "Kiro"
schema-version = "v0.9"

[[adapter.kiro.projection]]
primitive = "skill"
output-path = ".kiro/skills/<name>/"
mode = "direct-directory"

[[adapter.kiro.projection]]
primitive = "agent"
output-path = ".kiro/agents/<name>.md"
mode = "direct-file"
frontmatter-mapping = "kiro-agent-frontmatter-v0.9"

[[adapter.kiro.projection]]
primitive = "hook-wiring"
mode = "degraded-info-log"            # Kiro hook on-disk schema not yet published

[adapter.copilot]
target-tool = "GitHub Copilot"
schema-version = "v2025-12"

[[adapter.copilot.projection]]
primitive = "skill"
output-path = ".github/instructions/<name>.instructions.md"
mode = "instruction-file"
frontmatter-default = { applyTo = "**" }

[[adapter.copilot.projection]]
primitive = "agent"
mode = "dropped"

[adapter.codex]
target-tool = "OpenAI Codex CLI"
schema-version = "v2026-04"

[[adapter.codex.projection]]
primitive = "skill"
output-path = "AGENTS.md"
mode = "managed-block-inline"
managed-block-delimiter-start = "<!-- agent-skills:start -->"
managed-block-delimiter-end = "<!-- agent-skills:end -->"

# Named frontmatter mappings (referenced from adapter blocks)
[frontmatter-mapping.kiro-agent-frontmatter-v0.9.fields]
name        = { kiro-field = "name",        required = true }
description = { kiro-field = "description", required = false }
tools       = { kiro-field = "tools",       required = false, normalize = "to-list" }
model       = { kiro-field = "model",       required = false }
```

Seven projection modes are defined:
- `direct-directory` — full source directory copied to output path.
- `direct-file` — single source file copied (or transformed via frontmatter mapping).
- `merge-json` — JSON value merged into target file under a managed key.
- `instruction-file` — wrap content in target's instruction-file frontmatter.
- `managed-block-inline` — write content between delimiters in an existing file; preserves out-of-block content.
- `degraded-info-log` — emit an `[info]` log line at build time; no file written. Used when the target tool *should* receive this primitive but the on-disk format isn't known yet (e.g. Kiro hook wiring).
- `dropped` — explicit no-op; the primitive doesn't translate to this target at all (e.g. subagents for Copilot and Codex, which have no subagent concept).

The contract is the *specification*; `tools/build/adapters/` is one
reference implementation. Other implementations can validate against
the contract.

### The build pipeline

`tools/build/build.py` is a pure-stdlib Python script that operates
**per pack**. Its inputs:

- The adapter contract (`docs/contracts/adapter.toml`).
- A recipe template (e.g. `tools/build/recipes/per-pack-claude-plugin.toml`).
- A `packs/<pack>/pack.toml` (the pack's metadata).
- The pack's `.apm/` (skills, agents, hooks, commands).
- The pack's `seeds/` (files the pack drops into the adopter's repo).

Its outputs: one APM package and one Claude Code plugin per pack;
one shared `marketplace.json` listing all per-pack Claude plugins.

The build runs in two phases:

1. **Per-pack render.** For each `packs/<pack>/`, render an APM
   package to `dist/apm/<pack>/` and a Claude Code plugin to
   `dist/claude-plugins/<pack>/`. Each output is a self-contained
   installable unit (with its own `apm.yml` or `plugin.json`).
2. **Marketplace aggregation.** Produce a single
   `dist/claude-plugins/marketplace.json` listing every per-pack
   plugin entry. Adopters add the marketplace once and install
   any subset of packs.

**Pack-internal naming and collision policy.** Skill, agent, and
hook names are namespaced *to the pack* in source — a skill in
`packs/core/.apm/skills/work-loop/` is logically `core:work-loop`.
The Claude Code adapter's plugin-loader convention already prefixes
plugin-installed skills with the plugin name (a skill in plugin
`agent-ready-core` becomes `/agent-ready-core:work-loop`), so
cross-pack collisions don't reach the adopter's slash-command
surface even when two packs happen to ship a skill with the same
local name. The build pipeline validates pack-internal uniqueness
(no two skills with the same name inside the same pack); cross-pack
name reuse is allowed and resolved by the adopter's install tool's
namespacing convention. For self-hosting (RFC-0002), this repo's
packs by convention use distinct local names to keep the
`.claude/skills/` projection collision-free; the build pipeline
rejects a self-host build that would produce a collision.

A recipe TOML declares which adapter to use, how to read per-pack
metadata, and how to write per-pack output:

```toml
# tools/build/recipes/per-pack-claude-plugin.toml
[recipe]
name = "per-pack-claude-plugin"
description = "For each pack in packs/, render a Claude Code plugin."

[recipe.adapter]
target = "claude-code"

[recipe.iterate]
over = "packs/*/"
read-pack-metadata = "pack.toml"        # name, version, description, dependencies

[recipe.per-pack-output]
root = "dist/claude-plugins/{pack.name}/"
include-primitives = ["{pack.path}/.apm/**"]
plugin-manifest = "{pack.path}/.claude-plugin/plugin.json"
```

```toml
# tools/build/recipes/per-pack-apm-package.toml
[recipe]
name = "per-pack-apm-package"
description = "For each pack in packs/, render an APM package."

[recipe.adapter]
target = "passthrough"

[recipe.iterate]
over = "packs/*/"
read-pack-metadata = "pack.toml"

[recipe.per-pack-output]
root = "dist/apm/{pack.name}/"
copy = [
  { from = "{pack.path}/.apm/",   to = ".apm/" },
  { from = "{pack.path}/seeds/",  to = "seeds/" },   # pack's seeds (governance content, READMEs, templates) ship inside the APM package
]
generate-apm-yml-from = "{pack.path}/pack.toml"
```

```toml
# tools/build/recipes/marketplace.toml
[recipe]
name = "marketplace"
description = "Aggregate all per-pack Claude plugins into one marketplace.json."

[recipe.adapter]
target = "marketplace-aggregator"

[recipe.outputs]
marketplace-path = "dist/claude-plugins/marketplace.json"
plugins-from = "dist/claude-plugins/*/"
owner = { name = "eugenelim" }
```

Seven commands:
- `make build` — runs all recipes; produces `dist/apm/<pack>/` for
  every pack, `dist/claude-plugins/<pack>/` for every pack, and
  `dist/claude-plugins/marketplace.json`.
- `make build PACK=core` — single-pack build (the
  build pipeline still applies all recipes, but limits to one
  pack).
- `make build RECIPE=per-pack-claude-plugin` — single recipe across
  all packs.
- `make build --self` — runs the implicit self-host recipe (used by
  RFC-0002) which overlays Claude Code projections onto this repo
  itself; see RFC-0002 for the recipe's composite assembly rules.
- `make build --self --dry-run` — same as `--self` but writes to a
  temp directory and prints the diff against on-disk, never
  modifying the working tree. Underpins `make build --check`.
- `make build --check` — re-runs `--self` in dry-run mode against
  the current source and diffs against on-disk projections (the CI
  gate; see RFC-0002).
- `make build --scaffold OUTPUT=./` — runs the
  `core` pack's `seeds/` content (plus any other installed packs' `seeds/`) against the adopter's repo
  (post-install, optional). RFC-0003 promotes this to
  `agentbundle scaffold`.

`--force` is accepted by `make build --self` and `--self --dry-run`
to bypass the dirty-tree refusal; it does not bypass the comparison
gate (which is the whole point of `--check`).

### Distribution outputs

The build pipeline produces, **per pack**, one APM package and one
Claude Code plugin; one shared `marketplace.json` lists every
pack's Claude plugin entry. Adopters pick the packs they want and
install them à la carte. There is no "install the whole catalogue"
artifact.

Per environment:

| Adopter environment | Per-pack install command |
|---|---|
| **Claude Code users (any codebase)** | `/plugin marketplace add <our-repo>` then `/plugin install <pack-name>@agent-ready-repo` (repeat for each pack) |
| **Cross-tool adopters via APM** (Copilot, Cursor, OpenCode, Gemini, Windsurf, plus Claude Code via APM) | `apm install <our-repo>/<pack-name>` (repeat or list in your `apm.yml` deps) |
| **Constrained-network fallback** | `git clone <our-repo>` + `agentbundle install --pack <pack-name>` against the local checkout |

Common adoption patterns:

| Pattern | Packs installed |
|---|---|
| Standard agentbundle repo | `core` (covers AGENTS.md + docs/architecture/specs/knowledge/product/ + work-loop + reviewer agents + adapt-to-project) |

| Plus user documentation | `core` + `user-guide-diataxis` |

| `core` | work-loop, new-spec, bug-fix, adapt-to-project | adversarial-reviewer, security-reviewer, quality-engineer, implementer | session-start, pre-pr | conventions-check | `AGENTS.md` template + `docs/CHARTER.md` + `docs/CONVENTIONS.md` + `docs/architecture/` + `docs/specs/` + `docs/knowledge/` + `docs/product/` + `docs/_templates/{spec,plan}.md` (each directory with a README) |

THIS repo's own ADRs, RFCs, and specs (governance entries about
the catalogue's own evolution) live at the projected paths
(`docs/adrs/`, `docs/rfcs/`, `docs/specs/`) because this repo is
its own first adopter via self-host. They are *not* part of any
pack and don't ship — adopters get the empty `docs/rfcs/` and
`docs/adrs/` directories with the explanatory README seeds, and
author their own entries there.

After installing the packs they chose, adopters run the
`adapt-to-project` skill (which they should always include unless
they have a strong reason not to) to customize the just-installed
primitives to their actual project. The adaptation step is
pack-aware — it operates on exactly the packs the adopter
installed, never assuming the presence of packs that weren't
selected.

**This repo is not a "Use this template" target.** GitHub's
template-clone pattern hands adopters a static skeleton that
doesn't reflect their stack or conventions and forces an all-or-
nothing choice; the catalogue-installer model lets adopters
compose exactly the packs they need and then adapt them per-repo.

### Governance convention

The catalogue's governance layout maps onto the adopter's repo via
the per-pack `seeds/` content. Across the four packs, adopters
who install all four end up with this layout in their repo (one
README seed per directory; templates split across packs by
relevance):

| Path | Ships via pack | What it is |
|---|---|---|
| `AGENTS.md` | `core` | Root agent-context file (template; `<adapt:NAME>` markers resolved by adaptation step) |
| `docs/CHARTER.md` | `core` | What the project is, what's in/out of scope (template) |
| `docs/CONVENTIONS.md` | `core` | How the team works (template) |
| `docs/architecture/README.md`, `docs/architecture/overview.md` | `core` | Seed README + overview template |
| `docs/specs/README.md` | `core` | Explains the `docs/specs/<feature>/` convention |
| `docs/knowledge/README.md`, `docs/knowledge/patterns.jsonl` | `core` | Self-learning skills accumulate patterns here; `patterns.jsonl` ships empty |
| `docs/product/README.md`, `docs/product/roadmap.md`, `docs/product/changelog.md` | `core` | Roadmap + changelog convention (templates) |
| `docs/_templates/spec.md`, `plan.md` | `core` | Spec authoring templates (used by `new-spec` and `bug-fix`) |
| `docs/rfcs/README.md` | `governance-extras` | Explains the RFC convention |
| `docs/adrs/README.md` | `governance-extras` | Explains the ADR convention |
| `docs/_templates/rfc.md`, `adr.md` | `governance-extras` | RFC + ADR templates |
| `docs/guides/README.md`, plus `tutorials/`, `how-to/`, `reference/`, `explanation/` (each with their own README) | `user-guide-diataxis` | Diátaxis-aware user-documentation structure |
| `packages/README.md`, `packages/_example/` | `monorepo-extras` | Monorepo package convention + example skeleton |

Every directory the packs project comes with a `README.md`
explaining what goes there — the load-bearing principle is that
no projected directory is silent. Adopters can hand-edit any
seeded file post-install; the adaptation step's substitutions
apply only to `<adapt:NAME>` markers, leaving the rest untouched.

**Convention edit discipline.** `core` ships
`docs/CONVENTIONS.md` with a minimal in-place edit discipline at
the bottom of the file: "Trivial fixes (typos, broken links) are
normal PRs; substantive changes deserve a paragraph in the commit
message explaining why." Adopters who install `governance-extras`
upgrade this to an RFC-gated workflow via the `update-conventions`
skill, which intercepts convention edits and routes them through
`new-rfc`. Core adopters who don't install governance-extras get
the lightweight discipline; the upgrade path is documented in
`docs/CONVENTIONS.md` itself.

**Knowledge accumulation via skills.** `docs/knowledge/patterns.jsonl`
is the append-only home for cross-session learnings. The
`work-loop` skill's *capture-learnings* step (run during the
loop's review phase when a non-obvious lesson surfaces) and
`bug-fix`'s root-cause step both append one-line JSONL entries
describing the pattern, the trigger that surfaced it, and the
fix. Skills installed from other packs (and adopters' own skills)
that follow the same convention also append here — the file is
the catalogue's mechanism for skills to share institutional
knowledge across sessions.

**Why `docs/product/` ships with core.** Roadmap and changelog
aren't optional ceremony for agentbundle repos — they're the
agent-readable surface that lets a session-starting agent know
*what the project is building right now* (versus what's done,
versus what's deferred). Without a project-shaped product
narrative, the agent has to infer it from issue trackers or PR
history every time. `docs/product/roadmap.md` and
`docs/product/changelog.md` are templates with `<adapt:NAME>`
markers; the README explains the convention.

**Source-of-truth note.** Pack `seeds/` content is the canonical
upstream. This repo's own `docs/` is *self-host-projected* from
the relevant packs (see RFC-0002) — same paths as what we ship to
adopters, because this repo is its own first adopter. Our actual
charter, conventions, architecture, specs, RFCs, and ADRs live at
the projected `docs/` paths, drift-checked against the pack
`seeds/` upstream. No separate `governance/` directory exists in
this repo.

### Post-install adaptation

Installing the catalogue copies primitives into the adopter's repo
in their canonical, generic form. **The adaptation step turns
generic primitives into project-specific ones**, and is the load-
bearing capability for "superpower any codebase." Without it, every
adopter would manually rewrite skill bodies to reference their lint
command, their test runner, their project name, and their
conventions — friction that defeats the install-route ergonomics.

Adaptation ships as a skill inside the `core` pack:
`packs/core/.apm/skills/adapt-to-project/`. The skill is **part
of core, not a separate pack** — applying-to-project is
required for the catalogue to deliver any value to a real
project, and core shouldn't be installable without it. After installing core,
adopters invoke the skill from Claude Code with
`/adapt-to-project` (or the equivalent in their tool). The skill is **interactive,
conversational, and step-gated** — it walks the adopter through
decisions and only applies changes the adopter explicitly
approves at each step. The skill's responsibilities:

1. **Detect the adopter's project context.** Sniff for
   `pyproject.toml`, `package.json`, `Cargo.toml`, `go.mod`, etc.;
   identify primary language; read `package.json` scripts or
   `Makefile` targets to find lint/test/build commands; identify
   any existing `AGENTS.md`, `docs/`, `.cursor/rules`,
   pre-existing `docs/`-shaped directories, hook scripts, or other agent
   conventions already in the repo.
2. **Propose four classes of change, each gated by the adopter:**
   - **Substitution** — where a skill body says "the test command
     for this repo," propose the concrete command from the project
     context. Where a skill body says "the project name," propose
     the detected name.
   - **Augmentation of existing files via upstream-companion
     drops** — when the install step lands a pack seed at a path
     where an adopter file already exists (the Tier-2 case from
     the [Adopter file safety contract](#adopter-file-safety-contract)),
     the install path does *not* overwrite. Instead it writes the
     pack's version alongside as
     `<filename>.upstream.<ext>` (e.g. `CHARTER.md` stays, and
     `CHARTER.upstream.md` is dropped next to it). The
     `adapt-to-project` skill picks up these `.upstream.*` pairs
     and proposes an LLM-driven merge: it reads both files,
     surfaces the differences, proposes a merged result that
     preserves the adopter's intent while incorporating the
     catalogue's additions, and asks the adopter to accept,
     modify, or skip per file. On acceptance, the merged result
     replaces the adopter's file and the `.upstream.*` companion
     is removed. On skip, the companion stays in place for a
     later session.
   - **Discovery + restructuring** — the skill scans for adopter
     content under non-canonical names that maps to slots in the
     pack-shipped structure. Examples: `DESIGN.md`,
     `docs/decisions/`, `docs/howto/`, `ADRs/` (different casing
     or path), `README.md` containing what should be `CHARTER.md`
     content, scattered hook scripts under non-standard paths.
     For each finding the skill proposes a move (or content
     migration into an augmented file), showing tree-before vs
     tree-after; the adopter accepts, rejects, or modifies per
     finding. Where the adopter prefers to keep their existing
     layout, the skill proposes adapting our installed primitives
     to *their* structure rather than the reverse — the layout
     negotiation goes both ways. The skill maintains a
     `.adapt-discovery.toml` cache of accepted/declined moves so
     re-runs don't re-propose declined ones.
   - **Restructuring within our layout** — within the canonical
     layout we ship, propose consolidations (e.g. multiple
     overlapping skill bodies, redundant subagent definitions)
     the adopter accepts file-by-file.
3. **Walk the adopter through choices conversationally.** The
   skill doesn't bulk-apply a plan; it surfaces one decision at a
   time with context ("you have an existing `AGENTS.md` at 280
   lines covering X, Y, Z; I'd suggest adding our work-loop
   conventions under a new section labelled `## Agent workflow`
   while preserving your existing content above and below — accept,
   modify, or skip?"). Where the adopter says "do all the
   substitutions but defer restructuring," the skill honours that
   and writes a follow-up plan to a `.adapt-pending.md` file the
   adopter can revisit.
4. **Surface ambiguity rather than guess.** Where the project has
   multiple candidates (a polyglot repo with both `pyproject.toml`
   and `package.json`), ask which to use. Where no candidate
   exists, write a `TODO` and continue.
5. **Never overwrite without explicit approval.** Dry-run by
   default; show the diff per file; require explicit acceptance.
   On disagreement, defer — never paper over inference failures
   with plausible defaults. Where a change requires touching files
   the adopter has edited since install (per their git status),
   the skill detects the dirty state and *escalates* — surfaces
   the conflict, names the adopter's edit, asks how to proceed.

The adaptation step works the same way for greenfield and
brownfield — the difference is how much existing context the
skill has to negotiate with. Greenfield projects mostly run
substitutions; brownfield projects spend most of the session in
the augmentation and restructuring conversations, with the skill
*adapting to* the adopter's repo at least as often as the adopter
adapts to ours. The bidirectional negotiation is what turns "we
installed our stuff into your repo" into "your repo is now
agentbundle in a shape that fits."

**Per-pack adaptation manifest.** Each pack declares its
substitutable values and augmentation points in `pack.toml` under
a `[pack.adaptation]` table. The LLM skill and the deterministic
CLI variant (`agentbundle adapt`) read this table to know what to
infer and what to substitute:

```toml
# packs/core/pack.toml (excerpt)
[pack.adaptation]
substitutions = [
  { marker = "<adapt:project-name>",      infer-from = "project-name" },
  { marker = "<adapt:install-command>",   infer-from = "package-manager-install" },
  { marker = "<adapt:test-command>",      infer-from = "test-runner" },
  { marker = "<adapt:lint-command>",      infer-from = "linter" },
  { marker = "<adapt:build-command>",     infer-from = "build-system" },
]
augmentation-points = [
  { target = "AGENTS.md",       section-marker = "<adapt:agents-md-additions>",  describe = "skills + agents the pack ships, inlined as an `## Agent workflow` section" },
]
```

This brings back a *constrained* placeholder syntax for
adaptation only. Unlike the deleted generic `<placeholder>`
mechanism (which the earlier draft used at install time across
the whole bundle), the `<adapt:NAME>` markers are:

- Scoped to pack source files only (the `.apm/` and `seeds/`
  content inside `packs/<pack>/`).
- Processed *only* by the adaptation step (LLM skill or
  `agentbundle adapt`), never by install. Install copies the
  markers through unchanged.
- Bounded by what the pack declares in `[pack.adaptation]`. No
  free-form substitution; only declared markers.

The constrained syntax earns its keep because `agentbundle adapt`
must be deterministic for CI use — without explicit markers,
there's nothing to substitute. The LLM skill uses the same markers
plus its own inference for the augmentation/restructuring
proposals that fall outside the substitution table.

For CI environments without Claude Code, RFC-0003's `agentbundle
adapt` provides a deterministic, non-LLM scriptable variant that
runs the substitution logic from `[self-host]`-style answer files
without LLM inference. The LLM-driven skill is the headline use
case; the deterministic CLI variant covers the constrained-
environment fallback.

### Adopter file safety contract

**Scope of the contract.** This contract binds two lifecycle
phases the catalogue owns directly: **the `adapt-to-project` skill
run** (LLM-driven or via `agentbundle adapt`) and **the
constrained-network CLI install path** (`agentbundle install
--pack`). Install via APM (`apm install`) and Claude Code plugins
(`/plugin install`) is governed by *those tools' own* file-handling
behaviour — APM has its own conflict semantics during
`apm install`; Claude Code plugins install to a cache rather than
the working tree (so working-tree collisions don't arise for the
Claude plugin path at install time). The catalogue does not — and
cannot — intercept those upstream install routes. What the
contract guarantees against, in the phases the catalogue owns, is
that the adapt step and the CLI install path will never silently
overwrite adopter-edited content.

Files in the adopter's repo fall into three trust tiers. Each
tier's behaviour depends on which lifecycle phase is operating:

| Tier | What it is | Adapt step (LLM or CLI) | `agentbundle install` | `apm install` | Claude `/plugin install` |
|---|---|---|---|---|---|
| **Tier 1** — files at adapter-contract paths, never adopter-edited | Created by the install or absent before install | Free to rewrite | Free to write (creation) or rewrite (update) | APM-native (compile to working tree) | Cache-only; never collides |
| **Tier 2** — files at adapter-contract paths, adopter-edited since install | Detection: the CLI's `.agentbundle-state.toml` records a content hash at install time; adapt and CLI-install commands recompute and compare | Refuse to silently overwrite; surface and prompt | Refuse to silently overwrite; surface and prompt | APM-native (its lockfile is dep-version-only, not per-file hash; APM's compile behaviour applies) | Cache-only; not applicable |
| **Tier 3** — files outside adapter-contract paths | Adopter's pre-existing `AGENTS.md`, `docs/`, source code, `.gitignore`, etc. | May propose changes (augmentation, restructuring) with per-file approval; never touches without consent | Untouched | Untouched | Untouched |

**Tier-2 detection mechanism (CLI install + adapt).** The
constrained-network CLI maintains `.agentbundle-state.toml` in the
adopter's repo recording: (a) which packs are installed at which
versions; (b) for each projected path, the SHA-256 of the content
at install time. On subsequent `agentbundle install --pack`
(version upgrade) or `agentbundle adapt`, the tool recomputes
hashes against the current working tree and flags any divergence
as Tier-2. For APM and Claude plugin install routes,
`.agentbundle-state.toml` is opt-in — adopters who want
catalogue-level Tier-2 enforcement past install run
`agentbundle init-state` after their APM/Claude install, which
hashes the just-installed files. This step is optional; adopters
who skip it get the install tool's native conflict semantics with
no additional protection.

**Tier-2 behaviour on detection.** The adapt step and CLI install
refuse the rewrite for that file, surface the divergence
("path X has been modified since install"), and offer three
options: keep the adopter's edits (skip the update for this file),
overwrite with the upstream version (preserving a backup at
`<path>.pre-update.bak`), or invoke the `adapt-to-project` skill
to negotiate the merge interactively.

**Initial install collisions (Tier-3 → Tier-2 fast-path).** When
the install step (CLI route) attempts to write a pack seed at a
path the adopter already has — e.g. an adopter with their own
`AGENTS.md` installs the `core` pack — the install does *not*
clobber. It writes the pack's version alongside as
`<filename>.upstream.<ext>` (e.g. `AGENTS.md` is left alone;
`AGENTS.upstream.md` is dropped next to it). These
`.upstream.*` companion files are picked up by
`adapt-to-project`'s augmentation step (see [Post-install
adaptation](#post-install-adaptation)) which proposes an LLM-driven
merge with adopter approval. The companion file remains in place
across sessions until the merge is accepted or the adopter
explicitly deletes it. APM and Claude plugin install routes
inherit whatever their native conflict semantics are; the
`agentbundle init-state` follow-up can produce the same companion
pairs for adopters who opt into catalogue-level safety after the
native install.

**Update and uninstall semantics per install route:**

| Verb | CLI route (`agentbundle install --pack`) | APM route | Claude plugin route |
|---|---|---|---|
| **Update** | `agentbundle install --pack <name>` with a newer version; runs the Tier-2 detection per `.agentbundle-state.toml` | `apm install` after the upstream version moves; APM applies its native diff-and-apply; `.agentbundle-state.toml` (if present) flags Tier-2 on next `agentbundle adapt` | Claude plugin auto-update via marketplace metadata; cache-resident, no working-tree collisions |
| **Uninstall** | `agentbundle uninstall --pack <name>`; removes Tier-1 files; warns on Tier-2 (offers to keep or remove with backup); never touches Tier-3 | `apm uninstall`; APM's native behaviour | `/plugin uninstall <name>`; cache-only removal |

The adapter contract's seven projection modes each declare their
on-conflict behaviour explicitly. Schema extension to RFC-0001's
contract TOML:

```toml
[[adapter.claude-code.projection]]
primitive = "skill"
output-path = ".claude/skills/<name>/"
mode = "direct-directory"
on-conflict = "prompt-then-preserve"      # Tier-2 behaviour: prompt the adopter; default to preserve their edits

[[adapter.codex.projection]]
primitive = "skill"
output-path = "AGENTS.md"
mode = "managed-block-inline"
managed-block-delimiter-start = "<!-- agent-skills:start -->"
managed-block-delimiter-end = "<!-- agent-skills:end -->"
on-conflict = "preserve-outside-block"    # block contents overwritten; surrounding content preserved
```

Per-mode on-conflict defaults:

| Mode | Default on-conflict |
|---|---|
| `direct-directory` | `prompt-then-preserve` — surface each colliding file; preserve adopter's by default |
| `direct-file` | `prompt-then-preserve` — same as above for a single file |
| `merge-json` | `merge-managed-key-only` — rewrite the managed key (e.g. `hooks` in `.claude/settings.local.json`); never touch other keys |
| `instruction-file` | `prompt-then-preserve` — Copilot's per-file mechanism makes collisions rare; still prompt when they happen |
| `managed-block-inline` | `preserve-outside-block` — only rewrite content between delimiters; surrounding content untouched |
| `degraded-info-log` | n/a — no file written |
| `dropped` | n/a — no file written |

A projection rule may override the default with an explicit
`on-conflict = "<value>"` field. The valid override values:
`prompt-then-preserve`, `prompt-then-overwrite`,
`preserve-outside-block`, `merge-managed-key-only`,
`overwrite-without-prompt` (rare; requires explicit justification
in the contract).

**Self-hosting (RFC-0002) is explicitly authorized to overwrite
projections in this repo** — that's the entire point of self-host
mode, since this repo owns both source and projection. The CI
gate (`make build --check`) flags any drift between source and
projection; resolution is "edit the source," not "preserve the
projection."

### Upgrades — granularity

Adopters upgrade packs via whichever install tool they used; the
catalogue exposes three granularities, in order of increasing
specificity:

| Granularity | Trigger | What changes |
|---|---|---|
| **Whole-pack** (default) | `apm update <pack>`, `/plugin update <pack>@agent-ready-repo`, or `agentbundle install --pack <name>` with a newer version | Every file in the pack at the new version. Tier-1 files (catalogue-owned, no adopter edits) get rewritten silently. Tier-2 files (adopter-edited since install) trigger the safety contract's three-option prompt — keep adopter's edits, overwrite with `.pre-update.bak` backup, or invoke `adapt-to-project`. See [Adopter file safety contract § Tier-2 behaviour](#adopter-file-safety-contract). |
| **Per-skill / per-agent / per-hook** | `agentbundle upgrade --pack <name> --skill <skill-name>` (or `--agent`, `--hook`, `--seed <path>`) | Only the named primitive's files. Tier-2 detection applies per-file — a Tier-2 file in the targeted primitive triggers the same three-option prompt as whole-pack upgrades. The rest of the pack stays at the previously-installed version. The CLI logs a mixed-version warning into `.agentbundle-state.toml`; future whole-pack upgrades pull the still-old files forward. |
| **Per-file (adapt-step gate)** | `adapt-to-project` skill walking `<file>.upstream.<ext>` companions one at a time | Adopter accepts, merges, or skips each companion per file. Skipped files stay at the previous version; accepted files take the new content (merged with adopter edits where the LLM proposes a merge). This is the *de facto* per-file granularity for adopters who installed at whole-pack level. |

The mental model: install tools decide *which version of which
pack* the adopter has; the adapt step decides *which files within
the new version* actually land. Whole-pack-then-adapt is the
common flow; per-skill-via-CLI is the escape hatch for adopters
who need to take a specific upstream fix without pulling other
changes. Pure file-level CLI granularity isn't exposed (would
encourage state divergence the safety contract can't reason
about); adopters who want that level of control go through the
adapt step.

**Mixed-version state.** When per-skill upgrades produce a pack
whose files come from different upstream versions, the catalogue
tolerates this but `.agentbundle-state.toml` records each file's
source version. Future whole-pack upgrades flag mixed-version
packs and prompt the adopter — either pull everything forward to
the newest pack version (resolving the mixed state) or continue
in mixed mode. The catalogue does not claim correctness guarantees
for mixed-version packs at runtime; the adopter has taken
explicit responsibility by going through the per-skill path.

**Downgrades.** Out of scope for v0.1. Adopters who need to roll
back uninstall and reinstall at the prior version. `apm` and
Claude plugins handle the install-side primitives; adapt-step
state in `.agentbundle-state.toml` is reset to match by
`agentbundle init-state` post-downgrade.

**State-file schemas (sketch).** Both state files live at the
adopter's repo root and are TOML. Detailed schemas defer to F-cli
(RFC-0003); the shapes anchored here:

```toml
# .agentbundle-state.toml — sketch
schema-version = "0.1"

[packs.core]
installed-version = "0.2.0"
source = "agent-ready-repo"           # the catalogue identifier
install-route = "cli"                  # "cli" | "apm" | "claude-plugin"

# Per-file SHA-256 hashes recorded at install time (Tier-2 detection)
[packs.core.files]
"AGENTS.md"             = { sha = "abc123…", from-pack-version = "0.2.0" }
"docs/CHARTER.md"       = { sha = "def456…", from-pack-version = "0.2.0" }
".claude/skills/work-loop/SKILL.md" = { sha = "ghi789…", from-pack-version = "0.1.0" }   # mixed-version after per-skill upgrade

[packs.governance-extras]
installed-version = "0.1.0"
# …
```

```toml
# .adapt-discovery.toml — sketch
schema-version = "0.1"

# Adopter-accepted discovery proposals (move/migrate non-canonical content into the pack's slot)
[[accepted]]
finding-id = "core/charter-from-design-md"
source-path = "DESIGN.md"
destination-path = "docs/CHARTER.md"
action = "move-and-merge"               # "move" | "move-and-merge" | "augment-existing"
accepted-at = 2026-05-22T10:00:00Z

# Adopter-declined discovery proposals (don't re-propose on next adapt run)
[[declined]]
finding-id = "user-guide-diataxis/howto-from-docs-howto"
source-path = "docs/howto/"
destination-path = "docs/guides/how-to/"
declined-at = 2026-05-22T10:01:00Z
```

**Post-resolution state.** After an adopter resolves a Tier-2
prompt, `.agentbundle-state.toml` updates:
- *Keep adopter's edits* → re-hash to the adopter's current
  content; `from-pack-version` stays at the previous value. The
  next upgrade against this file's slot won't re-prompt for the
  same divergence, but a new upstream change will.
- *Overwrite with backup* → re-hash to the new upstream content;
  `from-pack-version` advances to the just-installed version. The
  `.pre-update.bak` file is left in place for the adopter to
  delete when they're satisfied.
- *Invoke adapt-to-project* → re-hash to whatever content the
  merge produced; `from-pack-version` advances. The adapt step
  records its merge decision in `.adapt-discovery.toml` if the
  resolution involved structural moves.

The `sha` value is hex-encoded SHA-256 (64 chars). Other field
types in the sketches are illustrative; F-cli pins them.

The schemas evolve under the `schema-version` field; F-cli pins
the v0.1 shape and defines parser behaviour on unknown fields
(ignore + log info) and unknown schema versions (refuse + clear
message).

### Per-IDE adapter contracts (concrete)

The contract TOML above is the spec; this section names the
concrete behaviour each adapter implements.

**Claude Code adapter.** Full directory preservation. `.apm/skills/`
→ `.claude/skills/`. `.apm/agents/` → `.claude/agents/`. `.apm/hooks/`
→ `tools/hooks/` (bodies) + `.claude/settings.local.json` (wiring,
merged under the `hooks` key).

**Kiro adapter.** Full directory preservation for skills
(`.kiro/skills/`). Subagent projection with frontmatter mapping
(Claude-Code-shape → Kiro-shape per the
`kiro-agent-frontmatter-v0.9` mapping in the contract). Steering
projection: `docs/CHARTER.md` etc. project to
`.kiro/steering/*.md` with `inclusion: always` frontmatter (RFC-0002
decides which governance files ship as steering vs as plain
markdown). Hook *bodies* project to `tools/hooks/`. Hook *wiring*
is `degraded-info-log` mode — emit a build-time message; Kiro's
on-disk hook schema isn't published.

**Copilot adapter.** Per-skill projection to
`.github/instructions/<name>.instructions.md` with `applyTo: "**"`
frontmatter. Each skill is independent — no managed block needed
(Copilot's path-specific instructions mechanism handles
composition). Subagents drop (Copilot has no analog). Hook bodies
project to `tools/hooks/`; wiring drops (Copilot has no hooks).

**Codex adapter.** Skills inline into a managed block in root
`AGENTS.md` delimited by `<!-- agent-skills:start -->` and
`<!-- agent-skills:end -->`. The block's content is the skill
descriptions only (Codex has no on-demand loading; the full SKILL.md
bodies stay in `.apm/skills/` and are available to AGENTS.md-aware
tools that traverse it). Subagents drop. Hook bodies project to
`tools/hooks/`; wiring drops.

Detailed mapping tables, frontmatter transforms, and edge cases live
in `docs/contracts/adapter.toml`; the contract is the
authoritative source.

### What we explicitly don't build

To make the scope reduction concrete:

| Capability | Where it lives instead |
|---|---|
| Bundle manifest format | APM's `apm.yml` |
| Resolver / installer | `apm install` / `claude /plugin install` |
| Lockfile / version pinning | `apm.lock.yaml` / Claude plugin SHA pinning |
| Composition surface (`--with`, `--without`) | APM's `dependencies` list |
| Registry | APM is git-source; Claude uses `marketplace.json` over git |
| Adopter-install drift detection | APM's `apm audit` |
| Install-time placeholder substitution | Adaptation step (`adapt-to-project` skill or `agentbundle adapt`) does post-install substitution against the per-pack `[pack.adaptation]` markers |
| Scheduled-action upgrade pattern | `dependabot` for `apm.yml`; Claude plugin auto-updates via `marketplace.json` |

The earlier draft of this RFC proposed F3 (manifest format), F5
(`/adopt-repo`), F6 (`bootstrap.py`), F7 (content gate), F8 (upgrade
scalability), F9 (MIGRATION.md), and F10 (CHANGELOG entry). The
first six collapse into "consume the ecosystem"; F9 and F10 reduce
to a short note in `USING_THIS_TEMPLATE.md`. The remaining
follow-ons appear below.

### From design to spec — the trajectory

This RFC defines the bundle design and ships the adapter contract
*in this repo* — our catalog, our build pipeline, our reference
adapter implementations. Two follow-on RFCs are planned to lift the
design into a reusable, externally-validatable shape:

- **RFC-0002 — Self-hosting.** This repo becomes the first consumer
  of its own build pipeline. The `.apm/` source primitives and
  `seeds/` content are projected into `.claude/skills/`,
  `.claude/agents/`, and `AGENTS.md` by the same build pipeline that
  produces our distribution outputs. Drift between source and
  projection becomes a merge-blocking CI condition. Self-hosting is
  what makes the adapter contract *testable rather than aspirational*
  — RFC-0001's spec stops being a paper artifact the moment we run
  on our own implementation of it.

- **[RFC-0003 — Adapter contract publication + reference CLI](0003-spec-and-cli.md).**
  Lift the adapter contract from
  `docs/contracts/adapter.toml` in this repo to a
  published open standard with versioning, a conformance test
  suite, and a reference CLI (working name `agentbundle`) — see
  the *Reference CLI* section of RFC-0003 for the full eleven-
  subcommand surface (install, upgrade, adapt, scaffold, validate,
  render, diff, list-packs, list-targets, init-state, uninstall).
  RFC-0003's scope is *publication and CLI*, not contract
  extraction — the contract already exists by then, in this repo.

The sequencing matters: RFC-0001 ships the contract and the
reference implementation; RFC-0002 makes the repo self-host on it
(so the contract is exercised on every PR rather than just by
hypothetical adopters); RFC-0003 publishes the contract under a
versioning model and ships the CLI as the spec's executable form.
Each step is independently shippable, and each produces the
conditions the next one needs.

### Success criteria

Three checks gate this RFC as delivered:

1. **The adapter contract TOML validates against its own schema** and
   covers Claude Code, Kiro, Copilot, and Codex with the primitive
   types we ship (skill, agent, hook-body, hook-wiring).
2. **The build pipeline produces installable artifacts on a clean
   checkout.** `claude /plugin install` and `apm install` against
   the rendered outputs both succeed against a fresh adopter repo.
3. **This repo self-hosts on its own build pipeline** (RFC-0002's
   success criterion). The build-vs-on-disk diff is a no-op.

External adopter reports are nice-to-have but not the gate.

## Alternatives considered

**Alt 1 — Do nothing.** Keep hand-maintaining skill bodies and the
existing scaffold; let the drift class recur. **Why not chosen:**
the drift bug compounds with every adopter and every Layer 0 doc.
The cost of doing nothing isn't zero; it's deferred.

**Alt 2 — Pure consumption (Posture A from research).** Drop the
adapter spec entirely. Restructure as a single Claude Code plugin
meta-bundle plus an APM package; let APM's (undocumented) adapter
behaviour be our cross-IDE story. **Why not chosen:** the per-IDE
adapter contract is the one artifact this work
produced. Throwing it away to save ~one TOML file's worth of
maintenance loses the only contributable architectural artifact we
have. The trade is asymmetric: keep the spec, gain the upstream
contribution path; drop the spec, save almost nothing.

**Alt 3 — Stay independent (the earlier RFC-0001 shape).** Keep our
own `bundle.toml`, `bootstrap.py`, three install routes, manifest
format, lockfile schema, etc. **Why not chosen:** every one of those
components now exists upstream in APM or Claude plugins, generally
in a more mature form than we'd ship. The four-principle test
(substantive, not duplicative) rejects re-implementation.

**Alt 4 — YAML or JSON for the adapter spec instead of TOML.** YAML
matches APM's manifest format and would feel more familiar to
adopters reading both. JSON matches Claude plugins and is the
schema-everywhere default. **Why TOML:** the adapter spec is a
schema-definition artifact, not an adopter-facing manifest. Schema
work benefits from TOML's typed scalars, comment support, and
section-oriented structure. The downstream consumers (APM, Claude
plugins) each have their own preferred format; making the spec
neutral to both rather than matching one over the other is the
honest call. The Python stdlib's `tomllib` makes the reference
implementation dependency-free.

**Alt 5 — Place governance docs under `.apm/`.** Map CHARTER.md to
`.apm/instructions/charter.md`, CONVENTIONS.md to
`.apm/context/conventions.md`, ADRs to `.apm/context/adr-*.md`,
etc. **Why not chosen:** APM's primitive taxonomy is shaped for
agent-runtime context (file-glob rules, reusable prompts, named
agents, chat-mode configs). Governance prose serves a different
audience (humans) and lifecycle (write-once, read-when-relevant).
Forcing it into APM primitives distorts both the writing and the
loader semantics; readers expect runtime-shaped content under
`.apm/`, and renaming our CHARTER as "instructions" misleads them.
The two trees living side-by-side is structurally cleaner.

**Alt 6 — Skip the build pipeline; hand-author each per-tool
manifest.** Maintain per-pack `.claude-plugin/plugin.json` and
per-pack `apm.yml` by hand alongside the source primitives, with
no `pack.toml`-as-unified-declaration. **Why not chosen:** recreates
the drift class in a new direction. Source-primitive edits would
demand coupled edits to two per-tool manifests; the build pipeline
removes the coupling structurally by treating `pack.toml` as the
single authoring input and deriving both tools' manifests.

## Drawbacks

**The build pipeline is real new code.** Smaller than the F3+F6+F7+F8
collapse it replaces (no manifest format spec, no installer, no
content gate, no upgrade scalability machinery), but non-zero.
Estimate: a few hundred lines of Python plus the TOML schema. The
reference adapter implementations are the bulk; each is ~50-150 lines.

**Hooks portability remains an open problem.** Claude Code has rich
hook wiring (`PostToolUse`, matchers, etc.). Kiro has the same
trigger types but the on-disk schema isn't published. Copilot and
Codex have no hooks at all. The contract's `degraded-info-log` mode
documents the gap honestly but doesn't close it; adopters using
non-Claude tools get hook bodies projected but no wiring.

**Subagent format is Claude-Code-specific in origin.** Our `.apm/agents/`
shape matches Claude Code's subagent frontmatter; Kiro takes a
subset (via the frontmatter mapping in the contract); Copilot and
Codex drop subagents entirely. The contract treats this as the
load-bearing portability gap; the mapping table makes it
mechanically explicit.

**Dependency on Microsoft (APM) and Anthropic (Claude plugins).**
Both ecosystems are young (mid-2025 launches, by the research
session's measurement). If either project loses momentum or
changes direction incompatibly, we either follow the change or
revisit the consume-the-ecosystem posture. Mitigation: the source
tree at `packs/<pack>/.apm/` + `packs/<pack>/seeds/` is self-contained;
adopters can always `git clone` the repo and install packs via the
CLI (`agentbundle install --pack`) regardless of upstream APM or
Claude marketplace status.

**The adapter contract's adoption upstream isn't guaranteed.**
Contributing the spec to APM upstream is plausible (APM welcomes
PRs) but isn't pre-negotiated. Publishing it standalone is always
available; whether other implementations validate against it is
adoption-driven.

**No install-time placeholder substitution.** The earlier draft
had a Jinja-style placeholder model (`<project-name>`, etc.) that
ran at install time, before files landed in the adopter's repo.
The rewrite shifts substitution to *after* install, into the
adaptation step. The constrained `<adapt:NAME>` markers in
`pack.toml`'s `[pack.adaptation]` table are what
`adapt-to-project` (or `agentbundle adapt`) resolves; install
itself copies them through unchanged. Adopters who skip the
adaptation step see the raw markers in their projected files —
which is honest about what the markers mean (placeholders awaiting
adoption) rather than auto-filling with potentially-wrong
inferences.

**One-shot migration.** Hand-extracting current `AGENTS.md`,
`docs/CHARTER.md` etc. into `packs/core/seeds/` is real work, done once.
The earlier draft had the same migration cost; this rewrite hasn't
changed it.

## Unresolved questions

The RFC-level design is settled in the body. Implementation-altitude
items defer to the follow-on specs:

1. **Kiro hook on-disk schema.** Kiro's docs describe trigger types
   and the UI for creating hooks, but don't publish the on-disk
   schema. The contract's `degraded-info-log` mode is the placeholder
   until Kiro publishes. A follow-on revision of the contract adds a
   direct projection mode once Kiro publishes.

2. **Which governance files project to steering vs ship as plain
   markdown.** RFC-0002 (or its follow-on spec) decides: should
   `CHARTER.md` project to `.kiro/steering/charter.md` so Kiro
   surfaces it in chat context, or ship as plain `docs/CHARTER.md`
   so it's read as documentation? Same question for CONVENTIONS,
   APPROACH. Trade is "agent-runtime visibility" vs "writer-shape
   fit."

3. **`<adapt:NAME>` resolution for Claude-plugin-installed packs.**
   Claude plugins install to `~/.claude/plugins/cache/`, not the
   adopter's working tree. The pack's skill bodies (with
   `<adapt:NAME>` markers) live in cache, where the adopter
   doesn't directly edit them. Two viable resolutions: (a) the
   adapt skill *vendors* the cached skill bodies into the adopter's
   working tree under `.claude/skills/<name>/` and substitutes
   markers there (collisions handled via Tier-2 detection); (b) a
   runtime resolution mechanism reads a `.adapt-values.toml` from
   the adopter's repo at Claude session start and substitutes
   markers in memory. Option (a) is simpler and matches the normal
   Claude Code skill-customization workflow but breaks plugin
   updates (vendored copies don't auto-update); option (b) is
   cleaner but requires runtime support Claude Code doesn't ship.
   The `adapt-to-project` skill spec (a follow-on to F-build)
   settles which.

(The adapter contract's *publication path* — contribute upstream to
APM, publish standalone under a versioning model, host the
conformance suite — is RFC-0003's territory, not an open question
at RFC-0001's altitude.)

## Follow-on artifacts

If accepted, this RFC produces:

- **F-spec — Adapter contract TOML.** Formal definition at
  `docs/contracts/adapter.toml` plus an `adapter.schema.json`
  validating the contract's own shape. Includes the seven projection
  modes (`direct-directory`, `direct-file`, `merge-json`,
  `instruction-file`, `managed-block-inline`, `degraded-info-log`,
  `dropped`), the frontmatter mapping syntax, and per-adapter
  examples. This is the **in-repo** spec; [RFC-0003](0003-spec-and-cli.md) lifts
  it to a published open standard with versioning and a conformance
  suite. F-spec doesn't block on RFC-0003 — the spec stands on its
  own in this repo, exercised by F-build and by RFC-0002's CI gate.
- **F-build — Build pipeline reference implementation.** Python
  stdlib only. `tools/build/build.py` + per-target adapter
  implementations + the three recipes (per-pack-claude-plugin,
  per-pack-apm-package, marketplace).
- **F-dist — Distribution authoring.** Per-pack
  `pack.toml` (the canonical metadata input; build pipeline
  derives the per-pack `apm.yml` and `plugin.json` from it),
  per-pack `.claude-plugin/plugin.json` (hand-authored content
  field, name/version derived from `pack.toml`), and the root
  `.claude-plugin/marketplace.json` (build-emitted, projected in
  self-host mode). Plus a `USING_THIS_TEMPLATE.md` rewrite naming
  the three install paths.
- **F-skills-decoupling.** (Carried over from the earlier draft.)
  Move skill assets into the owning skill's `assets/` directory;
  rewrite skill bodies to reference shape rather than file path;
  add a `lint-skill-isolation.py` that flags out-of-skill references.
- **F-hooks-python.** (Carried over.) Port `tools/hooks/*.sh` to
  Python; document Ralph as Mac/Linux-only.

The earlier draft's F3, F4, F5, F6, F7, F8, F9, F10, F11 all
collapse — they're either delegated to APM/Claude plugins or
absorbed into the build pipeline.
