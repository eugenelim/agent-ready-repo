## The ask

- **Recommendation (BLUF):** RFC-0044 ground the *build* loop in platform reality, but left the **design** phase upstream of it reasoning from model memory. Two real builds on different AWS stacks ([`0045-notes/field-report.md`](0045-notes/field-report.md)) show the `architect` pack produces designs that miss a **structurally fatal** platform fact the design phase never forced into view — a multi-round LLM-agent turn put behind a front door with a **non-configurable timeout** (two days of 504s), and a cluster of **managed-service contract** assumptions asserted from memory and corrected only at deploy. Close the gap **as prose only — one new reference (`lens-serverless.md`), a discipline sharpening, and review-rubric checks; no executable tooling, no new skill, no new reviewer** — reusing the workload-class routing axis RFC-0042 already shipped:
  - **(S) Back the deferred serverless workload-class lens.** Author `lens-serverless.md` (in both the `architect-design` and `architect-review` skill copies, kept byte-identical per the pack's per-skill shape), filling the slot `rubric-well-architected.md:17` names but never backed. Content is the **cloud-agnostic convergence of all three hyperscalers'** serverless guidance (AWS Serverless Applications Lens, Azure Functions WAF guide, Google Cloud Run best practices), spanning the **whole serverless class — compute, data, search/analytics, and event glue/messaging, not functions alone** (the field-report miss was a serverless *store*) — organized on five durable concerns: execution & throughput limits + the sync-vs-async gate; cold-start & readiness (compute *and* store scale-up); scale-to-zero economics, capacity floors & cost cliffs (per-request / per-byte-scanned); statelessness, idempotency & delivery semantics (at-least-once → idempotent consumers + DLQ + ordering); private-serverless network reachability.
  - **(C) A dual-consumed platform-contract grounding discipline — the design-time mirror of RFC-0044's A1.** For every managed service on a critical path, ground its **binding** contract (non-configurable limits, scaling floors, cold-start, network requirements) in an authoritative source — a curated platform skill, official docs, or `research` — with **stated confidence, never model memory**. **Consumed at both times:** `architect-design` grounds *by construction* (extending its existing "never fabricate landscape facts" discipline); `architect-review` **independently re-checks** and flags a load-bearing managed-service claim with no visible grounding as a finding. Not an oracle runner — at design time nothing is deployed to probe.
  - **(V) A dual-consumed synchronous-path viability check.** For any synchronous request path, **sum worst-case latency across every hop and compare it to the binding front-door timeout**; classify long-running operations (an LLM-agent turn, a multi-hop graph traversal) and force a **sync-vs-async decision gate** (stream / 202-accept-then-poll / fire-and-forget+callback). AWS-, Azure-, and GCP-documented prior art, not invented. The serverless lens owns the mechanics; the agentic lens gets a one-line cross-reference; the WA review rubric gets a Performance-pillar check.
  - **(K) Version-specific facts stay out of the agnostic lens.** The 29-second ceiling, NCU floors, Lambda duration/payload limits, and managed-runtime packaging models belong in **curated platform skills** (recommend-when-warranted, à la RFC-0044's T2 tier) — never bundled per-vendor (Principle 1). The lens carries the durable *rule* + an illustrative example with "confirm current specifics with the provider."
  - **Scope:** serverless only; **ML / SaaS stay deferred** (named-but-unbacked), consistent with ADR-0032. Update the backlog entry.

- **Why now (SCQA):** *Situation* — RFC-0042 made the `architect` pack apply a workload-class lens by construction at design *and* re-check at review, and RFC-0044 grounded the downstream build loop in the platform's real contract. *Complication* — the design phase upstream of the build loop still reasons from memory: a design was shipped that was **structurally impossible** under a front-door timeout the design never named, and the `architect-review` pass that graded it "SHIP WITH CHANGES" contained **zero** mentions of latency, duration, sync/async, or the timeout contract — so the single highest-impact flaw was invisible to both the design *and* its review. *Question* — should the design phase gain the discipline to ground its load-bearing platform claims in reality (and back the serverless lens that carries the recurring ones), at both construction and review time, or keep designing agentic/serverless systems against abstract pillars and discovering the binding constraint two days into the build?

## Problem & goals

**Diagnosis.** RFC-0044 grounded the *build* loop in platform reality; the design phase that precedes it does not. Two builds (field report) show the same root cause RFC-0044 named — *reasoning from memory/proxy rather than the platform's real contract* — but **one inner-loop stage earlier**, where there is no deployed stack and no toolchain oracle to run:

1. **The structurally-fatal latency miss.** The `architect-design` output for an agent system classified a 30–60 s multi-round LLM-agent turn as a *real-time conversational request*, placed it behind an API Gateway HTTP API (a **non-configurable ~29 s** integration timeout), and explicitly dismissed the async alternative as "over-engineered for a real-time agent." No section summed the end-to-end latency across the chain (facade → runtime cold-start → three sequential inference rounds → per-tool gateway hops) or compared it to the binding ceiling. The build then spent two days on 504s before the architecture was found non-viable. **The `architect-review` pass had graded the design "SHIP WITH CHANGES"** with thorough security/data findings — and not one word about latency, duration, sync/async, or the timeout. The design phase had no forced check for this, *and neither did its review*.
2. **The managed-service contract gaps.** A second build asserted managed-service behaviors from memory and corrected them only at deploy: "the store is VPC-only / has no public endpoint" (too absolute), a single-AZ subnet topology that fails to provision, and the load-bearing one — a "serverless" graph store assumed to scale to zero that in fact floors at a minimum capacity unit and bills continuously, plus per-query latency at that floor that made the chosen traversal shape blow the function timeout on every live call.

The unifying cause matches RFC-0044's: **the design is shaped, and reviewed, against model memory and abstract pillars, not the platform's binding contract.** RFC-0042 gave the pack the *routing machinery* to load a workload-class lens by construction and re-check it at review — but the serverless lens it would route to was deferred (ADR-0032 D5), and there is no discipline forcing the binding-platform-fact into view for *any* managed service. The Performance pillar names "latency budget (p50/p99)" abstractly (`well-architected-pillars.md`), but never as a *binding-constraint viability check*.

**Goals.**
- **Back the serverless workload-class lens** so serverless/managed-platform designs are shaped against the recurring binding concerns by construction and re-checked at review — the slot RFC-0042 deferred.
- **Ground load-bearing platform-service claims in reality at both construction and review** — the design-time mirror of RFC-0044's contract-acquisition, adapted to a phase with no deployable artifact (ground in docs/skills/`research`, not oracles).
- **Make the synchronous-path viability check a first-class, dual-consumed step**, so a structurally-impossible front-door/long-operation pairing is caught at design *and* at review, not at build.
- Keep the lens **cloud-agnostic** (converged across AWS/Azure/GCP) with version-specific facts routed to curated platform skills — durable rule in the lens, current number in the skill.
- Stay **prose-only at design altitude** (Principle 3), reuse RFC-0042's routing axis (no new mechanism), and add **no new skill, reviewer, or tooling**.

**Non-goals** (could-have-been goals, deliberately dropped):
- *Not* backing the ML or SaaS workload-class lenses. They stay named-but-unbacked in `rubric-well-architected.md` (status quo), consistent with ADR-0032. (Could have been a goal — declined to keep this RFC about serverless + the cross-cutting grounding discipline.)
- *Not* a per-vendor serverless knowledge base or a shipped contract for any platform — the lens is agnostic; version-specific facts route to a curated platform skill (recommend-when-warranted). Pre-enumerating clouds breaks Principle 1 and rots on arrival.
- *Not* authoring an AWS-serverless platform skill in this RFC — a broader AWS-serverless platform skill is a possible follow-on, not part of this scope. (An externally-distributed `aws-bedrock-agentcore-skill` already exists in the wild as a user-installable curated platform skill for the agent-runtime facts — a living instance of the K pattern, not a repo pack — but authoring or bundling one is out of scope here.)
- *Not* an oracle runner or any executable tooling — at design time nothing is deployed to validate/probe; RFC-0044 owns the build-time oracle protocol. (Principle 3.)
- *Not* a new skill (a routing branch + reference suffices — Principle 2 and the pack's three-peer-skill shape) or a new reviewer (the existing `architect-review` / `design-reviewer` carry it).
- *Not* re-authoring RFC-0042's routing axis — this RFC is a *second consumer* of that axis (the serverless lens) plus the grounding/viability disciplines; the axis itself is unchanged.

## Proposal

Organized under the requested decisions.

**Decision 1 — scope & vehicle: prose only, six-part change, reuse the existing axis.** Everything below is reference prose and SKILL.md/rubric edits on the existing `architect` pack: one new reference file `lens-serverless.md` (per-skill duplicated), a discipline sharpening in `architect-design`'s procedure, a one-line cross-reference in `lens-genai-agentic.md`, and check additions to `rubric-well-architected.md` + the design-doc rubric. The workload-class routing axis that loads a lens by construction (design Stage 0) and at review (WA mode) already exists (RFC-0042 / ADR-0032); serverless becomes the second backed value on it. No executable code, no new skill, no new reviewer. Vehicle is RFC → ADR → spec because ADR-0032 explicitly scoped backing any deferred lens as reopening-scope work for "a future RFC," and the change extends a shipped user-scope-default pack's well-architected coverage with a cross-cutting design discipline.

**Decision 2 — (S) the serverless workload-class lens.** `lens-serverless.md`, structured like `lens-genai-agentic.md` (a provider-agnostic design-quality overlay that names the concern and the question it forces, applies what bites, recites nothing), and converged from the three hyperscalers' own serverless guidance so it is not AWS-shaped.

**Scope — the whole serverless class, not functions alone.** The lens spans four serverless entity types, because the binding facts recur across all of them (a serverless graph store and a function fail along the *same* axes, in different detail — and the field-report miss was a serverless *store*, not a function): **serverless compute** (FaaS; scale-to-zero containers), **serverless data** (auto-scaling / on-demand databases), **serverless search/analytics**, and **serverless event glue / messaging** (managed queues, topics, event buses). Each concern below is applied to whichever entities it bites, never recited across all four. Five concerns:

- **Execution & throughput limits + the sync-vs-async gate.** Serverless entities carry **bounded, often non-configurable** limits: request/integration timeouts on compute and the front doors before it; **per-partition / per-unit throughput ceilings and throttling cliffs** on serverless data and search (a hot key, or a burst past a recent-peak allowance); and query-latency-at-floor on a scaled-down store. For any synchronous path the design establishes the binding ceiling and that the work fits under it; a **long-running or burst operation** (an LLM-agent turn, a multi-hop traversal, a heavy transform, a traffic spike past the throttle allowance) forces a sync-vs-async decision — **stream**, **202-accept-then-poll** (return a job id; process on an async-invoked worker; client polls a status store), **fire-and-forget+callback**, or a deliberate **pre-warm / capacity reservation**. This is the AWS/Azure/GCP-converged pattern (see Evidence), and it is the V check below stated in serverless terms.
- **Cold-start & readiness.** First-use latency is not a function-only concern: a scale-to-zero or newly-attached compute runtime pays a cold start (worsened by VPC/network attachment and large dependencies), **and** a scaled-down serverless store/search pays a scale-up / first-connection / buffer-cache-warm latency on the first request after idle. Budget it on the critical path and choose a posture — tolerate (retry/poll), keep-warm (minimum capacity / prewarmed / provisioned concurrency, at standing cost), pre-warm before a known spike, or shrink the cold path.
- **Scale-to-zero economics, capacity floors & cost cliffs.** "Serverless" names a billing/scaling *model*, not a guarantee of zero idle cost — and the model **differs by entity**. Some serverless components genuinely reach zero (FaaS; on-demand request-priced stores; scale-to-zero containers); others **floor at a minimum capacity unit and bill continuously** (several serverless databases and search tiers). Two cost cliffs to name: the **standing floor** (min-capacity components never reach zero), and the **per-request / per-byte-scanned cliff** (a pay-per-use store or query engine where an unbounded scan or a runaway caller is a denial-of-wallet surface). The design states which serverless components actually reach zero, quantifies the standing floor for those that don't, and bounds the per-use cliff for those priced per request/scan.
- **Statelessness, idempotency & delivery semantics.** Serverless compute is ephemeral and share-nothing — no in-process state survives, and work after a handler returns may not run; persistent state lives in an external store. And the serverless **event/messaging glue** wiring these components is **at-least-once by default** (standard queues, topics, event buses) — so **consumers must be idempotent**, poison messages need a **dead-letter path**, and **ordering is guaranteed only by a FIFO/ordered variant** (at a throughput cost). Multi-step flows are orchestrated by a state machine, not chained in-process. (The AWS Serverless Lens "share nothing" / "design for failures and duplicates" principles, the SQS at-least-once → idempotent-consumer rule, the Azure "design for idempotency" recommendation, and the GCP stateless / no-post-return-work rule converge here.)
- **Private-serverless network reachability.** A serverless component reaching a private (VPC/VNet-resident) dependency needs explicit network attachment, which adds cold-start latency and requires the full set of egress paths/endpoints; a missing path typically surfaces as a **silent timeout**, not a clear error. Some serverless hosting tiers cannot attach to a private network at all — a constraint that belongs in the design, not the build.

**Decision 3 — (C) the dual-consumed platform-contract grounding discipline.** The design-time mirror of RFC-0044's A1 contract-acquisition, adapted to a phase with no deployable artifact:

- **By construction in `architect-design`.** Extend the existing knowledge-surface discipline (Step 2/3: "never fabricate landscape/standards/in-flight facts"; degrade + lower confidence when a surface is absent) with an explicit **platform-service-contract** clause: for every managed service on a critical path, ground its *binding* contract — non-configurable limits, scaling floors, cold-start behavior, network/identity requirements — in an authoritative source (a curated platform skill if present; else official docs; else `research`), carry **source + confidence**, and **lower confidence + flag** any load-bearing claim that could not be grounded. Never assert a service contract from model memory.
- **Independently re-checked in `architect-review`.** WA mode gains a check: a **load-bearing managed-service claim with no visible grounding** is a finding (the reviewer does not trust the design's own assertion — it re-derives or flags, the same posture RFC-0044 gives `quality-engineer` at build time). This is the user-requested half: the *reviewer* is grounded on the platform contract, not only the author. Tagged 🔧 mechanical when the pillar spine determines the missing fact must be established; 🧭 judgment when grounding it needs a business/risk call.
- **Carve vs RFC-0044 (explicit).** RFC-0044 acquires the contract from the toolchain's **machine oracles** (validate/plan/synth + schema slice + runtime probe) at build/verify time, when IaC and a deployable artifact exist. This RFC grounds the contract from **authoritative prose** (platform skill / docs / `research`) at design time, when neither does. Same thesis (ground in reality, not memory), different evidence source dictated by the stage. The design-time grounding *hands off* to RFC-0044's build-time oracle as the artifact materializes.

**Decision 4 — (V) the dual-consumed synchronous-path viability check.** A first-class step, not an abstract pillar mention:

- **The check.** For any synchronous request path, **sum the worst-case latency across every hop** (front door → handler → cold-start → downstream/model rounds → per-call overhead) and **compare it to the binding front-door timeout**. If the operation is long-running, the synchronous shape is non-viable and the **sync-vs-async gate** (Decision 2's mechanics) fires.
- **Dual-consumed.** `architect-design` runs it by construction when shaping a synchronous path; `architect-review` WA mode re-checks it as a Performance-pillar item (an unbudgeted synchronous long-operation path is a finding — 🟥 blocker when it makes the design structurally impossible, as the field report's was). Added to `rubric-well-architected.md`'s Performance check and the design-doc rubric's cross-cutting prompt.
- **Homes & no duplication.** The serverless lens owns the mechanics (it is where managed-platform timeouts bite). The **agentic lens gets a one-line cross-reference** — an agent turn is the canonical long-running operation, so an agentic design behind a synchronous front door routes to the serverless gate (per the approved open-question default, a cross-reference, not a duplicated concern).

**Decision 5 — (K) version-specific facts stay in curated platform skills.** The lens carries durable rules and illustrative examples flagged "confirm current specifics with the provider" (the house style of `cloud-primitives.md` / `well-architected-pillars.md`). The binding *numbers* — a specific integration-timeout ceiling, a store's minimum-capacity-unit floor, a runtime's payload/duration limits, a managed-runtime's packaging/entrypoint model — live in a **curated platform skill** the contract-grounding discipline (C) recommends installing when absent on an unfamiliar managed surface (the same T2 detect-and-recommend RFC-0044 uses). The externally-distributed `aws-bedrock-agentcore-skill` — a user-installable skill, **not a repo pack** — is a living instance of this tier (it carries exactly the AgentCore runtime/packaging facts the field-report build needed); the `architect-diagram` skill (in the `architect` pack) also ships an AgentCore *reference* (`references/agentic-bedrock-agentcore.md`) as in-repo prior art for the shape. No per-vendor data is bundled into the agnostic lens.

**Decision 6 — scope: serverless only.** ML and SaaS stay named-but-unbacked in `rubric-well-architected.md`. Update `docs/backlog.md`'s `ml-saas-serverless-workload-class-lenses` entry: serverless **resolved by this RFC**; ML/SaaS still deferred, unblocking when a future RFC backs them.

## Options considered

**Axis: *where the serverless/contract guidance is applied in the design→review lifecycle* × *artifact form*** — the same cross-product RFC-0042 used (so the two follow-ons stay comparable), extended with the form question RFC-0044 used. MECE along {design-construction · review-only · both} × {lens-prose · executable tooling · per-vendor KB}, do-nothing included.

| Option | Lifecycle | Form | Verdict |
|---|---|---|---|
| **A. Both (design + review), lens-prose + dual-consumed disciplines** ★ | design **and** review | prose on existing skills | **Recommended** |
| B. Review-only (back the lens for `architect-review` WA mode alone) | review | prose | Half the fix — design still shapes against abstract pillars; the structural miss recurs and arrives as rework |
| C. Design-only (load the lens by construction, no review check) | design | prose | Loses the independent re-check — a design marks its own platform homework; regresses RFC-0032's value and the user's explicit reviewer-grounding ask |
| D. Executable viability/contract tooling (a latency-budget calculator, a contract fetcher) | either | code | Fails Principle 3; nothing is deployed at design time to probe; RFC-0044 already owns build-time oracles |
| E. Per-vendor serverless knowledge base bundled in the pack | either | bundled data | Fails Principle 1; can't cover the long tail; stale on arrival (the K decision is the bounded alternative) |
| F. Do-nothing (keep serverless deferred, no grounding discipline) | — | — | Cost of delay below |

- **A (recommended)** — mirrors RFC-0042/ADR-0032 (the dual-consumed workload-class lens is the accepted, prior-art-validated shape) and RFC-0044 (ground in reality, re-derive at review). Reuses the shipped routing axis; clears Principles 1–3.
- **B / C** — each is half the ask; the field report shows the miss survived *both* a missing design check and a blind review, so a one-sided fix leaves the recurrence open. C also contradicts the user's explicit instruction that the reviewer be grounded on the platform contract.
- **D** — a design-time calculator/fetcher is runtime infrastructure (Principle 3); the binding facts come from prose the agent already reads, and the build-time machine-oracle job is RFC-0044's.
- **E** — a bundled per-cloud KB rots and can't span the long tail; the curated *platform skill* (K) is the bounded, opt-in, recommend-when-warranted version.
- **F (do-nothing)** — keep serverless deferred. Cost of delay: every serverless/agentic design shaped in the pack repeats the binding-constraint blind spot; the human stays the relay (the field report shows two independent builds where the human had to force the structural realization mid-build). Recurring and confirmed, not hypothetical.

**Second axis — lens content structure** (flat list vs. pillar-keyed vs. concern-grouped). Chosen: **concern-grouped** (the five durable concerns), because the three hyperscalers' serverless guidance converges on these concerns rather than on a pillar partition, and a flat list would bury the sync-vs-async gate — the highest-impact concern — among lower-stakes ones. This matches the capability/concern-grouped shape ADR-0032 chose for the agentic lens.

## Risks & what would make this wrong

**Pre-mortem.**
- *The serverless lens drifts into a per-vendor recitation that rots.* Mitigation: the K decision routes all version-specific facts to curated platform skills; the lens carries durable rules + "confirm with the provider"; an adversarial pass checks for vendor-binding, exactly as RFC-0044 guards its tool-keyed protocol.
- *Box-ticking* — "contract grounded: yes" / "viability: ok" with no substance. Mitigation: C's output is a *cited contract slice with confidence*, not a checkbox; V's output is a *summed latency budget vs. a named ceiling*; the review re-check re-derives rather than trusting the design's assertion.
- *The grounding discipline over-fires*, demanding a citation for every trivial managed-service use. Mitigation: it is scoped to **load-bearing claims on a critical path** (a binding limit the design depends on), not every service mention — the same "apply what bites" altitude as the lenses.
- *Lens/agentic-lens duplication of the latency concern.* Mitigation: the serverless lens owns the mechanics; the agentic lens carries a one-line cross-reference only (the approved default).
- *Surface growth* — a fourth backed lens + a discipline clause + rubric checks to maintain. Mitigation: prose on the existing axis, progressive-disclosure (the lens loads only on serverless-class work); justified by a structural failure confirmed across two independent builds.

**Key assumptions (falsifiable).**
- *Serverless is a recognized first-class workload-class WA lens worth backing.* **Checked — holds**: the AWS Serverless Applications Lens is a published, pillar-organized workload-class lens with seven named design principles, and Azure (Functions WAF service guide) and GCP (Cloud Run best practices) ship the converging guidance (Evidence). The lens is not invented.
- *The sync-vs-async viability gate is recognized prior art, not invented doctrine.* **Checked — holds**: it is AWS Prescriptive Guidance (202-accept-then-poll, explicitly because of the 29 s ceiling) and Azure's Durable-Functions / "defer the work, return immediately" guidance; GCP forces it by forbidding post-return background work.
- *The workload-class routing axis can carry a second backed value with no change.* Believed true — RFC-0042 built it as a general axis (`rubric-well-architected.md:11-19`); serverless is already a named value, only the backing file is missing. Falsified if the axis turns out agentic-specific (it is not, by inspection).
- *Grounding load-bearing claims at design time changes outcomes, not just adds prose.* The assumption most worth watching post-ship — the field-report failures are exactly what C/V name, but whether the discipline shifts behavior is the live question (the same caution RFC-0044 flagged for its EXECUTE-brief craft prose).

**Drawbacks.** A fourth backed workload-class lens and a cross-cutting discipline clause are real surface to maintain as serverless practice moves across three vendors. The grounding discipline adds an early judgment call ("is this claim load-bearing enough to ground?") when the concept is least settled — mitigated by scoping it to critical-path binding limits. And the agentic-lens cross-reference creates a soft coupling between two lens files that must not drift.

## Evidence & prior art

- **Spike / de-risk result.** The riskiest assumption — *serverless is a recognized first-class workload-class WA lens applied at design and review* (parallel to RFC-0042's AWS-Agentic-Lens spike) — was checked by fetching the [AWS Well-Architected Serverless Applications Lens — Design principles](https://docs.aws.amazon.com/wellarchitected/latest/serverless-applications-lens/general-design-principles.html). Confirmed: it is a published workload-class lens with seven general design principles, including *speedy/simple/singular*, *share nothing*, *orchestrate with state machines not functions*, and ***design for failures and duplicates* ("Operations triggered from requests or events must be idempotent, as failures can occur and a given request or event can be delivered more than once")** — directly grounding the lens's statelessness/idempotency concern. The second assumption — *the sync-vs-async gate is documented prior art* — was checked by fetching [AWS Prescriptive Guidance — Process events asynchronously with API Gateway and Lambda](https://docs.aws.amazon.com/prescriptive-guidance/latest/patterns/process-events-asynchronously-with-amazon-api-gateway-and-aws-lambda.html), which states verbatim *"The hard limit of 29 seconds is generally acceptable for synchronous workloads. However, that limit represents a challenge for those developers who want to use API Gateway with asynchronous workloads,"* and documents the POST-returns-job-id → async-invoke → write-to-DynamoDB → client-polls-`GET /jobs/{jobId}` pattern (supporting jobs up to the 15-minute Lambda limit). This is the field report's remediation, verbatim AWS doctrine.
- **Repo precedent.**
  - RFC-0044 (`docs/rfc/0044-infra-contract-acquisition.md`) — the build-time companion this mirrors at design time; same memory-vs-ground-truth thesis, with the design-time adaptation (no oracle; ground in prose) carved explicitly in Decision 3.
  - RFC-0042 + ADR-0032 (`docs/adr/0032-agentic-overlay-is-a-design-and-review-workload-class-lens.md`) — built the dual-consumed workload-class routing axis and **deferred ML/SaaS/serverless** (RFC-0042 D5; recorded at ADR-0032's scope-boundary bullet); ADR-0032 §"Neutral / to revisit": *"ML / SaaS / serverless stay deferred… A future RFC that backs any of them reopens this scope."* This RFC is that future RFC for serverless.
  - `docs/backlog.md` `ml-saas-serverless-workload-class-lenses` — the explicit deferred slot, *"Unblocks when: a future RFC takes on backing one of them."*
