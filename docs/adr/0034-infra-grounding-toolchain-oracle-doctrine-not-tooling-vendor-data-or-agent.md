# ADR-0034: Grounding the infra inner loop in platform reality is toolchain-oracle doctrine + EXECUTE-loaded craft — one new `core` skill, not executable tooling, per-vendor data, or a new agent

- **Status:** Accepted
- **Date:** 2026-06-24
- **Decision-makers:** eugenelim
- **Consulted:** RFC-0044's two-build, scrubbed field report (`docs/rfc/0044-notes/field-report.md`); its external prior-art set (the IaC defect-taxonomy replication study, the deployability-centric IaC-generation loop, HolmesGPT / k8sgpt, Terratest + Google Cloud Terraform testing practice, the HashiCorp `terraform providers schema` reference verifying force-new is *not* machine-readable); the spec-stage adversarial + design review of this ADR and the implementing spec
- **Supersedes:** none
- **Related:** RFC-0044 (the accepted decision this records); ADR-0031 (the scaffold this extends — infra support is doctrine on existing reviewers, no new reviewer or runtime; this ADR adds the *grounding* layer, extends the `operational-safety` consumer to EXECUTE, and honors ADR-0031's no-executable-tooling / no-new-reviewer calls without reversing them); ADR-0023 (the three-reviewer ceiling — the constraint that forecloses a fourth, infra-contract-lens reviewer, deferred here behind an evidence trigger); ADR-0018 + RFC-0029 (shift review left + deliver depth via an orchestrator-loaded progressive-disclosure library — the `security-checklists` / `operational-safety` pattern the new skill and module reuse); RFC-0041 (P1 preflight extended by a fifth artifact, P2 smoke refined by V1/V2, `observability-and-smoke` strengthened by D, the no-new-reviewer ceiling honored); RFC-0025 + ADR-0014 (risk triggers already route a destructive/irreversible `apply` to full mode — the hook point); the repo's 3-tier dependency policy (T2's detect-and-recommend); CHARTER Principles 1–4

## Context

RFC-0041 / ADR-0031 gave `work-loop` the verification *shape* for infrastructure — preflight → plan → idempotent apply → active smoke → rollback, with operational safety on `quality-engineer` and a mandatory security reviewer + scanner pair. But that scaffold rests on two standing assumptions: that the agent **knows the platform's contract**, and that it will **exercise the whole chain** rather than trust a cheap proxy.

Two independent, multi-day infrastructure builds on *different* stacks (`docs/rfc/0044-notes/field-report.md`) falsify both. The loop builds, verifies, and reviews against **model memory or an early/cheap oracle** rather than the platform's **actual contract and runtime ground truth**, and the failures cluster into **six families that are MECE along the inner-loop-stage axis** (Author → Provision → Verify → Debug, with Author and Verify each splitting into two truth-kinds):

1. **Author · structural (contract).** Invented CLI flags, a violated naming regex, a wrong tool-schema shape, an immutable-field collision — *contract* failures, not reasoning failures; the human's fix was to hand the loop a curated platform skill mid-session.
2. **Author · behavioral (craft).** Permissions iterated reactively until authorized; propagation / eventual-consistency delays not designed for; timeouts escalated and `sleep`s sprinkled instead of designed backoff; no client cold-start tolerance; dependency cycles fixed by trial.
3. **Provision (identity & convergence).** Credentials re-resolved per call until an SSO wrapper timed out — the "establish once, reuse" fix rediscovered independently in *both* builds; a `ROLLBACK_COMPLETE` stack can't be updated and must be recreated; non-idempotent creates collided on re-run; live interaction ran as one-off command-spray that left nothing reusable behind.
4. **Verify · static (oracle fidelity).** The cheap early oracle (synth / validate) passed while the system was broken; "deployed ≠ working" treated as done.
5. **Verify · runtime (the probe).** Proving it works needs a probe that runs *inside the network if the resource is private*, does a *data-plane round-trip*, is *readiness-aware*, and *tears itself down* — a different shape than app verification, demanded by the human both times.
6. **Debug (localization).** A facade-emitted timeout was chased dozens of times in the backend log — no symptom→layer map, no chain bisection, no failure-signature catalog.

The compound symptom is the **human-as-relay anti-pattern ADR-0031 named, recurring one layer up** — at *grounding*, not just *driving*: in both builds the human had to push for real end-to-end verification and hand the loop platform knowledge mid-session. An independent-axis empirical IaC defect taxonomy (the replication study cited in RFC-0044) maps cleanly onto these six families, evidence the set captures the failure space rather than over-fitting two builds.

RFC-0044 settled *whether* and *how* to close this. This ADR records the load-bearing, expensive-to-reverse calls — the *form*, the *generality mechanism*, the *EXECUTE consumer extension*, and the *reviewer posture* — so a future maintainer doesn't re-litigate them. The mechanical detail (the exact protocol tiers, the per-tool oracle table, the module prose, the work-loop edits) is spec-level and lives in `docs/specs/infra-grounding/`, not here.

Constraints in force when deciding:

- **Principle 3 (a habit, not infrastructure).** The repo ships doctrine and prose the agent reasons from, never a runtime engine, parser, or wrapper. This is the bar that put browser-bridge out of charter and that ADR-0031 cleared.
- **Principle 1 (universal across stacks).** Anything that ships must hold for Terraform, Pulumi, CDK, CloudFormation, Kubernetes, and hand-rolled scripts alike — and reach on-prem / Hetzner / bare-metal, not just the big three — with no tool-specific bindings in normative prose.
- **Principle 2 (no duplication).** A capability belongs in exactly one place; the grounding belongs *in* the loop and the existing depth-library mechanism, not a parallel surface.
- **The three-reviewer ceiling (ADR-0023).** Core code-review has exactly three lenses — `adversarial-reviewer`, `security-reviewer`, `quality-engineer`. A fourth is out of bounds.
- **ADR-0031's calls are in force and must not be reversed** — no executable infra tooling, no new reviewer; this ADR extends the scaffold within them.

## Decision

> We will ground `work-loop`'s infra inner loop in the platform's real contract and runtime truth across all six MECE failure families, as **doctrine plus reference prose**: a new **protocol-shaped `core` skill (`infra-contract-acquisition`)** that drives the toolchain's own deterministic oracles — **tool-keyed, not vendor-keyed** — and declares its oracle tier + confidence; a new **`operational-safety` module (`cloud-implementation-craft`)** inlined into the **implementer's EXECUTE brief**; a strengthened **`observability-and-smoke`** module; and `work-loop` SKILL.md edits — with conformance review on the **existing `quality-engineer`** and **no new executable tooling, per-vendor knowledge base, or reviewer / agent.**

Seven sub-decisions, each expensive to reverse (cross-references to "Decision N" below are to **RFC-0044's** eight-decision numbering, not this ADR's bullet order):

- **Form & home — doctrine + references; one new protocol-shaped skill, not tooling and not per-vendor data.** Everything ships as prose: a new `infra-contract-acquisition` skill, a new `operational-safety` module, edits to `observability-and-smoke`, and `work-loop` SKILL.md edits. The new skill is **protocol-shaped** — a short tiered protocol body (T0→final) *plus* a per-tool `references/` oracle table — where the two existing depth libraries (`security-checklists`, `operational-safety`) are checklist-shaped: a deliberate, recorded family inconsistency. No plan-parser, schema-diff, probe-runner, or oracle runner ships (Principle 3); no bundled per-cloud knowledge base ships (Principle 1). The agent drives the oracles the toolchain already ships.
- **Generality is tool-keyed, not vendor-keyed; coverage is a declared capability spectrum.** The protocol routes on the **tool** — *strong* (Terraform / OpenTofu, Pulumi, CDK / CFN, k8s / Helm → any provider they address, incl. Hetzner / Proxmox / vSphere / OpenStack / on-prem-k8s), *medium* (Ansible `--check --diff`, Bicep, cloud-init), *weak / none* (bespoke REST + curl, hand-rolled bare-metal). The agent **declares its oracle tier + confidence** and on a weak oracle **shifts weight to the runtime probe** rather than faking static coverage. Schema coverage is heterogeneous and read accordingly: CFN `createOnlyProperties` and Pulumi `replaceOnChanges` come from schema, but **Terraform's `providers schema -json` does not expose force-new** (verified against the HashiCorp reference), so the replace signal is read from `plan` + provider docs.
- **Craft is applied at EXECUTE, not only reviewed at REVIEW — `operational-safety` gains a second consumer.** The new `cloud-implementation-craft` module (least-privilege-but-**sufficient** permissions derived from the contract in one pass; eventual-consistency / propagation readiness waits; timeouts set to the dependency's real latency + bounded backoff + **client / frontend cold-start tolerance**; dependency ordering; terminal-failed-state handling; and the managed runtime's **deployment-artifact packaging / entrypoint-import model** — confirm how the runtime packages and imports the entrypoint before writing imports, with platform specifics deferred to the T2 skill) is inlined into the **implementer's EXECUTE brief** on infra-flavored work. This deliberately extends the depth-library pattern: `operational-safety` is consumed at EXECUTE here, not only at REVIEW as ADR-0031 established. A curated platform skill (T2) carries the behavioural contract no schema encodes — **detect-and-recommend** under the 3-tier dependency policy, never bundled.
- **Provision is durable and convergent; live interaction goes through reusable scripts.** **Durable-credentials-once** is added as a fifth artifact to RFC-0041 P1's multi-artifact preflight (establish once into a static session via the toolchain's own export mechanism, assert with a single identity check, reuse for the loop, resolve profile-vs-static precedence up front), alongside its verify-status, teardown, test-seed, and policy-as-code / CSPM-scanner artifacts. **Terminal-failed-state** (e.g. `ROLLBACK_COMPLETE` → destroy-recreate) is named as a convergence case. Every live-environment interaction — deploy, smoke probe, log pull, debug step — goes through a **reusable, idempotent, credential-reusing script** that accumulates into the verify / probe / teardown harness, not one-off command-spray; this sharpens RFC-0041 P4 (agent-drives-verification), it is not a new family.
- **Verification is phased and high-fidelity.** *V1:* oracles are **phased and increasing-fidelity** (static lint / validate / policy-as-code < plan / preview < runtime deploy + smoke); the cheap-early oracle is **necessary, not sufficient** ("synth passed" is the local-typecheck analog, not a done-signal). *V2:* the infra smoke is a **readiness-aware data-plane probe** — in-network-if-private, write → read-back (drive a request path to the terminal user-visible result), poll with bounded backoff distinguishing not-yet-ready / propagating from broken, self-teardown against an ephemeral uniquely-named target — refining RFC-0041 P2 without re-authoring its GATES sequence or `## Rollout`.
- **Debugging gets a localization map, not a new module.** `observability-and-smoke` is strengthened with a **symptom→layer log playbook**: enumerate the path's log groups up front; map the failing status to the emitting layer (504 / timeout → proxy / facade; 403 → authorizer / IAM; 500 → handler); bisect the chain; carry a correlation id; and **match the symptom against a failure-signature → likely-cause catalog first** (the runbook-match step HolmesGPT reports resolves ~40% of investigations before deep log reading). No new module.
- **No new reviewer or agent; review on the existing `quality-engineer`; a dedicated `infra-contract-reviewer` is deferred behind an explicit evidence trigger.** Contract-conformance and craft review ride `quality-engineer` (already the infra reviewer in spirit — it owns `operational-safety` and is already dispatched on infra-flavored work), loading `infra-contract-acquisition` + `cloud-implementation-craft` and **re-deriving the contract independently from the oracles** — never trusting the implementer's own contract evidence, which would reproduce the field-report blind spot. The auth-flow-contradiction class is caught earlier, at spec stage, by `design-reviewer`. A dedicated `infra-contract-reviewer` is deferred behind a named trigger (a spike showing oracle-execution traffic — large `plan` / `synth` output — measurably degrades `quality-engineer`'s other lenses); fetching the contract in *slices* softens the a-priori isolation case.

Boundaries on the decision:

- This adds **no new risk trigger** for "infrastructure" — destructive / irreversible already routes `apply` to full mode (RFC-0025), and "infra-flavored" is already a defined signal (ADR-0031). The new references load on that existing classifier, now into **both** the implementer's EXECUTE brief and the reviewer's REVIEW brief.
- It adds **no new `operational-safety` module beyond `cloud-implementation-craft`** — the debug family (D) strengthens the *existing* `observability-and-smoke`.
- The curated platform skill (T2) is **opt-in, recommend-when-warranted content** — not a shipped contract for any platform, and not bundled per-vendor data.
- The carves are stated and must hold both ways: **`infra-contract-acquisition`** owns *IaC-against-the-platform's-structural-contract*; **`cloud-implementation-craft`** owns *under-permissioning + timing / retry* (will the call path even succeed?); **`security-checklists`** owns *over-permissioning + security config* (is this too open?); the **policy-as-code / CSPM scanner** owns *config-against-policy*. Four distinct questions, named in the artifacts' front matter.

## Decision drivers

- **Principle 3 (habit, not infrastructure)** — the hard gate; rules out a plan-parser / schema-diff / probe-runner / oracle runner (runtime infrastructure), forcing the capability to be prose that drives the toolchain's *own* oracles.
- **Principle 1 (universal across stacks)** — rules out a per-vendor knowledge base and vendor-keyed routing (no long-tail coverage; stale on arrival); forces tool-keyed routing, the declared capability spectrum, and a mandatory weak-oracle fallback.
- **Principle 2 (no duplication)** — rules out a standalone infra skill that re-implements the loop, and rules out a new module for the debug family when an existing one (`observability-and-smoke`) owns that concern.
- **The three-reviewer ceiling (ADR-0023)** — forecloses a fourth, infra-contract-lens reviewer now; `quality-engineer` already owns operational concerns and is already dispatched on infra-flavored work.
- **The recurring human-as-relay cost across two independent builds** — drives doing this now over do-nothing; the pain is confirmed and recurring, not hypothetical.
- **Box-ticking risk** — drives "a *cited contract slice* the build references" over "acquired: yes", and independent contract re-derivation at review over trusting the implementer's evidence.
- **A standards-grounded reviewer is not a per-provider rules engine** (carried from ADR-0031) — keeps per-provider depth in the scanner and the contract in the toolchain oracles, not in the reviewer's prose.

## Consequences

**Positive:**

- The capability reuses the proven orchestrator-loaded depth-library rail (the same mechanism that already inlines `security-checklists` / `operational-safety` modules and already fires on the infra-flavored signal), so it is a second lens and a new EXECUTE consumer on an existing rail, not new infrastructure — clearing Principle 3.
- The EXECUTE-loaded craft module and the contract-grounding gate attack the **Author** families (structural + behavioral) where the field report shows the most failures — the place RFC-0041's REVIEW-only operational safety couldn't reach.
- Tool-keyed routing with a declared spectrum **covers the strong / medium tiers statically and degrades the long tail (on-prem / Hetzner / bare-metal) to runtime-probe-only coverage, declared honestly** — the weak-oracle fallback means the long tail gets no static contract oracle but loses coverage transparently rather than silently faking static checks.
- Review stays within the three-reviewer ceiling, and the dedicated-agent escape hatch is pre-named with its exact evidence trigger, so the deferral is principled rather than indefinite.

**Negative:**

- A third orchestrator-loaded reference area to maintain, EXECUTE-brief growth on infra work, and longer `work-loop` prose — real surface-area cost, justified by a failure class confirmed across two independent builds.
- The new skill is **protocol-shaped where the two existing libraries are checklist-shaped** — a small, recorded family inconsistency that may later argue for a shared shape.
- For single-familiar-platform adopters the tiering and some craft is partly dormant (mitigated: loads only on infra-flavored work; T1 / V2 run the validator / probe they'd run anyway).
- **The load-bearing contract for an unfamiliar *managed* surface lives in an artifact the repo deliberately does not ship.** The field report's dominant Author·structural failures (managed-platform naming / schema / immutability, and a managed runtime's packaging / entrypoint-import model — none of which any IaC schema or T1 oracle exposes) were fixed in both builds by *handing the loop a curated platform skill mid-session*. Because Principle 1 forbids bundling per-vendor data, that knowledge is the **opt-in, recommend-only T2 skill** — so on an unfamiliar managed surface with **no T2 skill installed**, contract grounding falls back to the weak-oracle + docs-retrieval tiers and the field-report failure is **mitigated, not closed**, until someone authors / installs the skill. The detect-and-recommend step makes the gap visible and routes it to a decision; it does not eliminate it. This is the accepted residual of choosing Principle 1 over a bundled KB.

**Neutral / to revisit:**

- **Loading `operational-safety` into the EXECUTE brief is assumed to change build behavior, not just inflate context** — this is the assumption most worth watching post-ship; if the craft prose doesn't move the implementer, A2 is dead weight. *Falsification trigger:* a post-ship infra build that still shows the field-report symptoms with `cloud-implementation-craft` loaded — reactive permission-iteration, undesigned propagation waits, ad-hoc `sleep`s, or a raw cold-start timeout surfaced to the user — reopens A2's design.
- **`quality-engineer` is assumed to hold contract + craft without a dedicated window** — believed true given slice-fetching. *Falsification trigger:* this is routed into the same evidence trigger that gates the dedicated-`infra-contract-reviewer` deferral (Decision 8) — a spike showing oracle-execution traffic (large `plan` / `synth` output) measurably degrades `quality-engineer`'s other lenses reopens both the window assumption and the no-new-agent call together.
- **The protocol-vs-checklist family inconsistency** is accepted now; a future RFC may converge the three depth surfaces on one shape.

## Confirmation

- The implementing spec's acceptance criteria encode the doctrine — the tiered tool-keyed oracle protocol + tier-honesty, the EXECUTE contract-grounding gate, the `cloud-implementation-craft` module loaded at EXECUTE, the fifth preflight artifact + terminal-failed-state convergence + the reusable-script corollary, phased oracle fidelity, the readiness-aware data-plane probe, the symptom→layer playbook, and the `quality-engineer` infra-flavored wiring — so conformance is checkable against the spec.
- The adversarial + quality + security review passes on the implementing diff check that **no executable infra tooling creeps in** (the standing scope-inflation risk), that the protocol and every module stay **tool-neutral with the capability spectrum first-class** (Principle 1), and that the **four-way carve** holds (no security config in the craft module, no per-vendor data in the protocol, no operational config migrating into `security-checklists`).
- **Enforcement is deliberately review-time, not a standing CI gate** — matching ADR-0031 and Principle 3 (the bar that forecloses a code gate is the one that makes the controls prose). An optional later governance lint (asserting the `work-loop`, `operational-safety`, and `infra-contract-acquisition` skills ship no executable oracle / parser / probe runner) could harden it without reversing this ADR, and is recorded as a possible follow-up, not a requirement.

## Alternatives considered

- **Executable infra tooling** (a schema-diff, plan-parser, probe-runner, or oracle runner). Rejected against **Principle 3** — that is runtime infrastructure — and **Principle 1** (it would need per-tool bindings). The agent runs the existing oracles directly. This is the option the form-and-home sub-decision rejects.
- **A per-vendor contract / knowledge base.** Rejected against **Principle 1** — a bundled per-cloud KB can't cover the long tail (on-prem / Hetzner / bespoke), rots between releases, and breaks stack-neutrality. The curated *platform skill* (T2) is the bounded, opt-in, recommend-when-warranted version.
- **A dedicated `infra-contract-reviewer` agent now.** Rejected against the **three-reviewer ceiling (ADR-0023)** — viable but premature; the isolation case is real-but-unproven and softened by slice-fetching. Decision 8's evidence trigger names exactly when to revisit it.
- **Do nothing (RFC-0041 / ADR-0031 as-is).** Rejected — every unfamiliar-platform build repeats the contract-ignorance and verification-gap failures, and the human stays the relay (handing the loop a platform skill, demanding a probe), observed in **two** independent builds. The pain is recurring and present now.
