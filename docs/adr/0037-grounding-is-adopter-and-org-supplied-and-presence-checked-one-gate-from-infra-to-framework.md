# ADR-0037: Grounding context is adopter- and org-supplied and presence-checked — the EXECUTE contract-grounding gate generalizes from infra to framework/library through one detect-and-recommend tier, extending (not breaking) ADR-0034's no-bundled-KB rule

- **Status:** Accepted
- **Date:** 2026-06-25
- **Decision-makers:** eugenelim

## Context

RFC-0041 and RFC-0044 gave the work-loop rich *runtime* grounding for infrastructure: at EXECUTE the agent acquires a platform contract by driving the toolchain's own oracles (`infra-contract-acquisition`), and `quality-engineer` re-derives it independently at REVIEW. ADR-0034 set the load-bearing constraint under all of it — **ship awareness and doctrine, never a bundled knowledge base of per-vendor data**, the rule it derives from CHARTER Principle 1 (universality across stacks) at `0034:59`. That doctrine is correct and is not in question here.

But the grounding it produced has three gaps, each with a concrete symptom (RFC-0047 § Problem):

- **It stops at the infra boundary.** The EXECUTE gate (`packs/core/.apm/skills/work-loop/SKILL.md:388`) explicitly calls itself *"the infra generalization of AGENTS.md's 'Grep to verify a function exists before importing it'"* — yet it fires only on a CLI invocation, an IaC resource, or app code on a managed runtime. For an **unfamiliar internal framework or third-party library**, the agent falls back to the bare grep rule, which confirms a symbol *exists* but never its *behavioral contract* (versioned signature, deprecation, call-order constraint). The software case the gate was abstracted *from* has no detect-and-recommend tier.
- **Verification tooling is rediscovered cold every loop.** The infra preflight tells the agent to *find or create* the verify-status / teardown / seed / smoke scripts and to *establish a credential session* from scratch, every loop — because the adopter has nowhere canonical to record "the smoke check is `make smoke`, we deploy to X via Terraform."
- **Organizations have no documented path to ship their stack's golden path.** An org with a standard stack *can*, in principle, author a pack — but nothing documents how to make that pack carry a *filled-in* `reference.md` and framework skills, install it alongside `core`, and distribute it internally.

The forces in play when deciding *how* to close these gaps:

- **ADR-0034's no-bundled-KB rule is the fixed point.** Whatever we add must not become the per-vendor knowledge base the doctrine rejects. The question is whether the gaps can be closed by letting *someone else* supply the context — the adopter, their org, or a detected optional tool — rather than by the *universal catalogue* shipping it.
- **The gate already names itself the software generalization.** Restoring the software case is *narrowing the abstraction back to its origin*, not inventing a parallel mechanism — so a second skill would be a redundant front door.
- **Every layer must be free to be absent.** An adopter who fills nothing must get today's behavior unchanged. The repo already has the idiom — `agentbundle-layout.toml` optional resolution, `adapt-to-project`'s "(if present)" reads, `architect-design`'s "state which surface you detected (or 'none')".
- **The org tier must not depend on the upstream catalogue at runtime.** Per RFC-0046/ADR-0036 the catalogue is a *template to fork and own*, not a live dependency; an org pack installed from an org-owned fork has no upstream runtime or package dependency.
- **The catalogue's own seed lint must not fight an org pack.** `tools/lint-seeds.py` enforces a *placeholder-only* scaffold contract on first-party seeds; an org pack ships the exact opposite — *instance content* — so the lint's scope is the load-bearing compatibility question.

## Decision

> We will treat platform, framework/library, and verification grounding as **context the adopter and their organization supply, read if present and degraded honestly if absent — never bundled, never mandated, never CI-gated on presence** — and we will deliver it by **generalizing the one existing EXECUTE contract-grounding gate from infra to framework/library through a single detect-and-recommend tier**, by **sharpening files the adopter already owns** (`AGENTS.md`, `reference.md`) into the preflight's first read, and by **documenting an org-stack pack composed entirely of existing primitives**, distributed from an org-owned detached fork. This **extends ADR-0034's no-bundled-KB rule; it does not break it** — we still ship awareness and doctrine, and the per-vendor *content* now has named non-catalogue authors (the adopter, the org, an optional detected tool). Be precise about what moves: CHARTER Principle 1 (universality across stacks) binds the **universal catalogue** — `core` and the first-party packs — and that constraint is unchanged here. An org pack shipping a filled-in `reference.md` and internal-framework skills *is* stack-specific content, deliberately so; it is permitted not because the no-bundle rule loosened but because it is **adopter-owned and outside the universality scope by construction** (a fork the org owns, never a universal catalogue pack). The rule now reads "not bundled in a universal catalogue pack," which is the rule it always was once you separate the universal catalogue from an adopter's own fork.

Four load-bearing, expensive-to-reverse sub-decisions. (Numbering is the ADR's own; it records RFC-0047's six decisions, folding D1+D2 into ADR-D1, D3+D4 into ADR-D2, D5 into ADR-D3, and D6 into ADR-D4.)

- **D1 — One gate, two surfaces: extend the existing EXECUTE gate to framework/library contracts; do not add a parallel skill.** The gate's trigger widens so it also fires when generating code against an **unfamiliar internal framework or third-party library** whose behavioral contract the agent does not hold, and routes that to a **software detect-and-recommend tier that mirrors `infra-contract-acquisition`'s T2 exactly** — detect a framework-library skill (an installed *internal* one **or** a published cloud / application-SDK vendor skill), *or* a Context7-style `resolve-library-id` + docs-retrieval surface (MCP **or** CLI/skill), *or* official versioned docs reachable via `research`; **consult and cite** the contract slice the generated code relies on if present; **recommend-and-surface-as-a-decision** if absent on an unfamiliar framework (install a published vendor skill, or author an internal one via `author-a-skill`, or point the loop at a doc MCP) — **never silently guess the behavioral contract**. This is *prose + routing*, not a new skill: the gate is restoring the software case it was abstracted from, with the detect-recommend tier the bare grep rule lacks. `quality-engineer` re-derives the cited slice at REVIEW exactly as for infra. (The 3-tier dependency policy already settles how the optional doc-retrieval surface is treated — Tier-1 detect-and-stop, never Tier-2 auto-installed; that follows from existing doctrine, not from this decision, and is restated under Boundaries and Confirmation.)

- **D2 — The recording surface reuses files the adopter already owns; the preflight reads it first; presence-check is absolute.** No new top-level config file (no `grounding.toml`). `AGENTS.md`'s "Commands you'll need" gains a short, optional infra/verification command block (deploy / smoke / teardown / seed); `reference.md`'s existing arc42 slots (Constraints; Solution strategy → Key technology decisions; Crosscutting concepts → Observability / Testing standards) are sharpened to name the managed-runtime/platform targets, the framework-library contracts, and *where the verification tooling lives*; the infra preflight gains a first step — **read the recorded coordinates if present, then fall back to cold oracle discovery**. The recorded surface **seeds** acquisition, never **replaces** it: the agent still derives the live contract from the toolchain, and a recorded value that contradicts the oracle is a drift signal, surfaced, not trusted. **Every read is `if present`** — absence lowers only the starting information; it never fails the loop and is never enforced by a CI gate. This is the load-bearing constraint, and the one most expensive to walk back: it is what keeps the whole feature additive.

- **D3 — The org tier is a pack of existing primitives, distributed from an org-owned detached fork — no new catalogue-resolution machinery, and org-pack-over-live-upstream layering is out of scope.** An org ships its golden path as a pack carrying *filled-in* `seeds/docs/architecture/reference.md` (its real stack, not the placeholder), optional `seeds/docs/CONVENTIONS.md` / `seeds/AGENTS.md` deltas, and `.apm/skills/<framework>/SKILL.md` for its internal frameworks (the D1 detect target), installed via a repo-scope **profile** (RFC-0034) that lists the org's forked `core` first. Distribution is a **fully detached fork the organization owns outright** (RFC-0046 / ADR-0036 editable install with the packaged default source blanked) — **no runtime or package dependency on the upstream catalogue**. Layering an org pack *over* a live upstream catalogue is **out of scope** (it would also need the unbuilt RFC-0031 D6 virtual-catalogue work); the systematic mechanics of an org-owned detached fork are a **named follow-on RFC**, not this decision. The boundary is clean: **this decision authors the pack** (the seeds, the framework skills, the profile); **the follow-on RFC owns the distribution mechanism** (how an org forks the catalogue into one it owns outright and distributes internally with no upstream dependency).

- **D4 — The catalogue's seed lint enforces by opt-in, so an org pack is unenforced *by construction*.** The load-bearing invariant: **a downstream or org pack must be unenforced without editing the lint or any central pack list** — its placeholder-only scaffold contract is the *inverse* of what an instance-content-bearing org pack ships, so enforcing it on an org pack is a category error. The lint therefore enforces **only** where a pack opts in, the opt-in lives **in the pack's own manifest** (not a central list the lint carries), and it stays a **single tool, not split** — both the anti-leak blocklist and the scaffold-shape checks serve one "first-party seeds are clean placeholder scaffolds" contract and key off the same is-this-a-first-party-scaffold-pack predicate, and there is no seed check an *instance* pack would also want, so the "for other things" bucket is empty. *The mechanics that realize this invariant — the `lint-seeds` → `lint-catalogue-seeds` rename, the `[pack].lint-seeds = true` flag name, and the four first-party packs that carry it (`core`, `governance-extras`, `monorepo-extras`, `user-guide-diataxis`) added in the same change — are cheaply reversible and live in the `catalogue-seeds-lint` spec and this ADR's Confirmation, not in this sub-decision.*

Boundaries on the decision:

- **No bundled per-vendor or per-library contract data.** ADR-0034's no-bundled-KB rule stands. Context7, internal framework skills, and curated platform skills are *detected*, never bundled.
- **No shipped, run, required, or auto-installed MCP / retrieval backend.** We ship *awareness* of optional MCP surfaces only (consistent with `architect-knowledge-surfaces`).
- **No auto-population of `reference.md` from the running stack.** Here `reference.md` is a surface the adopter or org *writes into*; detect-and-fill is a separate future capability.
- **No org-pack-over-live-upstream layering and no new catalogue-resolution machinery** (D3).

## Decision drivers

- **The gate already calls itself the software generalization** — drives D1's extend-don't-fork: a parallel skill would be a redundant second front door for a case the existing gate was abstracted from.
- **ADR-0034's no-bundled-KB rule is the fixed point, and the gaps are real** — drives the whole supplied-not-bundled framing: the only way to close the gaps *without* a per-vendor knowledge base is to let a non-catalogue author supply the content.
- **An empty surface must cost nothing** — drives D2's absolute presence-check: a wrong "adopters will fill it" assumption is bounded precisely because absence degrades to today's cold discovery, never to a failure.
- **The org tier must not couple to upstream at runtime** — drives D3's detached-fork distribution and the out-of-scoping of live-upstream layering.
- **The seed lint enforces the *opposite* of what an org pack ships** — drives D4's opt-in flag: an org pack must be unenforced *by construction*, needing no edit to the lint or a central list, or every downstream fork would have to patch the catalogue's tooling.
- **Principle 3 (habit, not infrastructure)** — rules out a server/registry/daemon for any layer; every layer is prose, a file the adopter owns, or a detected optional tool.

## Consequences

**Positive:**

- **One mechanism across infra and software.** The same detect-and-recommend doctrine, the same `quality-engineer` re-derivation, now span both surfaces — no second front door to learn or maintain.
- **The feature is additive by construction.** Presence-check absolute (D2) means an existing repo with no `AGENTS.md` block and no filled `reference.md` keeps working unchanged; the migration risk is confined to D4's lint flip.
- **Orgs get a golden-path story with zero new machinery** — seeds + profiles + editable-fork install already exist; the org pack is composition, and the fork has no upstream runtime dependency.
- **The seed lint stops fighting downstream forks** — an org pack is unenforced by construction, and a first-party pack that drops the flag loses enforcement loudly (caught by the migration step adding the flag to all four in one change).
- **ADR-0034 is extended, not contradicted** — the doctrine's reader can hold both: awareness-not-knowledge-base still holds; the per-vendor content now has named non-catalogue authors.

**Negative:**

- **More optional surface for an adopter to understand** — mitigated by keeping the `AGENTS.md` block short (the ~200-line cap is real) and pointing detail to `reference.md`.
- **Org-pack guidance is documentation the catalogue must maintain** as the pack/profile/fork mechanics evolve.
- **Fork maintenance moves to the organization** — pulling upstream improvements is a deliberate re-sync, not an automatic dependency update; an accepted cost of the no-upstream-dependency posture, with the systematic mechanics handed to the named follow-on RFC.
- **Grounding is mitigated, not closed, on an unfamiliar framework with no source available this session** — the honest residual ADR-0034 already recorded for the managed-surface case (`0034:80`), now inherited by the framework/library generalization. With no framework skill installed, no doc-retrieval surface, and no reachable versioned docs, the detect-and-recommend tier makes the gap *visible* and routes it to a surfaced decision, but the agent still proceeds degraded; the gap is eliminated only when someone installs or authors a source. This is the accepted cost of choosing the no-bundled-KB rule over a per-vendor knowledge base.
- **D1 can be misread as a guessing license** if "detected nothing" silently proceeds — mitigated by the detect-**and-recommend-and-degrade** wording mirrored from the infra tier, plus the `quality-engineer` re-derivation that catches a guessed slice.

**Neutral / to revisit:**

- **Whether the new optional `[pack].lint-seeds` flag needs an ADR-0021 manifest contract-version bump.** The flag is additive and optional, so the working answer is *no bump* — but the `catalogue-seeds-lint` spec must confirm this against the contract-version rule rather than assume it.
- **Whether a framework-library-skill *starter* ever earns bundling.** Declined now (no proven recurring shape, and it would sit against Principle 1); revisit if internal-framework skills converge on a stable shape across adopters.
- **Whether `reference.md` should ever be auto-populated** from the running stack — a separate detect-and-fill capability, out of scope here.

## Confirmation

- The three implementing specs' acceptance criteria encode the decision: `framework-contract-grounding` (D1 — the widened gate trigger + the software detect-and-recommend tier + `quality-engineer` re-derivation, as *prose + routing* with **no new skill**); `adopter-grounding-surface` (D2 — the `AGENTS.md` block + sharpened `reference.md` prompts + the preflight "read recorded coordinates first" step + `adapt-to-project`/`init-project` elicitation, every read presence-checked); `catalogue-seeds-lint` (D4 — the `lint-seeds` → `lint-catalogue-seeds` rename, the opt-in `[pack].lint-seeds` flag on the four first-party scaffold packs in the *same* change, the CI-step rename, and the manifest-bump confirmation). The org-pack guide (D3) and the org-owned detached-fork distribution RFC are named follow-on artifacts.
- **Review-time, not a new CI gate** (Principle 3). The spec-stage and diff `adversarial-reviewer` passes confirm no parallel framework skill creeps in (D1), no read becomes mandatory or CI-gated (D2), and the lint flip lands the flag on all four first-party packs in the same change (D4). A `security-reviewer` pass applies where the framework-grounding tier touches an optional MCP/retrieval surface (the 3-tier dependency policy bans Tier-3 auto-install; MCP stays Tier-1 detect-and-stop).
- **The presence-check is the regression fence**: every grounding read must degrade honestly to today's behavior when its surface is absent — an existing repo with no recorded coordinates and no org pack must run the loop exactly as before this decision.

## Alternatives considered

- **(a) Do nothing — keep grounding infra-only.** Rejected against the three concrete gaps: software guessing, cold preflight, orgs reinventing — each compounding per loop and per adopter.
- **(b) Bundle vendor/library contract data in the catalogue.** Rejected against ADR-0034's no-bundled-KB rule: it *is* the per-vendor knowledge base the doctrine was written to avoid, and is unmaintainable by enumeration.
- **(c) Mandate a specific external toolchain** (require Context7 / a steering-file standard). Rejected against the no-mandate constraint and the Tier-3 dependency ban: couples the loop to a tool that may be absent or unauthenticated.
- **(d) A new parallel `framework-contract-acquisition` skill for the software case.** Rejected against the **gate-already-names-itself-the-generalization** driver: a redundant second front door for a case the existing gate was abstracted from; extend the one gate instead (D1).
- **(e) A new `grounding.toml` recording surface.** Rejected against Principle 3 and the prior-art convergence (Cursor "reference files, don't copy"): the adopter already owns `AGENTS.md` and `reference.md`; a new file is surface for surface's sake (D2).
- **(f) Split the seed lint into a first-party tool and an org tool.** Rejected against the **empty "for other things" bucket** finding: both check kinds serve one contract and key off one predicate, and no seed check an instance pack wants exists — one tool with one opt-in flag is simpler and has nothing to drift (D4).
- **(g) Layer an org pack over a live upstream catalogue.** Rejected/out-of-scoped against the **no-upstream-runtime-dependency** posture: it needs the unbuilt RFC-0031 D6 virtual catalogue; the detached-fork model the org owns outright is the chosen distribution, with its systematic mechanics handed to a named follow-on RFC (D3).
- **(h) Supplied-not-bundled, present-or-not (chosen).** Closes all three gaps by naming non-catalogue authors for the per-vendor content, reuses the proven infra detect-and-recommend doctrine and existing primitives, and stays additive throughout — at the cost of more optional surface and org-pack docs to maintain.
