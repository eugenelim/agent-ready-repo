# ADR-0035: Grounding the architect *design* phase in platform reality — the deferred serverless lens + a dual-consumed grounding discipline and sync-path viability check, prose-only on the existing routing axis (no new skill, reviewer, or tooling)

- **Status:** Accepted
- **Date:** 2026-06-24
- **Decision-makers:** eugenelim
- **Consulted:** RFC-0045 (the accepted decision this records, incl. its two-build scrubbed field report and the D1–D6 decision set); its external prior-art set (the AWS Well-Architected Serverless Applications Lens design principles; AWS Prescriptive Guidance's 29 s ceiling + 202-accept-then-poll pattern; Azure Functions WAF service guide; Google Cloud Run functions best practices; the Neptune Serverless / DynamoDB on-demand / OpenSearch Serverless / SQS at-least-once anchors confirming the lens spans the serverless *class*); the spec-stage adversarial + design review of this ADR and the implementing spec
- **Supersedes:** none
- **Related:** RFC-0045 (the accepted decision this ADR records); RFC-0044 + ADR-0034 (the *build*-loop companion this mirrors one inner-loop stage earlier — same memory-vs-ground-truth thesis, design-time adaptation carved explicitly); RFC-0042 + ADR-0032 (built the dual-consumed workload-class routing axis at design Stage 0 *and* `architect-review` WA mode, and **explicitly deferred ML/SaaS/serverless** — ADR-0032 §"Neutral / to revisit": *"A future RFC that backs any of them reopens this scope"*; this is that future RFC for serverless); `docs/specs/well-architected-cloud/` (Shipped/frozen — origin of the pillar spine + the concern × workload-class lens-axis model); `docs/specs/agentic-well-architected-overlay/` (the sibling lens whose dual-consumed shape this mirrors); the repo's 3-tier dependency policy (T2's detect-and-recommend, which the K decision reuses); CHARTER Principles 1 (no per-vendor enumeration), 2 (no duplication), 3 (habit, not infrastructure)

## Context

RFC-0042 / ADR-0032 gave the `architect` pack the machinery to apply a workload-class lens **by construction at design time *and* re-check it at review** — an orthogonal concern × workload-class routing axis consumed at both `architect-design` Stage 0 and `architect-review` WA mode. RFC-0044 / ADR-0034 grounded the downstream *build* loop in the platform's real contract and runtime truth. But the design phase that sits **between** them still reasons from model memory: the routing axis is in place, yet the serverless lens it would route to was deferred (ADR-0032's scope-boundary call, RFC-0042 D5), and **no discipline forces a binding-platform fact into view for any managed service** at design time.

Two independent, multi-day builds on *different* AWS stacks (`docs/rfc/0045-notes/field-report.md`) falsify the implicit assumption that abstract pillars are enough at design altitude. Both used the `architect` pack to shape the design *before* building, and both shipped designs that missed a **structurally fatal** platform fact the design phase never forced into view:

1. **The structurally-fatal latency miss (an agent system).** The design modelled a 30–60 s multi-round LLM-agent turn as a *fully synchronous, blocking* request behind a managed HTTP API with a **non-configurable ~29 s integration timeout**, and explicitly dismissed the async alternative as "over-engineered for a real-time agent." No section summed end-to-end latency across the chain (facade → runtime cold-start → three sequential inference rounds → per-tool gateway hops) or compared it to the binding ceiling. The build then spent two days on `504 Gateway Timeout` before the architecture was found non-viable. Critically, **the `architect-review` pass had graded this design "SHIP WITH CHANGES"** with thorough security and data findings — and **zero** mentions of latency, duration, sync/async, or the timeout contract. The single highest-impact, two-day-costing flaw was invisible to **both the design and its review.**
2. **The managed-service contract gaps (a GraphRAG stack).** A second build asserted managed-service behaviours from memory and corrected them only at deploy: a store claimed VPC-only when a later version added an optional public endpoint; a single-AZ subnet topology that fails to provision (the DB subnet group requires ≥2 AZs); and the load-bearing one — a "serverless" graph store assumed to scale to zero that in fact **floors at a minimum capacity unit and bills continuously**, with per-query latency at that floor that made the chosen per-edge sequential traversal blow the function timeout on **every** live call.

The unifying cause matches RFC-0044's, **one inner-loop stage earlier**: the design is shaped, and reviewed, against model memory and abstract pillars rather than the platform's **binding contract** — but at a stage where **nothing is deployed and no toolchain oracle can be run**. The Performance pillar names "latency budget (p50/p99)" abstractly (`well-architected-pillars.md`), never as a *binding-constraint viability check*. An independent-axis read of the field-report failures clusters them into exactly what a serverless lens carries (execution/throughput limits, cold-start, scale-to-zero economics, statelessness/delivery semantics, private-network reachability) plus a cross-cutting grounding gap — evidence the fix targets a real failure space, not two over-fit anecdotes.

RFC-0045 settled *whether* and *how* to close this. This ADR records the load-bearing, expensive-to-reverse calls — the *form*, the *lifecycle position*, the *grounding evidence source*, the *taxonomy shape*, and the *version-fact home* — so a future maintainer doesn't re-litigate them. The mechanical detail (the exact lens prose, the precise rubric-check wording, whether the grounding clause is a new short reference or an in-procedure paragraph) is spec-level and lives in `docs/specs/architect-platform-grounding/`, not here.

Constraints in force when deciding:

- **CHARTER Principle 3 (a habit, not infrastructure).** The repo ships doctrine and prose the agent reasons from, never a runtime engine or executable tooling. This forecloses a design-time latency calculator or contract fetcher.
- **CHARTER Principle 1 (no per-vendor enumeration).** Anything that ships must hold across AWS / Azure / GCP and not rot on arrival; a bundled per-cloud serverless knowledge base is out of bounds.
- **CHARTER Principle 2 (no duplication).** A capability belongs in exactly one place; a parallel skill that re-implements `architect-design`'s framing, or a second copy of the latency concern in the agentic lens, is rejected.
- **The dual-consumed workload-class routing axis (RFC-0042 / ADR-0032) is already shipped and is a *general* axis** (`rubric-well-architected.md:11-19`); serverless is already a named-but-unbacked value on it.
- **The three-reviewer ceiling (ADR-0023)** scopes the core code-review lenses; cited to show a design-time skill lens and the existing `architect-review` / `design-reviewer` do not engage it — no new reviewer is added.
- **The pack's "no required composition / per-skill duplication" shape** — each skill stands alone; shared references are duplicated per skill with a duplication note, kept byte-identical.

## Decision

> We will ground the `architect` pack's **design phase** in the platform's real contract by **backing the deferred serverless workload-class lens** (`lens-serverless.md`, cloud-agnostic, spanning the whole serverless class) and adding a **dual-consumed platform-contract grounding discipline** and a **dual-consumed synchronous-path viability check** — each consumed *by construction* at `architect-design` and *independently re-checked* at `architect-review` — as **prose only on the existing RFC-0042 routing axis**, with **no new skill, reviewer, or executable tooling**. It is the **design-time companion to RFC-0044**: same ground-in-reality thesis, evidence drawn from authoritative prose rather than toolchain oracles, because at design time neither IaC nor a deployable artifact exists.

Five sub-decisions, each expensive to reverse (cross-references to "Decision N" below are to **RFC-0045's** six-decision numbering, not this ADR's bullet order; RFC-0045's **Decision 6** — the serverless-only scope and the backlog deferral of ML / SaaS — is recorded under *Boundaries*, so all six are accounted for):

- **Form & home — prose on the existing axis; a second backed lens, not a new mechanism (Decision 1).** Everything ships as reference prose and SKILL.md / rubric edits on the existing `architect` pack: one new `lens-serverless.md` (per-skill duplicated, byte-identical), a platform-contract-grounding clause in `architect-design`'s knowledge-surface discipline, a one-line cross-reference in `lens-genai-agentic.md`, and check additions to `rubric-well-architected.md`'s Performance pillar + the design-doc rubric. The routing axis that loads a lens by construction (design Stage 0) and at review (WA mode) **already exists** (RFC-0042 / ADR-0032); serverless becomes the **second backed value** on it. No executable code, no new skill (a routing value + reference suffices — Principle 2 and the pack's three-peer-skill shape), no new reviewer (the existing `architect-review` / `design-reviewer` carry it; the three-reviewer code-review ceiling is cited, not engaged).
- **Lifecycle position — dual-consumed: design grounds by construction *and* review independently re-checks (the reversal-resistant call) (Decisions 3, 4).** The grounding discipline and the viability check are each consumed at **both** times. `architect-design` grounds *by construction* (extending its existing "never fabricate landscape facts" discipline) and runs the viability check when shaping a synchronous path. `architect-review` **independently re-checks** — it does not trust the design's own assertion: a load-bearing managed-service claim with no visible grounding, or an unbudgeted synchronous long-operation path, is a **finding** (the same re-derive-don't-trust posture RFC-0044 gives `quality-engineer` at build time). This is the user's explicit ask — **the reviewer must be grounded on the platform contract, not only the author** — and it is what the field report demands: the latency miss survived *both* a missing design check *and* a blind review, so a one-sided fix (design-only or review-only) leaves the recurrence open.
- **Grounding evidence source — authoritative prose at design time, not oracles; the explicit carve vs RFC-0044 (Decision 3).** RFC-0044 acquires the contract from the toolchain's **machine oracles** (validate / plan / synth + schema slice + runtime probe) at build / verify time, when IaC and a deployable artifact exist. This ADR grounds the contract from **authoritative prose** — a curated platform skill if present, else official docs, else `research` — at design time, when **neither exists**. Same thesis (ground in reality, not memory), different evidence source dictated by the stage; the binding contract carries **source + confidence**, never model memory, and a load-bearing claim that can't be grounded is **flagged at lowered confidence**, not asserted. The design-time grounding **hands off** to RFC-0044's build-time oracle as the artifact materialises. This is not an oracle runner and ships no executable tooling.
- **Taxonomy shape — cloud-agnostic, concern-grouped, spanning the whole serverless class, not functions alone (Decision 2).** `lens-serverless.md` is converged from the three hyperscalers' own serverless guidance (so it is not AWS-shaped) and structured like `lens-genai-agentic.md` — name the concern and the question it forces, apply what bites, recite nothing. The load-bearing, expensive-to-reverse calls recorded here are the **organising shape**, not the prose within each section (that is spec-level, per §Context): the lens is **concern-grouped** rather than flat or pillar-keyed — because the three vendors' guidance converges on concerns rather than a pillar partition and a flat list would bury the sync-vs-async gate, the highest-impact concern, among lower-stakes ones — and it **spans the whole serverless class, not functions alone** (the binding facts recur across compute, data, search/analytics, and event-glue/messaging entities, and the field-report miss was a serverless *store*, not a function). The concern set the spec enumerates — execution & throughput limits + the sync-vs-async gate; cold-start & readiness; scale-to-zero economics, capacity floors & cost cliffs; statelessness, idempotency & delivery semantics; private-serverless network reachability — is the realisation of that shape; the spec owns its exact wording and may refine it as vendor practice moves without reopening this ADR.
- **Version-specific facts live in curated platform skills, never bundled per-vendor (Decision 5, K).** The lens carries durable *rules* + an illustrative example flagged "confirm current specifics with the provider" (the `cloud-primitives.md` / `well-architected-pillars.md` house style). The binding *numbers* — a specific integration-timeout ceiling, a store's minimum-capacity-unit floor, a runtime's payload / duration limits, a managed runtime's packaging / entrypoint model — live in a **curated platform skill** that the grounding discipline (C) **recommends installing when absent** on an unfamiliar managed surface (the same T2 detect-and-recommend RFC-0044 uses). Authoring an AWS-serverless platform skill is a **non-goal** of this decision; the externally-distributed `aws-bedrock-agentcore-skill` is a living instance of this tier (a user-installable skill, **not a repo pack**), and `architect-diagram`'s `agentic-bedrock-agentcore.md` reference is in-repo prior art for the shape. No per-vendor data is bundled into the agnostic lens (Principle 1).

Boundaries on the decision:

- **Scope is serverless only.** ML and SaaS stay **named-but-unbacked** in `rubric-well-architected.md` (status quo), consistent with ADR-0032 — this decision neither backs nor removes them. The deferral is recorded in `docs/backlog.md` (`ml-saas-serverless-workload-class-lenses`), updated to mark **serverless resolved by this RFC** and ML / SaaS still deferred (Decision 6).
- **No new mechanism and no new risk trigger.** This is a second consumer of the RFC-0042 routing axis plus the grounding / viability disciplines; the axis itself is unchanged and is *not* re-authored.
- **The viability check (V) has one home and a cross-reference, not two copies.** The **serverless lens owns the mechanics** (it is where managed-platform timeouts bite hardest — a non-configurable integration timeout); the **agentic lens carries the *trigger*, not a copy of the mechanics** — an agent turn is the canonical long-running operation, so the agentic lens (always loaded for an agentic design, on any provider) names the sync-long-operation question and **points to the serverless gate** for the latency-budget-vs-binding-timeout mechanics (per RFC-0045's approved open-question default). The cross-reference is what keeps the gate **reachable for an agentic design that is *not* itself serverless** (a long-running agent behind an always-on front door whose timeout is configurable rather than fixed): the agentic lens forces the *question*, and the reader follows the pointer to the serverless lens's mechanics on demand — so the gate is not stranded in a lens that Stage 0 didn't route in. The residual the home-choice accepts: the *binding-timeout* facet is sharpest on a managed/serverless front door, so a self-managed-front-door agentic design gets the question and the budget discipline but a softer (configurable) ceiling. No duplicated concern.
- **No oracle runner or executable tooling.** At design time nothing is deployed to probe; RFC-0044 owns the build-time oracle protocol. (Principle 3.)
- **The grounding discipline is scoped to load-bearing critical-path claims**, not every managed-service mention — the same "apply what bites" altitude as the lenses, to keep it from over-firing into a citation-for-everything demand.

## Decision drivers

- **The design-time grounding gap is real, present pain across two independent builds** — drives doing this now over do-nothing. A structurally-impossible design shipped and cost two days; a second build corrected managed-service contracts at deploy. The pain is recurring and confirmed, not hypothetical.
- **The reviewer was as blind as the author** — drives the **dual-consumed** lifecycle (design *and* review), and specifically the *independent re-check* posture. The "SHIP WITH CHANGES" grade with zero latency findings is the proof that grounding the author alone is insufficient.
- **Leading practice treats serverless as a recognised first-class workload-class WA lens, applied at design *and* review** — drives backing the deferred lens. The AWS Serverless Applications Lens is a published, pillar-organised workload-class lens with seven design principles; Azure (Functions WAF service guide) and GCP (Cloud Run best practices) ship the converging guidance. The lens is distilled, not invented.
- **The sync-vs-async gate is documented prior art, not invented doctrine** — drives the V check's framing. AWS Prescriptive Guidance documents the 202-accept-then-poll pattern *explicitly because of* the 29 s ceiling; Azure's Durable Functions / "defer the work, return immediately" guidance and GCP's no-post-return-work rule converge on it.
- **Principle 1 (no per-vendor enumeration)** — rules out a bundled per-cloud serverless KB (can't cover the long tail, rots on arrival); forces the cloud-agnostic lens + the K routing-to-platform-skills call.
- **Principle 2 (no duplication)** — rules out a new `serverless-design` skill (a routing value + reference suffices) and a second copy of the latency concern in the agentic lens (cross-reference, not duplicate).
- **Principle 3 (habit, not infrastructure)** — rules out a design-time latency calculator or contract fetcher; the binding facts come from prose the agent already reads, and the build-time machine-oracle job is RFC-0044's.
- **Box-ticking risk** — drives "a *cited contract slice with confidence*" over "contract grounded: yes" and "a *summed latency budget vs. a named ceiling*" over "viability: ok", and independent re-derivation at review over trusting the design's assertion.

## Consequences

**Positive:**

- The design phase gains the discipline to **ground load-bearing platform claims in reality at both construction and review**, attacking the field-report failure class **one inner-loop stage before** RFC-0044's build-time grounding can reach it — the place where a structural miss is cheapest to catch (a sentence) and most expensive to discover later (a two-day re-do).
- The change is a **second consumer of an existing rail** (the RFC-0042 dual-consumed routing axis), not a new mechanism — so it clears Principle 3 the same way ADR-0032 did, and reuses the prior-art-validated dual-consumed shape rather than inventing one.
- **Backing the serverless lens resolves the slot RFC-0042 deferred** without re-opening ML / SaaS; serverless / managed-platform designs are now shaped against the recurring binding concerns by construction and re-checked at review.
- The lens stays **cloud-agnostic with version-specific facts routed to curated platform skills**, so the durable rule lives in the lens and the current number lives in the (opt-in, recommend-when-warranted) skill — no per-vendor data to rot.

**Negative:**

- A **second backed workload-class lens** (serverless, after GenAI/agentic — the rubric names four lens slots, ML / GenAI-agentic / SaaS / serverless, of which only agentic was backed before this) **plus a cross-cutting grounding discipline and a viability check** are real surface to maintain as serverless practice moves across three vendors — justified by a structural failure confirmed across two independent builds.
- The grounding discipline adds an **early judgment call** ("is this claim load-bearing enough to ground?") when the concept is least settled — mitigated by scoping it to critical-path binding limits, the same apply-what-bites altitude as the lenses.
- The agentic-lens **cross-reference creates a soft coupling** between two lens files that must not drift — mitigated by the cross-reference being one line (the agentic lens carries no copy of the mechanics) and a possible later byte-identity / link governance lint.

**Neutral / to revisit:**

- **Whether grounding load-bearing claims at design time changes outcomes, or only adds prose, is the assumption most worth watching post-ship** — the same caution RFC-0044 flagged for its EXECUTE-brief craft prose. The field-report failures are exactly what C / V name, but whether the discipline *shifts behaviour* is the live question. *Falsification trigger:* a post-ship serverless / agentic design that still ships a structurally-impossible synchronous long-operation path or an ungrounded load-bearing contract claim with the serverless lens + grounding clause loaded reopens C / V's design.
- **On an unfamiliar managed surface with no curated platform skill installed, the *number-bearing* half of grounding is mitigated, not guaranteed — but the *rule-bearing* half still fires.** The discipline splits cleanly: the durable rule (sum the latency across the hops, compare it to *the* front-door's binding timeout, flag a synchronous long-operation; for a contract claim, demand a source + confidence and flag what can't be grounded) fires **without any platform skill**, forcing the *question* into view — which is exactly the design-time check the field report shows was missing. What the absent skill withholds is the *specific number* (the ~29 s ceiling, the capacity-unit floor) that turns the flagged question into a settled answer; that falls back to docs-retrieval / `research`, and the binding figure is mitigated, not guaranteed. So the headline two-day failure's *precondition* — the question never being asked — is closed even with no skill; what a skill adds is the authoritative figure that resolves it faster. This is the design-time mirror of ADR-0034's accepted residual of choosing Principle 1 over a bundled KB; the detect-and-recommend step makes the number-gap visible and routes it to a decision rather than eliminating it.
- **ML / SaaS stay deferred**, named-but-unbacked. A future RFC that backs either reopens this scope (the ADR-0032 pattern, now with serverless removed from the deferred set).

## Confirmation

- The implementing spec's acceptance criteria encode the decision — `lens-serverless.md` (cloud-agnostic, five concerns, whole-class) in **both** skill copies kept byte-identical; the rubric's serverless route now resolving to it; the platform-contract grounding clause in `architect-design` + the WA-mode independent re-check; the synchronous-path viability check in design + the rubric Performance check + the one-line agentic-lens cross-reference; the K routing-to-platform-skill note; the backlog update; and a dogfood run — so conformance is checkable against the spec.
- The spec-stage and diff review passes (adversarial + design, security where a boundary is crossed) confirm that **no new skill, reviewer, or executable tooling creeps in** (the standing scope-inflation risk), that the lens **stays cloud-agnostic with version-specific facts routed out** (Principle 1), that the grounding / viability disciplines **stay at design altitude** and are **genuinely dual-consumed** (not author-only), and that the **two lens copies stay byte-identical** modulo the duplication note.
- **Enforcement is deliberately review-time, not a standing CI gate** — matching ADR-0032 / ADR-0034 and Principle 3 (the bar that forecloses a code gate is the one that keeps the lens prose). An optional later governance lint (asserting the two `lens-serverless.md` copies are byte-identical modulo the duplication note, or that the agentic↔serverless cross-reference link resolves) could harden it without reversing this ADR, and is recorded as a possible follow-up, not a requirement.

## Alternatives considered

- **Do nothing — keep serverless deferred and the design phase reasoning from abstract pillars.** Rejected against the **present-pain** driver: every serverless / agentic design shaped in the pack repeats the binding-constraint blind spot, and the human stays the relay (the field report shows two independent builds where the human had to force the structural realisation mid-build). This is the status-quo cell and the outlier against the prior art.
- **Review-only — back the lens for `architect-review` WA mode alone.** Rejected as **half the fix**: design still shapes against abstract pillars, so the structural miss recurs and arrives as rework. The field report shows the miss survived a missing design check.
- **Design-only — load the lens by construction, no review re-check.** Rejected against **the field report's blind-review finding** and the **user's explicit reviewer-grounding ask**: a design would mark its own platform homework, regressing the independent-review value RFC-0032 / ADR-0032 established. The "SHIP WITH CHANGES" grade with zero latency findings is exactly the failure this leaves open.
- **Executable viability / contract tooling** (a design-time latency-budget calculator, a contract fetcher). Rejected against **Principle 3** — that is runtime infrastructure — and because nothing is deployed at design time to probe; RFC-0044 already owns the build-time machine oracles. The binding facts come from prose the agent already reads.
- **A per-vendor serverless knowledge base bundled in the pack.** Rejected against **Principle 1** — a bundled per-cloud KB can't cover the long tail, rots between releases, and breaks cloud-agnosticism. The curated *platform skill* (K) is the bounded, opt-in, recommend-when-warranted version.
- **A new standalone "serverless-design" skill or a fourth reviewer.** Rejected against **Principle 2 and the pack's three-peer-skill shape** (a routing value + reference suffices) and **the three-reviewer ceiling (ADR-0023)** (the design-time lens is a skill behaviour, not a code-review reviewer; the existing `architect-review` / `design-reviewer` carry the re-check).
- **A flat or pillar-keyed lens** instead of concern-grouped. Rejected against the **cross-vendor convergence**: the three hyperscalers' serverless guidance converges on the five concerns rather than a pillar partition, and a flat list would bury the sync-vs-async gate — the highest-impact concern.
- **Sync-vs-async as a first-class concern duplicated into the agentic lens.** Rejected against **Principle 2** — the serverless lens owns the mechanics; the agentic lens carries a one-line cross-reference (RFC-0045's approved open-question default).
