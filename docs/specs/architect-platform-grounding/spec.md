## Objective

A solution architect using the `architect` pack designs a system on a managed / serverless platform and gets its **binding platform contract forced into view at design time and re-checked at review** — not discovered two days into the build. Backing the deferred **serverless workload-class lens** (`lens-serverless.md`), a serverless / managed-platform design is shaped against the recurring binding concerns by construction and re-checked at review, exactly as RFC-0042 made the agentic lens dual-consumed. Two cross-cutting disciplines ride the same axis: a **platform-contract grounding discipline** (for every managed service on a critical path, ground its binding contract in an authoritative source with stated confidence — never model memory; the reviewer **independently re-checks** and flags an ungrounded load-bearing claim) and a **synchronous-path viability check** (sum worst-case latency across every hop, compare it to the binding front-door timeout, and force a sync-vs-async gate for a long-running operation — caught at design *and* at review). The lens stays **cloud-agnostic** (converged across AWS / Azure / GCP), version-specific numbers route to **curated platform skills**, and everything keeps the pack's altitude: name the concern and the question it forces, name no frameworks, route control-level verification out. It is the **design-time companion to RFC-0044** — same ground-in-reality thesis, evidence from prose because nothing is deployed yet.

## Boundaries

The three-tier guard that keeps an implementing agent inside the lines.
*Always do* applies without asking; *Ask first* requires human sign-off
before proceeding; *Never do* is a hard rule, even under time pressure.

### Always do

- Keep `lens-serverless.md` **cloud-agnostic** — converged from the three hyperscalers' serverless guidance, durable rules + illustrative examples flagged "confirm current specifics with the provider"; route version-specific numbers to curated platform skills.
- Keep the lens and the disciplines at **design altitude** — name the concern and the question it forces; route control-level verification to `security-reviewer` / `security-checklists` where a security boundary is crossed; name no frameworks.
- Author **both** copies of `lens-serverless.md` (`architect-design` and `architect-review`) identically, each carrying its one-line duplication note (the pack's per-skill duplication is the principle, not a bug).
- Keep the lens's concerns a **filter, not a worklist** — preserve the "apply the concerns that bite for THIS system" instruction the sibling lenses carry.
- Make the grounding discipline and the viability check **genuinely dual-consumed** — `architect-design` by construction, `architect-review` re-checking *independently* (it does not trust the design's own assertion).
- Bump the `architect` pack version (`pack.toml` + `plugin.json`) and refresh its `marketplace.json` aggregation, since this changes a user-facing capability.

### Ask first

- Any change to `architect-review`'s WA-mode rubric (`rubric-well-architected.md`) beyond (a) pointing its already-named serverless route at the new lens and (b) adding the contract-grounding + viability re-checks — the review side's lens-selection model is shipped and out of scope to redesign.
- Backing ML or SaaS (they stay explicitly deferred — see the backlog register, and the final AC's non-goal note).
- Authoring or bundling an AWS-serverless (or any per-vendor) platform skill in this PR — the K decision is recommend-when-warranted; a platform skill is a possible follow-on, not part of this scope.

### Never do

- Ship a new reviewer agent, a new skill, or any executable tooling (a latency calculator, a contract fetcher, an oracle runner) for the lens or the disciplines (Principle 3; ADR-0023 ceiling; ADR-0035).
- Bundle per-vendor serverless numbers (a specific timeout ceiling, a min-capacity floor, payload / duration limits) into the agnostic lens (Principle 1) — the durable rule lives in the lens, the current number in a curated platform skill.
- Duplicate the sync-vs-async viability *mechanics* into `lens-genai-agentic.md` — the serverless lens owns the mechanics; the agentic lens gets a **one-line cross-reference** only.
- Edit the frozen `well-architected-cloud` spec body or the frozen RFC-0042 / RFC-0044 / RFC-0045 bodies, or re-author the RFC-0042 routing axis.
- Name an internal RFC / ADR / spec-path / backlog anchor inside shipped `.apm/**` content (`AGENTS.local.md` adopter-clean rule) — provenance lives in this spec and the ADR, not in the lens prose.

- **Lens authored + both copies identical + cloud-agnostic, rubric serverless route repointed, grounding clause + re-check wired, viability check + Performance-rubric check + agentic cross-reference wired, pack-version bump** — *goal-based*: a `grep` / `diff` / build one-liner verifies each outcome. - **The lens + disciplines fire and gate as intended on a real serverless / agentic concept** — *manual QA, exercised end-to-end*: run `architect-design` on a serverless concept with a synchronous long-running path (the field report's agent-behind-a-sync-front-door shape) and record that the workload-class branch loads the lens, the viability check sums the latency budget against the binding ceiling and fires the sync-vs-async gate, and the grounding clause forces the binding contract into view with source + confidence; then run `architect-review` WA mode on a design that omits the grounding and confirm it raises the finding. A passing grep alone does not satisfy this — the built artifact (the skill, run as a user would) must produce the right behaviour.

## Acceptance Criteria

- [x] **The serverless lens is backed.** `lens-serverless.md` is authored in `architect-design/references/` and `architect-review/references/`, filling the slot `rubric-well-architected.md:17` names but never backed. It is structured like `lens-genai-agentic.md` — names each concern and the question it forces, applies what bites, recites nothing — and carries the one-line per-skill duplication note.
- [x] **The lens is cloud-agnostic, converged from the three hyperscalers' serverless guidance** (AWS Serverless Applications Lens, Azure Functions WAF service guide, Google Cloud Run best practices), not AWS-shaped. Durable rules + illustrative examples flagged "confirm current specifics with the provider"; no per-vendor binding numbers bundled in.
- [x] **The lens spans the whole serverless class, not functions alone** — it names that the concerns apply across four serverless entity types: **serverless compute** (FaaS; scale-to-zero containers), **serverless data** (auto-scaling / on-demand databases), **serverless search/analytics**, and **serverless event glue / messaging** (managed queues, topics, event buses), applying each concern to whichever entities it bites.
- [x] **The lens carries the five durable concerns**, concern-grouped (not flat, not pillar-keyed):
  - **Execution & throughput limits + the sync-vs-async gate** — bounded, often non-configurable request / integration timeouts on compute and the front doors before it; per-partition / per-unit throughput ceilings and throttling cliffs on serverless data and search; query-latency-at-floor on a scaled-down store. A long-running or burst operation forces a sync-vs-async decision — stream / 202-accept-then-poll / fire-and-forget+callback / deliberate pre-warm.
  - **Cold-start & readiness** — first-use latency is not function-only: a scale-to-zero or newly-attached compute runtime pays a cold start (worsened by VPC/network attachment and large dependencies), *and* a scaled-down serverless store / search pays a scale-up / first-connection / cache-warm latency on the first request after idle. Budget it on the critical path and choose a posture (tolerate / keep-warm / pre-warm / shrink the cold path).
  - **Scale-to-zero economics, capacity floors & cost cliffs** — "serverless" names a billing/scaling *model*, not a guarantee of zero idle cost, and the model **differs by entity**: some components reach zero (FaaS; on-demand request-priced stores; scale-to-zero containers), others **floor at a minimum capacity unit and bill continuously**. Names the two cost cliffs — the **standing floor** (min-capacity components never reach zero) and the **per-request / per-byte-scanned cliff** (a pay-per-use store or query engine where an unbounded scan is a denial-of-wallet surface).
  - **Statelessness, idempotency & delivery semantics** — serverless compute is ephemeral and share-nothing (no in-process state survives; work after a handler returns may not run); the serverless event/messaging glue is **at-least-once by default**, so consumers must be idempotent, poison messages need a dead-letter path, and ordering is guaranteed only by a FIFO/ordered variant. Multi-step flows are orchestrated by a state machine, not chained in-process.
  - **Private-serverless network reachability** — a serverless component reaching a private (VPC/VNet-resident) dependency needs explicit network attachment (which adds cold-start latency and requires the full set of egress paths/endpoints; a missing path surfaces as a silent timeout, not a clear error), and some serverless hosting tiers cannot attach to a private network at all.
- [x] **The rubric's serverless route resolves to the lens.** `rubric-well-architected.md`'s workload-class lens selection (`:17`) now loads `lens-serverless.md` for the serverless class, the same way it loads `lens-genai-agentic.md` for GenAI/agentic — with no regression to its concern-lens × workload-class axes.
