## Approach

This is the codex `dropped`‚Üífirst-class move (RFC-0009 / `dropped-primitives-coverage`)
re-run for copilot, plus a user-scope-capability flip. The shape:

1. **Write the two new projection modes first, behind their own unit tests** ‚Äî
   `copilot_agent_md.py` (markdown ‚Üí `.agent.md`, with the fail-closed tool allow-list and
   the `model`-drop / `target`-omit) and `copilot_hooks_json.py` (wiring `.toml` ‚Üí one
   self-contained `<name>.json`, with the frozen event-name map and fail-closed on unmapped
   events). These are pure functions over filesystem inputs ‚Äî the riskiest *logic*, so they
   lead, TDD.
2. **Bump the contract** ‚Äî `adapter.toml` v0.9‚Üív0.10 (copilot agent/hook-wiring flip,
   hook-body retarget, skill user target, scope-table user capability) and the schema enum at
   every `dropped` site. This is the riskiest *blast-radius* step: a wrong enum site rejects
   every contract load. Goal-based + the existing contract-load tests guard it.
3. **Dispatch the modes from `copilot.py`** ‚Äî replace the `ValueError`-on-unknown-mode with
   branches for `copilot-agent-md` / `copilot-hooks-json`, and teach it the user-scope
   targets. The resolver needs no edit (it keys off the scope table).
4. **Adapt the two packs** ‚Äî `research` (`allowed-adapters` += copilot, v0.10, degradation
   doc) and `core` (v0.10). Then the warning-rail regression confirms only `command` drops.
5. **Docs + gates** ‚Äî distribution-adapters changelog, backlog, and the live-CLI smoke.

The riskiest part is **not** the code ‚Äî it's the claim that the generated bytes load on the
real Copilot CLI. The unit/integration tests pin the *shapes*; only the manual T1‚ÄìT8 smoke
(T-gate) proves the tool actually reads them. RFC-0024 Runs 2‚Äì4 already de-risked this against
1.0.59; this plan re-runs the smoke against the then-current CLI before merge.

The modes lead the contract so that when the contract flips a primitive to a new mode, a
dispatcher already exists for it ‚Äî never a window where the contract names a mode no code
handles.

## Constraints

- **RFC-0024 / ADR-0013** ‚Äî the decision: full-parity, user-scope-capable copilot; the two
  new modes; `model`-drop; fail-closed tool allow-list; `command` stays dropped; two-pack
  (not all-pack) bump.
- **`dropped-primitives-coverage`** ‚Äî the serialisation-mode + warning-rail + atomic-bump
  precedent. The per-file hook mode deliberately *diverges* from codex's `merge-json`.
- **RFC-0022** ‚Äî the "a contract bump need not bump every pack" precedent that licenses the
  two-pack bump.
- **RFC-0012** ‚Äî superseded in part (copilot repo-only scope); its Errata already links
  forward (do not re-author).
- **RFC-0011** ‚Äî the `allowed-adapters` field shape edited for `research`.

## Design (LLD)

Shape: **integration** ‚Äî wiring the catalogue's build pipeline to Copilot's external
`.github/` + `~/.copilot/` surfaces. Stack: the `agentbundle` Python build pipeline (no
`docs/architecture/reference.md` in this repo; stack derived from the existing
`packages/agentbundle/agentbundle/build/` modules). Sub-sections pruned to the integration set.

### Dependencies & integration
<!-- Traces to: AC2, AC6, AC8, AC9-repo, AC9-user, AC11 ¬∑ contracts: adapter.toml -->
- **External dependency:** GitHub Copilot app + CLI's filesystem config layout ‚Äî
  `.github/{instructions,agents,hooks}/` (repo) and `~/.copilot/{instructions,agents,hooks}/`
  (user). Version-sensitive (preview); verified against 1.0.59 (RFC-0024 Runs 2‚Äì4). The build
  emits files; Copilot reads them ‚Äî no API coupling, only the file shape + path.
- **Internal coupling:** the two new modes depend on the contract's
  `[[adapter.copilot.projection]]` rules + `[frontmatter-mapping."copilot-agent-frontmatter-v0.10"]`;
  `copilot.py` dispatches by `mode`. The scope table feeds
  `user_scope_capable_adapters_from_contract()` (`scope.py`) with no `scope.py` edit ‚Äî a pure
  data‚Üíbehaviour coupling. **But `install.py` needs two edits** (surfaced by adversarial review):
  `_render_for_user_scope` (`install.py:~2128`) has a per-adapter dispatch whose `else` raises
  for copilot ‚Üí add a `copilot` branch; and the divergent user prefix needs a post-render rewrite
  (see Interfaces & contracts below). The warning rail (`commands/_drop_warning.py`) reads the
  contract's `dropped` modes; flipping agent/hook-wiring out of `dropped` quiets it for those
  types with no rail edit.

### Interfaces & contracts
<!-- Traces to: AC1‚ÄìAC5, AC7 ¬∑ contracts: adapter.toml, adapter.schema.json -->
- **`copilot-agent-md` output contract** (`.github/agents/<name>.agent.md` /
  `~/.copilot/agents/<name>.agent.md`): YAML frontmatter `{name, description, tools?}` + body.
  `model` dropped; `target` omitted. `tools` emitted verbatim from source after allow-list
  validation. Frontmatter emission mirrors `copilot.py:_emit_frontmatter` (sorted keys, quoted
  values) ‚Äî but `tools` must emit as the source comma-string, not a re-quoted scalar, so the
  Claude-format parser accepts it.
- **`copilot-hooks-json` output contract** (`.github/hooks/<name>.json` /
  `~/.copilot/hooks/<name>.json`): `{"version":1,"hooks":{<copilot-event>:[{"type":"command",
  "bash":‚Ä¶,"powershell":‚Ä¶}]}}`. Event keys via the frozen map; unmapped ‚Üí build error.
- **Contract schema** (`adapter.schema.json`): the projection-mode enum admits
  `copilot-agent-md` + `copilot-hooks-json` at all 4 `dropped` sites. **Dual-copy:** every
  `adapter.toml` / `adapter.schema.json` edit lands byte-identically in
  `packages/agentbundle/agentbundle/_data/` **and** `docs/contracts/`. Two distinct gates:
  the TOML by `test_contract_files_byte_identical` (`build/tests/test_contract.py`), the schema
  by `test_adapter_schema_copies_match` (`tests/unit/test_contract_v0_3_schema.py`, CI-only root).
- **User-scope divergent-prefix mechanism** (AC10b ‚Äî the load-bearing design decision):
  the build adapter (`copilot.py`) stays scope-agnostic and emits **repo-relpaths**
  (`.github/{instructions,agents,hooks}/‚Ä¶`); user-scope install rewrites the prefix
  `.github/X/`‚Üí`.copilot/X/` for **all** copilot primitives in `install.py`, post-render and
  pre-path-jail. **Chosen approach:** a copilot user-scope prefix rewrite sibling to
  `_rewrite_user_scope_hook_paths` (`install.py:700`) ‚Äî but **not** hook-gated, since copilot's
  whole prefix diverges (claude-code only rewrites hooks because its skills share `.claude/` at
  both scopes). *Alternative rejected:* making the build adapter scope-aware via the
  `[adapter.copilot.projections.<primitive>]` `target.repo`/`target.user` tables (claude-code
  declares these but the array form is still pipeline-of-record, `adapter.toml:134‚Äì147`) ‚Äî a
  larger blast radius that changes the scope-agnostic build-adapter signature for one adapter's
  benefit. The implementer may revisit if the rewrite proves leakier than the table.
- **Tool allow-list** (in `copilot_agent_md.py`): `{Read, Grep, Glob, Edit, Write, MultiEdit,
  Bash}` map/pass-through; `{WebFetch, WebSearch}` known-and-recorded (pass through, inert on
  Copilot); anything else ‚Üí build error. The list is sited in the mode module (the validation
  surface), not the contract ‚Äî it is enforcement logic, not adapter data. (Alternative
  considered: a contract `values`-map like kiro's; rejected ‚Äî Copilot self-resolves the names,
  so a rewrite map would be redundant, and the *validation* is the actual job here.)

### Failure, edge cases & resilience
<!-- Traces to: AC7, AC8, AC12, AC13, AC23 ¬∑ contracts: adapter.toml -->
- **Unknown tool token ‚Üí build error** (fail-closed; never silently dropped to be ignored by
  Copilot). **Unmapped hook event ‚Üí build error** (never emit an unrecognised event key).
- **Read-only preservation:** an agent declaring only read tokens emits only those ‚Äî no
  widening, no injected write/execute, never an omitted `tools` field (which would grant all).
- **Web-tool degradation (not a failure):** `WebFetch`/`WebSearch` pass the allow-list and are
  emitted, but Copilot custom agents ignore them ‚Äî documented in `research`'s projection notes,
  not treated as an error.
- **Empty body / missing optional fields:** empty body ‚Üí empty instructions; missing `tools` ‚Üí
  no `tools` line (Copilot default, acceptable for an all-tools agent ‚Äî but every shipped agent
  declares `tools`, so this path is for completeness).
- **Preview-version drift:** the event map + tool surface are CLI-version-sensitive; the T-gate
  smoke (AC23) re-checks against the then-current CLI and records the version.

### T2: `copilot-hooks-json` serialiser passes its unit suite

**Depends on:** none

**Touches:** packages/agentbundle/agentbundle/build/projections/copilot_hooks_json.py, packages/agentbundle/tests/unit/test_copilot_hooks_json.py

**Tests:** (AC8, AC20) ‚Äî TDD
- One source wiring `.toml` ‚Üí one `<name>.json` (per-file, not merged).
- `[[hooks.SessionStart]]` with a `command` handler ‚Üí JSON `{"version":1,"hooks":
  {"sessionStart":[{"type":"command","bash":‚Ä¶,"powershell":‚Ä¶}]}}`; `json.loads` round-trip
  asserts `version == 1` and the `sessionStart` handler shape.
- The frozen event map for all six events (`SessionStart`/`SessionEnd`/`UserPromptSubmit`/
  `PreToolUse`/`PostToolUse`/`Stop` ‚Üí their Copilot names).
- An unmapped source event (e.g. `BogusEvent`) ‚Üí build error.

**Approach:**
- New module `build/projections/copilot_hooks_json.py` with `project_copilot_hooks_json(
  source_dir, output_root, rule)`.
- Parse each wiring `.toml` (the Claude-Code nested-event shape: `[[hooks.<Event>]]` with an
  inner `hooks = [{type, command}]` array) via `tomllib`.
- Translate the outer event name through the frozen `_EVENT_MAP` constant; raise on an unmapped
  event. Serialise each handler to `{"type":"command","bash":<command>,"powershell":<command>}`
  (carry the source command into both shells ‚Äî the source is shell-agnostic `python tools/...`;
  this is the **shell-agnostic-source precondition** pinned in AC8; per-shell commands are a
  follow-on). Wrap as
  `{"version":1,"hooks":{<event>:[...]}}` and write `<source-stem>.json`.

**Done when:** `pytest packages/agentbundle/tests/unit/test_copilot_hooks_json.py` is green,
including the unmapped-event failure.

## Rollout

- **Delivery:** big-bang within one PR (atomic contract+modes+packs+docs). Reversible by
  revert ‚Äî no data migration, no published event. Adopters on packs still at v0.8 are
  unaffected (they keep prior copilot behaviour); only `research`/`core` adopters at v0.10 see
  the new copilot projections.
- **Infrastructure:** none ‚Äî pure build-pipeline + contract-data change.

## Risks

- **Preview-version drift.** Copilot is preview; the event map / tool surface verified at 1.0.59
  could shift. Mitigation: the T9 smoke re-checks the then-current CLI and records the version;
  the event map + tool allow-list are fail-closed, so a regression surfaces as a build error, not
  silent data loss.
- **`.github/` adopter collision.** `.github/agents/` and `.github/hooks/` overlap
  adopter-authored content more than `.claude/`. Mitigation: Tier-1/2/3 + `.upstream.*`
  companions already absorb this (RFC-0012); the new prefixes inherit it. `.upstream.*` files
  will appear more often ‚Äî expected, not a defect.
- **`tools/hooks/` retirement.** Moving copilot hook-body to `.github/hooks/` is a behaviour
  change; safe today (no shipped copilot hook-body consumer) but a hard cut. Mitigation: verified
  no consumer ships; documented in the changelog.
- **`build-self` projection revert.** Editing only a projected path silently reverts on
  `make build-self`. Mitigation: edit pack sources, then `build-self`, then check `git status`
  for unexpected reverts (the known trap).
- **CI-only package-test wiring.** `make build-check` doesn't run package pytest; the new test
  modules gate only if CI's explicit per-path wiring covers `packages/agentbundle/tests/`.
  Mitigation: confirm the CI step already globs the new modules' directory (it does today for
  `tests/unit` + `tests/integration`); no new CI step needed, but verify before relying on it.
