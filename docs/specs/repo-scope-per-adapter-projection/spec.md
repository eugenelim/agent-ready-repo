> **Spec contract:** this document defines what "done" means. The implementing PR must match this spec, or update it. Verification must be derivable from it.

> **Scope: one PR, one contract bump.** RFC-0012's *Follow-on artifacts* list bundles deliverables that cannot ship independently without leaving the catalogue incoherent ŌĆö the contract bump, the schema rule, the resolver rename + repo-scope branching, the constant rename, the eight packs' v0.7 bump (four user-scope-capable + four repo-only), the install-time message rail, the orphan-projection reliability fix, the migration guide, and the README's `Where primitives land` extension. Splitting risks a partial v0.6 ŌåÆ v0.7 transition where (a) the CLI accepts `--adapter` at repo scope but the four user-scope-capable packs still pin v0.6 (silent fall-through to dist-tree shape), or (b) the schema accepts `allowed-prefixes.repo` but the resolver still routes by the agents-presence heuristic at repo scope (a known-bad outcome with no adopter recourse).

## Objective

Close the gap RFC-0012 names: today `agentbundle install --pack <name> --scope repo .` produces dist-tree artifacts (`<repo>/claude-plugins/<pack>/...` and `<repo>/apm/<pack>/...`) regardless of adapter choice, and a Kiro or Codex adopter has no supported path through this CLI to land a pack at `<repo>/.kiro/skills/<skill>/SKILL.md` or `<repo>/.agents/skills/<skill>/SKILL.md`. **After this spec ships**, a Kiro adopter running `agentbundle install --pack atlassian --scope repo --adapter kiro .` lands the pack's primitives at `<repo>/.kiro/skills/`, `<repo>/.kiro/agents/`, etc.; a Codex adopter passing `--adapter codex` lands them at `<repo>/.agents/skills/`; a Claude Code adopter passing no flag picks up `DEFAULT_ADAPTER` (today `"claude-code"`) and lands them at `<repo>/.claude/skills/`. The dist-tree shape becomes opt-in via `--emit-install-routes` for catalogue-publishing workflows that script the legacy producer. The pack author's `[pack.install] allowed-adapters` declaration validates at **both** scopes; the user-scope-capability subcheck (RFC-0011 AC22 / pack-allowed-adapters/spec AC13) becomes scope-conditional so Copilot ŌĆö admissible at repo scope but not at user scope ŌĆö refuses cleanly at the right scope. The module-level constant `DEFAULT_USER_SCOPE_ADAPTER` is renamed `DEFAULT_ADAPTER` to reflect its widened semantics (a one-line flip covers both `--scope user` and `--scope repo` for enterprise rebrands); a one-release deprecation alias preserves backward shape.

Success for the **Kiro / Codex adopter:** a single CLI command lands the pack at the IDE's project-local skills directory; subsequent upgrades respect `state.adapter`; the file-safety contract (Tier-1/2/3 + companion files) applies unchanged. Success for the **Claude Code adopter at the default:** no flag change required; the pack lands at `<repo>/.claude/skills/` instead of `<repo>/claude-plugins/<pack>/...` (a visible on-disk diff documented in the migration guide). Success for the **catalogue maintainer scripting publishing:** a one-line addition of `--emit-install-routes` to existing `agentbundle install --scope repo .` invocations restores the dist-tree producer behaviour. Success for the **enterprise rebrand:** flipping `DEFAULT_ADAPTER` to `"kiro"` / `"codex"` / `"copilot"` covers both scopes from a single constant. Success for the **contract:** v0.7's `allowed-prefixes.repo` per adapter is the single declared input the path-jail consults at repo scope; the resolver's six-step lookup (with the repo-scope branching at steps 0, 4, and 5) is documented in the docstring and pinned by unit tests; the publisher-vs-installer drift refusal at step 0 is scope-uniform.

### Ask first

- **Any change to the dist-tree-producing recipes** (`per-pack-claude-plugin`, `per-pack-apm-package`, `marketplace`) at `build/main.py:167-171` or under `build/recipes/`. RFC-0012 deliberately keeps the recipes unchanged and gates them behind `--emit-install-routes`; touching them is a separate RFC's surface.
- **Adding a new install route** (e.g. `codex-plugins`). RFC-0012 ┬¦ *Alternatives considered* #2 explicitly rejects this as out of scope; a sibling RFC modeled on RFC-0008 is the path.
- **Dropping `--emit-install-routes` in this PR rather than the next minor release.** RFC-0012 ┬¦ *Alternatives* #6 commits to "one transitional release with `DeprecationWarning` on use" and "Unresolved #1 ŌåÆ remove in next minor"; collapsing to immediate removal needs RFC amendment.
- **Auto-probing `<repo>/.<ide>/` at repo scope.** RFC-0012 ┬¦ *Alternatives* #4 explicitly rejects symmetric probing. Drawback #4 + AC `test_repo_scope_does_not_probe_dot_claude` pin the asymmetry as load-bearing.
- **Modifying the upgrade-side cross-adapter refusal at `upgrade.py:365-377`** (the `if old_adapter_recorded != new_target_adapter: raise ...` block) for repo scope. RFC-0011 AC10b's behaviour extends to repo scope identically; touching this code is a future RFC's surface unless EXECUTE surfaces a defect.
- **Splitting this spec into multiple PRs.** RFC-0004 atomicity precedent (inherited from `pack-allowed-adapters/spec.md`'s *Scope* note). Splitting requires re-deciding that.

### Never do

- **No new top-level directory.** RFC-0012's surface fits inside existing trees (`packages/agentbundle/`, `packs/`, `docs/`, `.claude/`).
- **No new pack format.** `pack.toml` gains no fields beyond what RFC-0011 shipped; `allowed-adapters` validation widens to both scopes but the field itself is unchanged.
- **No new install route.** Codex plugins is explicitly out of scope per RFC-0012 ┬¦ *Alternatives* #2.
- **No new third-party Python dependency.** stdlib (`tomllib`, `pathlib`, `argparse`) plus the existing `jsonschema`.
- **No silent default when `--adapter` is unset at `--scope repo`.** Falls back to `DEFAULT_ADAPTER` (currently `"claude-code"`); the install-time stdout line names which adapter resolved. No probe-based silent routing ŌĆö that's RFC-0012 ┬¦ *Alternatives* #4 (rejected).
- **No regression of user-scope behaviour.** Every RFC-0011 AC continues to hold byte-identically. The user-scope test suite at `tests/unit/test_resolve_user_scope_target_adapter.py` extends with repo-scope parametrise cases; existing cases must continue to pass.
- **No fan-out at repo scope.** One install ŌåÆ one adapter directory. RFC-0012 ┬¦ *Alternatives* #7 explicitly rejects multi-IDE fan-out from one install; multi-IDE adopters run `agentbundle install` twice with different `--adapter` values.
- **No deletion of the legacy heuristic at step 5.** Retained for `< 0.7` packs at both scopes per RFC-0012 ┬¦ *Resolver* step 5. Retirement is a future RFC's call; Drawback #7 names this explicitly.
- **No `--adapter` allowed when combined with `--emit-install-routes` at `--scope repo`.** Hard mutex with the pinned message at the handler. The exclusion is **handler-level, not argparse `add_mutually_exclusive_group`**, because the exclusion is scope-conditional (`--emit-install-routes` is meaningless at user scope and refused independently there with its own message).
- **No removal of the codex `direct-directory` projection** at `adapter.toml`. That's RFC-0009's code, ratified governance, and this spec depends on it.
- **No state-schema version bump.** `STATE_SCHEMA_VERSION` is `"0.3"` today (`config.py:30`) and stays at `"0.3"`. `PackState.adapter` already exists since RFC-0005 / RFC-0011 AC10a; this spec just changes how the value is computed at repo scope.

### Migration safety ŌĆö in-band detection of pre-RFC-0012 state

- **AC24.** At install start at `--scope repo`, three triggers are evaluated **per-pack** (each test consults `state.toml[packs.<pack_name>]`, not global state) in precedence order **(b) ŌåÆ (a) ŌåÆ (c)**; only the first matching trigger emits. **Detection runs only on the per-IDE-install code path** ŌĆö `--scope repo` without `--emit-install-routes`. AC25's migration guide names this case explicitly so the false positive is documented rather than surprising.
  - **(b) Shape-mismatch:** `state.toml[packs.<pack_name>]` exists AND projection files exist for `<pack_name>` under the dist-tree shape (`<repo>/claude-plugins/<pack_name>/` or `<repo>/apm/<pack_name>/`). Reaching the detection block AND finding dist-tree files is the pre-RFC-0012 signal because post-RFC-0012 the only code path that produces dist-tree files (`--emit-install-routes`) doesn't enter detection. Stderr names the dist-tree paths to remove with pinned wording.
  - **(a) Adapter disagreement:** `state.toml[packs.<pack_name>]` exists with a recorded `state.adapter` AND resolver's pick disagrees with `state.adapter` AND **no dist-tree files exist** (so (b) didn't already fire). Pinned stderr names the recorded vs. resolved adapter.
  - **(c) Orphan recovery:** the AC22 path ŌĆö `state.toml` has no row for `<pack_name>` but projection files for `<pack_name>` exist on disk. (Re-listed here for the precedence chain; mutually exclusive with (b) by construction since (b) requires the state row to exist.)
