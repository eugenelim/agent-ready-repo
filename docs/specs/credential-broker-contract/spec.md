> **Spec contract:** this document defines what "done" means. The implementing PR sequence must match this spec, or update it. Verification must be derivable from it.

> **Scope: phased rollout, single coherent spec.** RFC-0013 commits to migrating every in-tree consumer of `agentbundle.credentials` as part of its acceptance scope (§ 9). Per RFC-0013 § 9's *sequencing rule*, the broker contract lands first, then the in-tree consumers migrate in dependency order, then `agentbundle.credentials` + `agentbundle/creds/` are removed in the **last** PR (which also bumps `agentbundle` 0.1.x → 0.2.0). All construction tests, ACs, and tasks are tracked in this single spec so the broker contract gate cannot drift between the broker and its consumers. Splitting the spec would re-introduce the "two resolvers concurrently" risk RFC-0013 Alternative J rejected.

## Objective

Close the three load-bearing problems RFC-0013 § Motivation names — portability (the `pip install agentbundle` step every credentialed skill carries today), SSO mismatch (the three-tier model has no shape for cookie+TTL+refresh), and lint conflation (security invariants vs. resolver implementation) — by landing the four-broker contract (`env`, `cli`, `creds`, `sso-cookie`) plus the `credential-brokers` user-scope pack plus the two new build-pipeline primitive classes (`shared-libs/` and `adapter-root-bins/`) that make the contract operable, plus the in-tree migration of every consumer.

**After this spec ships:**

- A credentialed skill declares `metadata.auth: <broker-id>` in its `SKILL.md` frontmatter; the broker-agnostic security contract (Don't-block presence, argv ban, no plaintext dotfile reads except by opt-in marker, never-logged, corporate-network propagation) applies to every credentialed primitive regardless of broker; broker-specific lint extensions layer on top.
- A skill declaring `auth: creds` receives `credentials_shim.py` (plus the per-platform Tier-2 backends `_keychain_macos.py` and `_credman_windows.py`) projected into its `scripts/` directory by `make build-self`. The skill imports `from .credentials_shim import load_credentials` — same API surface RFC-0006 froze. Cleartext lives only inside the consumer's interpreter; no PyPI dependency.
- A skill declaring `auth: sso-cookie` subprocess-invokes `~/.agentbundle/bin/sso-broker.py` (projected to the catalogue's user-scope artifact root) with `get-cookies <profile>`. The broker performs headed-browser SSO via Playwright, captures cookies, refreshes on 401, and emits the cookie-jar *path* (not the values). The broker lives outside every adapter's skills directory, so the LLM cannot auto-discover or address it.
- A skill declaring `auth: env` reads `<NAMESPACE>_<KEY>` from the environment; the catalogue contributes naming convention and invariant lint only (no runtime).
- A skill declaring `auth: cli` shells out to a vendor-authenticated binary (`gh`, `aws`, `kubectl`, `gcloud`); the vendor's CLI owns the credential.
- The six in-tree consumers (`example-credentialed-skill`, `figma`, `jira`, `jira-align`, `confluence-publisher`, `confluence-crawler`) plus the seventh site (the `add-credentialed-skill` author-skill teaching block) all use the new `from .credentials_shim` import. `agentbundle.credentials` and `agentbundle/creds/` are removed from the `agentbundle` package; the `agentbundle` package bumps 0.1.x → 0.2.0.
- `tools/lint-credentialed-skills.sh` enforces the broker-agnostic contract on every credentialed primitive plus broker-specific AST-walk checks (`auth: creds` imports the shim; `auth: env` reads every declared `<NAMESPACE>_<KEY>` at least once; `auth: sso-cookie` subprocess-invokes the broker at the canonical path; `auth: cli` falls through to broker-agnostic checks only).
- `make build-self` is the idempotent projector (resolves drift in all three outcomes — modified / missing / orphaned); `make build-check` is the read-only gate that errors on drift and names the regeneration command.

Success for the adopter installing the broker pack: their credentialed skills resolve secrets without `pip install agentbundle`. Success for the skill author: a single broker-id choice in frontmatter determines the resolution shape, the lint rules, and the template. Success for the security posture: cleartext for static tokens never crosses a process boundary, SSO cookie *values* never reach LLM-visible scope, and brokers are not LLM-auto-discoverable.

## Boundaries

The three-tier guard that keeps an implementing agent inside the lines. *Always do* applies without asking; *Ask first* requires human sign-off before proceeding; *Never do* is a hard rule, even under time pressure.

### Always do

- **Every new code path lands with at least one regression test** under `packages/agentbundle/tests/` (CLI / resolver / lint surface; `unit/` and `integration/` subtrees per existing convention) or `packages/agentbundle/agentbundle/build/tests/` (contract / schema / projection). There are two test trees per `reference_agentbundle_two_test_roots`; pick the right one and don't defer the test to "the next task."
- **Preserve RFC-0006's storage semantics byte-equivalent.** `_keychain_macos.py` (macOS Tier-2 via `/usr/bin/security`) and `_credman_windows.py` (Windows Tier-2 via `ctypes` over `advapi32.CredReadW`/`CredWriteW`/`CredDeleteW`/`CredFree`) carry over from `agentbundle/creds/` byte-for-byte. The Win32 error-code dispatch matrix (`ERROR_NOT_FOUND` = legitimate-empty; `ERROR_NO_SUCH_LOGON_SESSION` / `ERROR_INVALID_FLAGS` / `ERROR_LOGON_FAILURE` = hard-fail) is preserved verbatim. Atomic-write discipline (`os.replace` with same-mount tempfile + `chmod 0600`) is preserved.
- **Stdlib-only constraint on `credentials_shim.py`.** Same as RFC-0006 § 2 — `os`, `pathlib`, `subprocess`, `getpass`, `tomllib`, `ctypes` on Windows. No Python dependencies on the consumer side; the shim is imported as a sibling file.
- **Two distinct surfaces, each pinned separately.** (1) The **`### Security rules (non-negotiable)` Don't-block prose** that lives inside every credentialed SKILL.md / template carries verbatim text from RFC-0013 § 5 with per-broker substitutions (`auth: env` drops the dotfile sentence; `auth: cli` substitutes the vendor-CLI auth-store reference; `auth: creds` keeps RFC-0006's block; `auth: sso-cookie` substitutes "cookie jar in OS keychain" for "token in dotfile"); the lint checks for the *presence* of these substring tokens, not for their byte-exact reproduction. (2) The **lint's own stderr strings** that fire when a refusal triggers are pinned as named constants at the top of `tools/lint-credentialed-skills.sh` (or its Python sibling) and asserted by tests via the constant *name*, not by inline literal duplication. RFC-0013 § 6 specifies *check rules*; the stderr strings carry from the existing AC26(a/b/c) messages plus broker-specific extensions (e.g. `lint: skill 'foo' declares 'auth: creds' but no 'from .credentials_shim import' found in scripts/`).
- **Compose path checks via basename and `Path.parts`** in the lint, never via the literal full path string — per `feedback_credentialed_lint_substring_trap`. Both the existing AC26(c) trap and the new positive-grep targets for broker-specific checks fall under this rail; lint checks for `auth: creds` and `auth: sso-cookie` are AST-based (`ImportFrom` / `subprocess.run` shape match), not raw grep.
- **Run `make build-self FORCE=1` after every change to `packs/credential-brokers/.apm/`** so projected copies in consumer skills' `scripts/` stay in sync. Closing a build-self drift is in-scope for the PR that introduces it (per `feedback_build_self_undoes_projection_only_edits`).
- **Honour corporate-network env passthrough in every broker subprocess** — `subprocess.run(..., env={**os.environ, ...})`; never a fresh empty env. Playwright browser context launched via `chromium.launch_persistent_context(user_data_dir, env={**os.environ, ...})` with `HTTPS_PROXY` / `NO_PROXY` / `REQUESTS_CA_BUNDLE` / `SSL_CERT_FILE` / `SSL_CERT_DIR` forwarded explicitly. A unit test pins the env-passthrough; the manual-QA matrix covers browser-context propagation.
- **Use `eugenelim <eugenelim@users.noreply.github.com>` commit identity**, no Claude trailer; run `gh auth status` before push (per `feedback_check_github_auth`).

### Ask first

- **Any change to the four broker ids** (renaming `sso-cookie` to `sso`, adding a fifth broker, collapsing `env` and `cli`). RFC-0013 § Unresolved questions named these as committed decisions; revisiting after Acceptance is a follow-on RFC, not a spec amendment.
- **Any change to the reserved-prefix list for `agentbundle:*` keychain target names.** `sso` is reserved in v1 (§ 4b cookie-jar continuation scheme); the credential-setup skill rejects user attempts to use it. Adding a second reserved prefix (e.g. for a future broker) is a contract change.
- **Any change to the `~/.agentbundle/bin/` path.** RFC-0013 § 4b commits to this canonical location; per-adapter paths were rejected in the RFC (§ 4b "Why `~/.agentbundle/bin/` and not per-adapter…"). Moving it would invalidate every consuming skill's `Path.home() / ".agentbundle" / "bin" / "sso-broker.py"` resolution.
- **Switching any in-tree atlassian skill to `auth: sso-cookie` instead of `auth: creds`.** RFC-0013 § What this RFC does NOT do is explicit: all four atlassian skills migrate to `auth: creds` (PAT path); the SSO path is a per-adopter frontmatter swap in their own deployment.
- **Adding a runtime override for the shim path** (e.g. `AGENTBUNDLE_SHIM_DIR`). RFC-0013 Drawbacks explicitly rejected this as a slippery slope; the inner-loop workflow is documented as "run `make build-self` after editing the shim, before running consumer tests."
- **Splitting this spec into multiple parallel PRs** beyond the sequencing RFC-0013 § 9 pins (broker first; consumers in order; cleanup last). Splitting risks the two-resolvers-concurrently shape Alternative J rejected.

### Never do

- **No new top-level directory.** RFC-0013's surface fits inside existing trees (`packages/agentbundle/`, `packs/`, `docs/`, `tools/`).
- **No new third-party Python dependency in the `agentbundle` package.** The shim is stdlib-only. Playwright is a runtime dependency of the `sso-broker.py` *script* (per RFC-0007 conventions), not of the `agentbundle` package, and is opt-in (the broker's import-guard emits a stderr install instruction; the package does not declare it).
- **No PyPI module for `credentials_shim`.** RFC-0013 Alternative I rejected this. The shim is build-pipeline-projected; the byte-identity guarantee is enforced by the drift gate.
- **No silent degradation on Tier-2 backend hard-fail.** RFC-0006 § 2 specifies: legitimate-empty (`ERROR_NOT_FOUND` on Tier-2-capable platforms) triggers `register` (interactive); hard-fail (RFC-0006 Win32 matrix codes) exits non-zero. Linux's deferred-by-policy floor to the dotfile / 0600 file is the *documented* path, not a degradation. The `sso-cookie` broker preserves this distinction verbatim.
- **No subprocess `creds` broker.** RFC-0013 Alternative G rejected this — the wrap-and-leak shape RFC-0006 § 5 explicitly refused. The shim is in-process by construction.
- **No broker skills under `<skills-dir>/`.** RFC-0013 Alternative H rejected this. `credentials_shim.py` is projected as a sibling file in each consumer's `scripts/`; `sso-broker.py` is projected to `~/.agentbundle/bin/`. The single exception is the `credential-setup` skill (§ 4e), whose LLM-cooperative discipline is named explicitly in the pack manifest description.
- **No removal of `agentbundle.credentials` / `agentbundle/creds/` before every in-tree consumer migrates.** The cleanup is the **last** PR in the sequence. Intermediate PRs leave the module importable so a half-migrated tree continues to pass tests.
- **No `pip install playwright` invocation by the broker itself.** RFC-0013 § 4f is explicit: import-guard + stderr instruction is the v1 shape. Corporate environments gate `pip install` behind tickets; auto-invocation breaks that contract.
- **No automatic `make build-self` invocation by the `add-credentialed-skill` author skill.** RFC-0013 § 7 is explicit: the author runs it. Auto-invocation makes the build-pipeline's responsibility opaque.
- **No exhaustivity enforcement on `auth: env` declared keys vs. read keys in v1.** Presence-only per RFC-0013 § 6 — for each declared `<NAMESPACE>_<KEY>`, at least one read must exist; non-declared env reads (`os.getenv("PATH")` etc.) are not flagged. Tightening is deferred until a concrete false-positive surfaces.
- **No DOM-scraping of CSRF/XSRF tokens beyond cookies** in the `sso-broker`. RFC-0013 § What this RFC does NOT do; cookies only in v1.
- **No `sso-pat-mint` broker.** Deferred until a concrete consumer.
- **No MCP-server transport for any broker in v1.** Additive evolution per RFC-0013 § 8.
- **No collision in `shared-libs/` basenames across packs.** Two packs declaring the same `shared-libs/<file>` basename is a hard-error at projection time. Silent collisions would produce non-determinism the drift gate cannot diagnose.
- **No PATH manipulation by the build pipeline.** Consumers resolve the broker via `Path.home() / ".agentbundle" / "bin" / "sso-broker.py"` only.

## Acceptance Criteria

The spec is closed when each of the following observable outcomes is verifiable in the merged PR sequence. ACs group by surface; the plan maps each AC to one task.

### Contract surface

- **AC1.** `packages/agentbundle/agentbundle/_data/adapter.toml`'s `[contract] version` is bumped to the **next available label above whatever the contract sits at when T1 lands** — and the header comment names this spec / RFC-0013 alongside the existing RFC pointers. **Version-label reconciliation with RFC-0012** (added post-merge after origin/main landed RFC-0012's spec, which also claims v0.6 → v0.7): RFC-0013's body text targets v0.7, but RFC-0012's `repo-scope-per-adapter-projection` spec — Accepted before this PR — also targets v0.7 with a substantive scope-table + `allowed-prefixes.repo` change across every adapter. Per the RFC-0013 § 4 disjoint-version-label rule ("the two contract changes do not share a version label"), the two specs cannot both land at v0.7. The implementing PR (T1) reads the live `[contract] version` at PR-open time and bumps to `version + 1` (e.g. v0.7 → v0.8 if RFC-0012's implementation lands first; v0.6 → v0.7 if this lands first and RFC-0012's spec then bumps to v0.8). The header comment names RFC-0013 unambiguously regardless of label. - **AC25.** Broker-specific lint checks:
  - `auth: creds` — AST-walk for `ImportFrom(module="credentials_shim")` in `scripts/`; refuse if absent.
  - `auth: env` — schema requires `namespace` and `keys` (refuse if absent); AST-walk for at least one `os.environ[...]` / `os.getenv(...)` read whose first-argument `Constant.value` is **exact-string-equal** (no substring, no prefix-match, no `endswith`, no regex) to the expected `<NAMESPACE>_<KEY>` for each declared key. A fixture skill that reads `os.getenv("FOO_BAR_BAZ")` does *not* satisfy declared key `BAR` even though `BAR` is a substring — per `feedback_credentialed_lint_substring_trap`. Presence-only, not exhaustivity: non-declared env reads (`os.getenv("PATH")`, `os.environ["LANG"]`) are not flagged in v1. T8's tests include a fixture pair distinguishing `FOO_BAR` declared + `FOO_BAR` read (passes) vs. `FOO_BAR` declared + only `FOO_BAR_BAZ` read (fails).
  - `auth: sso-cookie` — Two coupled checks: **(a)** module-wide AST survey for a path expression that resolves to `Path.home() / ".agentbundle" / "bin" / "sso-broker.py"` (the rooted target), AND **(b)** at least one `subprocess.run(...)` call in the same module — restricted to `subprocess.run` or `run` aliased from `subprocess` via `from subprocess import run` so an unrelated callable named `run` cannot satisfy the rule. The decomposition (instead of "first argument of subprocess.run resolves to") accommodates the common idiom `broker = str(Path.home() / ".agentbundle" / "bin" / "sso-broker.py"); subprocess.run([sys.executable, broker, ...])` where the path is assigned to a local name and passed by reference; tracking every intra-function assignment would duplicate Python's scoping logic for marginal added precision. Refuse hard-coded non-`Path.home()` absolute string-literal paths whose tail is `sso-broker.py`. Refuse inline Playwright usage (Playwright is the broker's dependency, not the consumer's). The path-component comparison uses `pathlib.PurePosixPath.parts` rather than literal-string substring (per `feedback_credentialed_lint_substring_trap`).
  - `auth: cli` — no positive-grep enforcement; broker-agnostic checks apply unchanged.

  **Argv ban (AC24 + AC25 cross-cut):** both the first-arg flag and `dest=` keyword arguments of `argparse.ArgumentParser.add_argument(...)` calls are normalised and matched against the banned set. The `dest=` branch closes the evasion `add_argument("--xyzzy", dest="token")` where the visible flag is innocuous but `args.token` is still credential-shaped at runtime. AST-walks resolve `dest=`'s argument literally; same `_literal_string` analyser as the first-arg check.

  **`auth: sso-cookie` Popen / os.system / os.exec* ban:** the four shapes refuse-listed for credentialed consumers under `auth: sso-cookie`. `subprocess.run` is the only permitted subprocess entry point — Popen widens to stdin/stdout pipes, `os.system` interpolates a shell string, `os.exec*` replaces the in-process address space. Refusing them shrinks the exfiltration arm where a consumer ships a decoy broker invocation but uses an alternate subprocess shape to push cookies out.
