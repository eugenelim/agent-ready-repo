## Approach

Four parallel skill edits (T1–T4) followed by a single build-and-ship task (T5).
T1–T4 touch different files and have no intra-plan dependencies; they can run
concurrently or be done in any order. T5 runs build-self, verifies the projected
output, and closes the workspace.toml + roadmap updates.

The riskiest part is the capture-work SKILL.md rewrite (T1): it must preserve
all existing queue-add routing behaviors while adding the classification +
hand-off layer. The safest approach is to start from the current queue-add
SKILL.md verbatim, then add the classify/route/hand-off sections rather than
authoring from scratch — reducing the diff to net-new additions only.

## Constraints

- RFC-0064 Amendment #3 (2026-07-20): defines facets 1–4 and the companion
  items (mode tags, guard).
- RFC-0064 D9: shaping_queue `type` vocabulary is fixed: `research` / `strategy`
  / `shape` / `signal` / `design` — do not add new type values.
- Core pack must never hard-depend on an optional pack (PE, desk-research,
  experience-design). The hand-off is a conditional offer based on a probe for
  the skill file, never an import or a hard call.
- Comment-preserving write is mandatory (tomlkit / targeted insertion). A
  tomllib + tomli_w round-trip is forbidden.
- `packs/core/.apm/skills/` is the canonical source for all skill edits;
  `.claude/skills/` and `.agents/skills/` are generated by `make build-self`.

## Design (LLD)

### Design decisions

**Classification rubric (AC2).** The skill uses the item's text to classify:
build signals = "implement", "fix", "refactor", "ship", "spec"; shaping signals =
"research", "investigate", "assess", "map capabilities", "frame", "strategy",
"design review", "competitive", "signal". Ambiguous items surface the question.
Traces to: AC2, AC3, AC4.

**[backlog].open type extension (AC5).** The only schema change: `[backlog].open`
entries may now carry `type = "<subtype>"` for shaping work. Build entries never
carry `type`. No other new key on any entry. Traces to: AC5.

**Capability detection for hand-off (AC6).** The skill probes for the skill file
at `.claude/skills/<skill-name>/SKILL.md` OR `.agents/skills/<skill-name>/SKILL.md`
(for `shape` → `frame-intent`; for `research` → `desk-research-project-start`;
etc.). Pack is present if either path exists; the core pack probes both since
non-Claude adapters surface skills under `.agents/skills/`. Never import from the
pack. `signal` items skip the hand-off entirely. Traces to: AC6.

**workspace-status [build]/[shape]/[brief] prefix (AC7).** Labels change in
the "Ready to start" and Backlog sections:
- Was: `- [work] spec/foo — run work-loop …`
- Now: `- [build] spec/foo — run work-loop …`
- Was: `- [shaping_queue] bar (research) — run desk-research-project-start`
- Now: `- [shape] bar (research) — run desk-research-project-start`
- Was: `- [brief_queue] spec/baz (Ready) — run receive-brief on …`
- Now: `- [brief] spec/baz (Ready) — run receive-brief on …`
- Backlog entry with `type` field: `- [shape] slug — <comment>`
- Backlog entry without `type` field: `- [build] slug — <comment>`
The subtype and skill prompt rendering is unchanged. Traces to: AC7.

**work-loop guard insertion point (AC9).** The guard inserts after the stale-
queue check paragraph in Step 0 and before the "After orienting … proceed to
step 1 (PLAN)" sentence. Pseudocode:
```
if workspace.toml present:
    derived_slug = strip("docs/specs/", argument_path).rstrip("/")
    # Check all shaping_queue entries across active initiatives
    for each active initiative ini_slug:
        for entry in workspace[ini_slug]["shaping_queue"]["active"] + workspace[ini_slug]["shaping_queue"]["backlog"]:
            if entry["slug"] == derived_slug:
                emit redirect(type=entry["type"]) + stop
    # Also check [backlog].open for typed entries (AC9 extension)
    for entry in workspace["backlog"]["open"]:
        if entry.get("slug") == derived_slug and "type" in entry:
            emit redirect(type=entry["type"]) + stop
```
The guard resolves against workspace.toml only — it does not read the spec file.
Traces to: AC9.

## Tasks

### T1: Rename queue-add → capture-work and rewrite SKILL.md with classification + routing + hand-off

**Depends on:** none

**Touches:** `packs/core/.apm/skills/queue-add/`, `packs/core/.apm/skills/capture-work/`

**Tests:**
- Manual QA: run capture-work with a mixed build+shaping list; verify
  classification surfaced, routing correct, comments preserved, hand-off offered
  (AC1, AC2, AC3, AC4, AC5, AC6)
- Goal-based: skill directory is `capture-work/`; frontmatter `name: capture-work`;
  no `queue-add` in the SKILL.md body (AC1)

**Approach:**

1. Copy `packs/core/.apm/skills/queue-add/SKILL.md` to
   `packs/core/.apm/skills/capture-work/SKILL.md`.
2. Update the YAML frontmatter: `name: capture-work`; update `description` to
   name the classification + triage capability; update activation phrases to
   include "capture this", "capture these as queue items" in addition to
   existing phrases.
3. Add a **Classification** step before the current "Ingest" step:
   - For each item, infer `[build]` (implement, fix, spec, refactor) or
     `[shape]` (research, investigate, frame, strategy, design review, signal).
   - For `[shape]`, infer the subtype: `research`, `strategy`, `shape`,
     `signal`, `design`.
   - Surface classification to user; ask when ambiguous.
4. Update **Route** step to handle shaping destinations:
   - `[shape]` + `signal` subtype + initiative-scoped → `[shaping_queue].active`
     (signals are ongoing monitoring context, not work to pick up later).
   - `[shape]` + non-signal subtype + initiative-scoped → `[shaping_queue].backlog`
     as `{slug = "...", type = "<subtype>"}`.
   - `[shape]` + not initiative-scoped → `[backlog].open` as
     `{slug = "...", type = "<subtype>", needs?, source?}`.
   - `[build]` routing is unchanged from queue-add.
5. Add a **Hand-off** step (after Write, before Hand off):
   - For `[shape]` items (non-signal): probe for the matching skill file in
     `.claude/skills/<skill>/SKILL.md` AND `.agents/skills/<skill>/SKILL.md`.
     Pack is present if either path exists.
   - If present: surface offer to invoke. If user confirms, invoke the skill.
   - If absent: emit install hint (`requires <pack> pack — install to work this
     item`).
   - For `signal` items: skip the hand-off — signals have no action skill.
   - Always write the entry before the hand-off check.
6. Update the confirmation step to include classification in the proposed-change
   summary.
7. Delete `packs/core/.apm/skills/queue-add/` directory.

**Done when:** `packs/core/.apm/skills/capture-work/SKILL.md` exists; no
`packs/core/.apm/skills/queue-add/` directory; manual QA pass 1 succeeds.

## Rollout

Pure skill edits — no database, no infrastructure, no breaking schema change.
The `capture-work` rename is not breaking: any session that ran `queue-add`
sessions before this ships will simply use the new name at the next session.
Projected copies are regenerated by `make build-self` atomically. No flag,
no canary.

## Risks

- **Missed queue-add reference.** The consumer sweep is grep-verified, but a
  reference in a prose comment (not a code path) could survive. Mitigation:
  run the grep gate in T5 across the full projected tree, not just the edited
  files.
- **SKILL.md rewrite loses a queue-add AC.** Starting from the queue-add
  SKILL.md verbatim (T1 step 1) minimizes this; diffing the "routing" section
  before and after confirms no behavior was dropped.
- **workspace-status prefix breaks existing manual-QA transcripts.** The prefix
  change is cosmetic in the SKILL.md prose; any recorded transcript that checks
  for `[work]` instead of `[build]` will need updating. Mitigation: this is
  internal documentation only — no external contract depends on the exact
  workspace-status output format.
