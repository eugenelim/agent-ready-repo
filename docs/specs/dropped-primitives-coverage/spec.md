> **Spec contract:** this document defines what "done" means. The implementing PR must match this spec, or update it. Verification must be derivable from it.

> **Scope: one PR, one contract bump.** The contract bump, the new frontmatter mapping, the codex projection logic, the eight packs' v0.8 bump, the warning rail, and the migration guide all land in a single PR per RFC-0004 atomicity (inherited from `pack-allowed-adapters/` and `repo-scope-per-adapter-projection/`). Splitting risks (a) the contract claims `.codex/agents/` / `.codex/hooks.json` projection but no implementation writes there, or (b) packs declare v0.8 but the resolver can't dispatch codex agents.

## Objective

Close the silent-degradation footgun adopters hit when they install a pack via an adapter that projects some primitive types as `dropped` — and update the contract for adapters where the upstream tool has gained the missing primitive support since our last bump.

**For codex adopters:** today our contract says codex drops `agent` and `hook-wiring`. The upstream codex CLI gained native [subagents at `.codex/agents/<name>.toml`](https://developers.openai.com/codex/subagents) and [hooks at `<repo>/.codex/hooks.json`](https://developers.openai.com/codex/hooks) in April 2026 (v0.117.0). After this spec, `agentbundle install --pack core --scope repo --adapter codex .` lands `core`'s agents at `<repo>/.codex/agents/<agent>.toml` (markdown → TOML serialisation per a new `codex-agent-frontmatter-v0.8` mapping) and `core`'s hook-wiring at `<repo>/.codex/hooks.json` (reusing the existing `merge-json` infrastructure that backs claude-code's `.claude/settings.local.json`). The codex hooks.json shape (`event → matcher group → handlers`) is functionally identical to claude-code's `settings.json` `hooks` table, so no new merge mode is needed.

**For kiro / copilot adopters and codex's residual `command` drop:** the upstream IDE has no first-class projection target for the dropped primitive types — kiro's slash-commands are manual-trigger hooks (not standalone files), copilot has no native agent / command / hook-wiring surface, and codex's custom-prompts directory is deprecated in favour of skills. These genuinely-unprojectable cases get a pinned stderr warning at install time naming what's dropped, so the partial install is **visible** instead of silent.

Success for the **codex adopter:** `agentbundle install --pack core --scope repo --adapter codex .` produces `.codex/agents/<agent>.toml` + `.codex/hooks.json` + `.agents/skills/<skill>/SKILL.md` — three primitive types projected, only `command` dropped (and the warning rail surfaces the drop). Success for the **kiro / copilot adopter:** install proceeds; the warning rail emits a pinned line listing what the resolved adapter drops, so they can decide whether the partial install is good enough or whether to install under a different adapter. Success for the **catalogue's degradation visibility surface:** the rail is contract-driven — adding a future adapter with any `dropped` mode automatically gains the warning without code edits, and the contract bumping `dropped` → first-class for codex agents/hooks automatically quiets the rail for those types.

## Boundaries

The three-tier guard that keeps an implementing agent inside the lines. *Always do* applies without asking; *Ask first* requires human sign-off before proceeding; *Never do* is a hard rule, even under time pressure.

### Always do

- **Use the exact pinned warning wording** the AC names. The warning template is `warning: pack <name> ships <count-list> that <adapter> projects as 'dropped'; these primitives will not be installed. The compatible primitives (<compatible-list>) will proceed.` `<count-list>` enumerates non-zero counts as singular-or-plural per count: `1 agent` (no `s`) for `N == 1`, `<N> agents` for `N > 1`. Items joined by serial-comma-plus-`and` (the project's existing list-formatter convention from RFC-0012's route-list rule). `<compatible-list>` enumerates the primitive types the adapter does project for the types the pack actually ships, joined the same way. Tests assert against the exact string for both N=1 and N>1 cases.
- **Derive the warning's trigger from the contract**, not from a hardcoded adapter literal. The trigger reads `[[adapter.<resolved-adapter>.projection]]` entries with `mode = "dropped"` and walks `<pack_dir>/.apm/<dropped-type>/`; no `if adapter == "copilot"` branches.
- **Reuse the `merge-json` mode for codex hook-wiring** at the contract layer rather than introducing a new mode. The codex hooks.json shape (`{ "hooks": { "<EventName>": [{ "matcher": "...", "hooks": [{ "type": "command", "command": "..." }] }] } }`) matches what claude-code's `merge-json` + `managed-key = "hooks"` already handles for `.claude/settings.local.json`. **The build-pipeline implementation is NOT shared today** — `claude_code.py`'s `_project_merge_json` is private to that adapter module; lifting it into a sibling module `packages/agentbundle/agentbundle/build/projections/merge_json.py` (the `build/projections/` package already exists with five modules — `direct_directory.py`, `hook_id.py`, `kiro_ide_hook.py`, `merge_into_agent_json.py`, `user_merge_json.py`; this adds a sixth sibling) is part of this spec so codex.py can reuse it without re-implementing.
- **Add a new projection mode `codex-agent-toml`** for the codex `agent` projection. Markdown source (YAML frontmatter + body) serialises to TOML (`name`, `description`, `developer_instructions`) at write time. The serialiser is small and the mode is scoped to codex's agent TOML shape — a hypothetical sibling adapter with a different TOML-agent schema gets a sibling mode + sibling frontmatter mapping, not a generalised mode. The serialiser lives in **one place** — `packages/agentbundle/agentbundle/build/projections/codex_agent_toml.py` — and is dispatched **only** from the build-pipeline path (`packages/agentbundle/agentbundle/build/adapters/codex.py` gains a `codex-agent-toml` branch alongside existing `direct-directory` / `direct-file` cases). There is no independent install-side projection-mode dispatcher: the install handler routes through `render.py` → `agentbundle.build.main.run_recipe` → `build/adapters/codex.py`, so the build-pipeline dispatcher is the sole consumer.
- **Bump every pack's `[pack.adapter-contract] version` from `"0.7"` to `"0.8"`.** Eight packs (the four user-scope-capable + four repo-only) per the RFC-0012 / pack-allowed-adapters atomicity precedent. A `< 0.8` pack at codex adopts the legacy resolver path which still treats `agent` / `hook-wiring` as dropped — fine for backward compat but won't benefit from the new projections.
- **Run the warning rail at both `--scope repo` AND `--scope user`.** Today no user-scope-capable pack ships drop-eligible primitives (the four are skills-only), so the rail is silent at user scope in practice — but the hook fires whenever a resolved user-scope adapter has any `dropped` mode for a primitive type the pack actually ships. Future user-scope packs get the safety for free.

### Ask first

- **Removing `command` from the dropped-primitives warning rail for any adapter.** Kiro's slash-commands and codex's deprecated custom-prompts directory both lack a projection target the catalogue would want to use; the warning is load-bearing until a future RFC names a replacement primitive (e.g., a `manual-trigger-hook` mode for kiro).
- **Extending the `codex-agent-toml` mode to other adapters.** Codex is the only user; a hypothetical sibling adapter shipping TOML-format agents needs a sibling mode + sibling frontmatter mapping, not a renaming or generalisation.
- **Touching the upstream codex hook event vocabulary.** Codex's 10 events (`SessionStart`, `SubagentStart`, `PreToolUse`, `PermissionRequest`, `PostToolUse`, `PreCompact`, `PostCompact`, `UserPromptSubmit`, `SubagentStop`, `Stop`) are accepted as-is for the `merge-json` target. Packs that ship hook-wiring targeting events codex doesn't support fall under a separate concern (event-vocabulary refusal), not this spec.

### Never do

- **No new top-level directory.** The contract bump fits inside existing trees (`packages/agentbundle/`, `packs/`, `docs/`).
- **No new pack manifest field.** The warning rail and the codex projection both derive from existing data — `[[adapter.<name>.projection]]` for the trigger, `<pack_dir>/.apm/<type>/` for the count, the existing `[pack.install]` for scope dimension.
- **No new CLI flag.** The warning is default-on; if it proves noisy, a future RFC adds `--accept-degraded` to silence (out of scope here).
- **No refusal on dropped primitives.** This spec deliberately ships a warning, not a refusal. The install proceeds; the adopter sees the partial install up front rather than being denied access to the compatible primitives.
- **No state-schema version bump.** `STATE_SCHEMA_VERSION` is `"0.3"` and stays `"0.3"`; nothing about `state.toml` shape changes.
- **No regression of claude-code or kiro projection behaviour.** Both adapters' existing projection tables (claude-code projects all 5 primitives; kiro projects 4 of 5 with `command: dropped`) are byte-identical post-bump.
- **No code path that emits the warning without consulting the contract.** A hardcoded adapter literal anywhere in the warning's trigger condition is a contract violation. The trigger reads `mode = "dropped"` entries dynamically from the contract data.
- **No projection at codex's deprecated `~/.codex/prompts` location for commands.** The upstream deprecation is explicit; the catalogue follows.

## Acceptance Criteria

The spec is closed when each of the following observable outcomes is verifiable in the merged PR.

### Contract surface

- **AC1.** `packages/agentbundle/agentbundle/_data/adapter.toml`'s `[contract] version` is `"0.8"` (previously `"0.7"`). The header comment names this spec alongside existing RFC pointers. The sibling schema at `packages/agentbundle/agentbundle/_data/adapter.schema.json` extends the projection-mode `enum` to admit `"codex-agent-toml"` at **every site that currently enumerates `"dropped"`** — there are four such sites in the schema today (two for the top-level array-form `[[adapter.<name>.projection]]` shapes and two for the scope-conditional `[adapter.<name>.projections.<primitive>]` shapes; verify the count via `grep -c '"dropped"' adapter.schema.json` at implementation time, since the schema may evolve). Without the enum extension at every site, schema-validated contract loads reject the v0.8 contract on first run.
- **AC2.** Codex projection table changes:
  - `[[adapter.codex.projection]]` entry with `primitive = "agent"` changes from `mode = "dropped"` to `mode = "codex-agent-toml"`, `target-path = ".codex/agents/"`, `frontmatter-mapping = "codex-agent-frontmatter-v0.8"`, `on-conflict = "prompt-then-preserve"`. The conflict mode reuses the existing `prompt-then-preserve` semantics already in use by kiro skills / agents / hook-bodies and codex skills / hook-bodies (no new conflict-mode semantics introduced).
  - `[[adapter.codex.projection]]` entry with `primitive = "hook-wiring"` changes from `mode = "dropped"` to `mode = "merge-json"`, `target-path = ".codex/hooks.json"`, `managed-key = "hooks"`, `on-conflict = "merge-managed-key-only"`.
  - `[[adapter.codex.projection]]` entry with `primitive = "command"` **stays at** `mode = "dropped"` (no upstream codex target — custom-prompts deprecated per [OpenAI docs](https://developers.openai.com/codex/custom-prompts)).
- **AC3.** `[adapter.codex.scope]` table gains `.codex/` in `allowed-prefixes.repo` and `allowed-prefixes.user`. Final values: `allowed-prefixes.repo = [".agents/skills/", ".codex/", ".agentbundle/", "tools/hooks/"]`; `allowed-prefixes.user = [".agents/skills/", ".codex/", ".agentbundle/"]`.
- **AC4.** New frontmatter mapping `[frontmatter-mapping."codex-agent-frontmatter-v0.8"]` declares the YAML-frontmatter-to-TOML **field-rename** rules for codex agent projection. Required per-key sub-tables: `[frontmatter-mapping."codex-agent-frontmatter-v0.8".name]` with `rename = "name"`, and `[frontmatter-mapping."codex-agent-frontmatter-v0.8".description]` with `rename = "description"`. The shape follows the existing per-key sub-table convention (the kiro-agent-frontmatter-v0.9 mapping at `adapter.toml` uses the same per-key sub-table form for its own keyset `description` / `tools` / `model` — codex's mapping declares its own keys, not kiro's). The markdown body lands in the TOML `developer_instructions` key via a **mode-level convention** built into the `codex-agent-toml` projection mode itself — NOT via a frontmatter-mapping sub-table, since the mapping mechanism is keyed on frontmatter fields and the markdown body isn't one. YAML frontmatter fields not in the mapping (e.g., `tools`) drop silently — codex's TOML agent format has no equivalent slot for them, and a future RFC can add MCP-server mapping if needed.
- **AC5.** Kiro and copilot projection tables are byte-identical to v0.7 (both adapters' upstream tools have no new projection target since the last contract bump).

### Projection-mode surface

- **AC6.** New projection mode `codex-agent-toml` lands at the projection-dispatch surface. The mode reads a markdown agent file (YAML frontmatter + body), applies the frontmatter-mapping rename rules, and emits a TOML file at the target path. Observable outcome: input `---\nname: foo\ndescription: bar\n---\nBody.` produces a `.toml` file whose `tomllib.loads(...)` parses to `{"name": "foo", "description": "bar", "developer_instructions": "Body."}` (multi-line basic strings preserved byte-for-byte through round-trip).
- **AC7.** The `merge-json` mode used for codex hook-wiring is the existing implementation — no code change beyond the contract entry. A regression test installs a hook-wiring-bearing fixture pack via `--adapter codex --scope repo` and asserts `<repo>/.codex/hooks.json`'s `hooks` key contains the expected entries in `{event: [{matcher, hooks: [{type, command}]}]}` shape.

### Warning rail surface

- **AC8.** The install handler computes a `{type: count}` mapping of dropped primitives for the resolved adapter — types from `[[adapter.<adapter>.projection]]` entries where `mode = "dropped"`, counts from `<pack_dir>/.apm/<type>/`. Empty mapping when the adapter has no `dropped` modes OR when the pack ships no primitives of any dropped type. Observable outcome: the warning either fires with the right count list, or stays silent.
- **AC9.** The install handler computes a list of primitive types the resolved adapter does project (`mode != "dropped"`) for types the pack actually ships, used to populate the `<compatible-list>` portion of the pinned warning. Observable outcome: the warning's compatible-list names exactly the projectable types this pack contributes to disk.
- **AC10.** The install handler emits the pinned warning at the **pre-write barrier point** — after Step 5's `scopes_to_install` is finalised, each scope's target adapter is resolved (`repo_target_adapter` for repo scope; `user_target_adapter` for user scope; both for the dual-scope `--force + other_already` path), and Step 5's per-scope plan-list (`plans`) is built; before Step 6's pre-flight rails (the kiro-only user-scope hook-wiring rail) and before Step 9's writes. The plan loop that builds `plans` is itself part of Step 5 (`install.py` around line 463); the warning emits *after* that loop completes and *before* any code path that touches the filesystem. The condition is: for any scope in `scopes_to_install`, the resolved adapter for that scope has dropped modes AND the pack ships primitives of any of those dropped types. **Dual-scope behaviour:** the warning fires per-scope where the condition holds — at most two warnings (one for repo's resolved adapter, one for user's), each carrying that scope's `<adapter>` name. When repo and user resolve to the **same adapter** with the **same dropped types**, both warnings still fire (the 4-tuple short-circuit key differs by `<scope>`); the wording is identical except where the install handler interleaves scope-prefixing into the per-scope output rails downstream. This is the cleaner default: each scope's adopter-visible install line carries its own warning, and a future change to suppress the duplicate can wrap with a `(repo, pack, adapter, dropped-set)` dedup key without breaking the per-scope contract. Pinned wording: `warning: pack <name> ships <count-list> that <adapter> projects as 'dropped'; these primitives will not be installed. The compatible primitives (<compatible-list>) will proceed.` Singular-or-plural per count (`1 agent` vs `2 agents`); list-formatting uses serial-comma-plus-`and`; zero-count types elided.
- **AC11.** The warning short-circuits on repeat invocations within the same process — keyed by `(<root>, <pack_name>, <adapter>, <scope>)`. Second call for the same key emits no warning. The fourth tuple component (`<scope>`) is load-bearing: in the dual-scope case the repo-side warning and user-side warning are independently silenceable. A focused unit test pins the short-circuit by invoking the install handler twice in the same process and asserting the second call emits no warning (single-scope case); a second test exercises the dual-scope case.

## Assumptions

- Technical: Adapter contract uses `[[adapter.<name>.projection]]` arrays with `mode = "dropped"` for unsupported primitives. Today three adapters (kiro `command`, codex `agent` + `command` + `hook-wiring`, copilot `agent` + `command` + `hook-wiring`) have `dropped` entries; claude-code projects all five (source: `packages/agentbundle/agentbundle/_data/adapter.toml` grep 2026-05-26).
- Technical: Codex CLI has gained native `.codex/agents/<name>.toml` (TOML format) subagent support and `<repo>/.codex/hooks.json` (10 lifecycle events, `event → matcher group → handlers` shape) hook support since v0.117.0 around April 2026 (source: [Codex Subagents docs](https://developers.openai.com/codex/subagents), [Codex Hooks docs](https://developers.openai.com/codex/hooks) fetched 2026-05-26).
