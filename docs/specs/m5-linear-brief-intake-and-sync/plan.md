**Done when:**
- SKILL.md frontmatter: `name: linear-brief-intake`, description triggers on turning
  a Linear Issue or Project into a product brief.
- SKILL.md body covers (ordered):
  1. **Prerequisites:** `linear: check` (hard dep — exit 2 → tell user to run
     `credential-setup` and stop); probe for `receive-brief` (soft dep — absent →
     graceful degradation path).
  2. **Stage 1 — Intake:** `linear: get-issue <IDENTIFIER>` (fields: title,
     description, identifier, children, project); or `linear: get-project <SLUG>`
     + `project.issues`; single-issue-no-children guard: recommend `new-spec` and
     confirm before proceeding; project-scope >10 cap: surface count and ask to filter.
  3. **Stage 2 — Map to brief:** title+description → `## Outcome` (verbatim);
     children → `## User stories` Shape B (`- **US-n.** (LIN-NNN) <text>`); project.url
     → `Epic:` (omit when null); Scope, Appetite, Success metrics left blank; never
     invent content; write to `docs/product/briefs/<slug>.md`; confirm slug before
     overwriting existing file; register under `[brief_queue].draft` in `workspace.toml`.
  4. **Stage 3 — Handoff:** invoke `receive-brief` by name (not path); hand off and
     stop — do not reimplement elicitation or decomposition.
  5. **Stage 4 — Graceful degradation:** when `receive-brief` absent, inline compact
     decompose/execute instruction; note `new-spec`/`work-loop` may also be absent.
  6. **Don't:** no Linear write verbs; no path-based sibling references; no cross-repo
     coordination; no invented Outcome.
- manifest.json `deps.skills`: `linear` (hard dep, purpose: all reads, no write
  verbs); `receive-brief` (soft dep, purpose: elicitation, decomposition, execution).
- references/examples.md: at minimum one happy-path example (single Issue → brief →
  `receive-brief` handoff) and one edge case (single issue no children → `new-spec`
  recommendation).

### T4: `linear-brief-sync` skill
**Verification mode:** manual QA
**Depends on:** T1
**Files:**
- `packs/linear/.apm/skills/linear-brief-sync/SKILL.md`
- `packs/linear/.apm/skills/linear-brief-sync/manifest.json`

**Done when:**
- SKILL.md frontmatter: `name: linear-brief-sync`, description triggers on catching
  up a brief from a changed Linear Issue.
- SKILL.md body covers:
  1. **Prerequisite:** `linear: check` (hard dep — exit 2 → stop).
  2. **Status guard — three cases:**
     - `Ready` → proceed without confirmation.
     - `Executing` → **refuse** and stop with the current status displayed (no
       confirmation path — this is a hard lock per spec Never-do rule).
     - `Draft` or `Shipped` → surface the current status and ask to confirm before
       proceeding; only proceed after PE confirms.
  3. **Re-fetch:** `linear: get-issue <IDENTIFIER>` (title, description, children).
  4. **Diff — Linear-sourced fields only:** compare `## Outcome` section (title +
     description) and `## User stories` section (children) against re-fetched data.
     Protected fields by fixed convention: Scope/Non-goals, Appetite, Rabbit holes,
     Instrumentation, Success metrics — NEVER proposed for change.
  5. **Present delta — section-level before/after** per RFC-0068 D3:
     ```
     ## Outcome — proposed change
     CURRENT:
     ---
     <verbatim current brief section>
     ---
     PROPOSED (from Linear re-fetch YYYY-MM-DD):
     ---
     <verbatim proposed replacement>
     ---
     Accept this change? (y / n / edit)
     ```
     For User stories: show diff items — `+ US-n. (LIN-NNN) <new>`, `~ US-n.` (title
     changed), `- US-n.` (removed). Present each field independently.
  6. **Write only PE-approved changes.** Unapproved fields left unchanged.
  7. **Don't:** no write verbs toward Linear; no overwriting protected fields; no sync
     when `Status: Executing` (refuse, do not ask to confirm override).
- manifest.json `deps.skills`: `linear` (hard dep).

**No stub** (manual QA — prose skill).

**`linear-brief-intake` happy path:**
Trace through: agent invokes `linear: check` → succeeds → `linear: get-issue ENG-123`
→ maps title+description to `## Outcome` verbatim → maps children to `## User stories`
using `- **US-1.** (ENG-456) <text>` format → sets `Epic:` from project.url (or
omits when null) → writes `docs/product/briefs/<slug>.md` → registers in
`[brief_queue].draft` → invokes `receive-brief` by name and stops.

**`linear-brief-intake` single-issue-no-children:**
Input: issue with empty `children`. Agent must recommend `new-spec` and confirm
before proceeding.

**`linear-brief-intake` project-scope >10:**
Input: project with 15 issues. Agent must surface "15 issues found" and ask to
filter.

**`linear-brief-intake` graceful degradation:**
With `receive-brief` absent: agent produces brief, inlines compact
decompose/execute instruction, does NOT stop.

**`linear-brief-sync` Executing-lock:**
Brief with `- **Status:** Executing`. Agent must refuse and surface the status
without writing.

**`linear-brief-sync` happy path:**
Re-fetch → `## Outcome` changed → agent presents before/after block →
PE answers `y` → agent writes only `## Outcome`; PE answers `n` for User stories
→ User stories section unchanged.
